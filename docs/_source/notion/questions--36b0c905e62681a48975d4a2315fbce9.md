## ОНЛАЙН-ДЕПЛОЙ И МУЛЬТИ-АРЕНДА (multi-tenant) — от 25.05.2026
> Расширение архитектуры за пределы локального деплоя: вариант размещения на VDS с обслуживанием нескольких контор. Локальный деплой при этом НЕ убираем — должны жить оба варианта.
- ОНЛАЙН-ДЕПЛОЙ НА VDS: добавить в реализацию вариант развёртывания доков и агента на VDS (онлайн), параллельно с локальным. Продумать, что общего в кодовой базе у локального и онлайн-варианта, а что развилкой в конфиге.
- МУЛЬТИ-АРЕНДА (несколько контор на одном VDS): продумать механизм подключения нескольких фирм. У каждой — отдельная конфигурация 1С и личная папка документов. Изоляция данных между конторами обязательна (одна не должна видеть данные/индекс другой). Решить: отдельные коллекции Qdrant на фирму vs общая коллекция с tenant-фильтром; где границы изоляции (индекс, папки, ключи).
- ВЕБ-МОРДА: продумать веб-интерфейс для заливки и просмотра файлов конфигурации и документов. Кто заливает (сотрудник конторы / админ), что видит, как привязывается к конкретной фирме.
- АВТОРИЗАЦИЯ ФИРМЫ: продумать авторизацию конторы и на MCP, и на веб-морде. Как фирма идентифицируется при запросе к MCP-серверу, как переиспользуются/различаются креды между MCP и вебом, где хранятся секреты.
- ЛОКАЛЬНЫЙ ДЕПЛОЙ СОХРАНЯЕМ: вариант локального запуска (для Сергея и мамы) остаётся как есть. Онлайн/мульти-аренда — это дополнительный режим, не замена.
- МОНИТОРИНГ ПО ТЕНАНТАМ: при мониторинге разделять две фирмы на одном VDS как РАЗНЫХ клиентов — у них разный набор данных и разные запросы. Метрики канарейки (латентность, здоровье индекса, RAM, просадка качества, фолбэки) и алерты в Sentry должны быть размечены по tenant_id, иначе деградация одной конторы утонет в усреднении по обеим. Стыкуется с задачей 0c (канарейка) и Sentry-цепочкой.
### Влияние мульти-аренды на ранее принятые решения
- QDRANT EMBEDDED vs SERVER становится развилкой по режиму, а не глобальным решением. Embedded выбирался ради «маме не ставить вторую сущность» (локальный деплой) — для VDS с несколькими конторами это ограничение отпадает, там скорее нужен серверный Qdrant с разделением по коллекциям/тенантам. Пересмотреть ADR: зафиксировать embedded для локального режима, server — для онлайн/мульти-аренды.
- КАНАРЕЙКА «ТОЛЬКО В ПОТОКЕ, БЕЗ ФОНОВОГО ДЕМОНА» под вопросом на VDS. На мульти-аренде «протухшее в покое» (истёкший ключ, недоступный индекс) у одной конторы некому разбудить, если её пользователь давно не заходил — а вторая контора работает и ждёт корректных метрик. Фоновый сторож на VDS может оказаться оправдан. Решение «без демона» оставить для локального режима, для онлайн — пересмотреть.
- PUSH ПОЛЬЗОВАТЕЛЮ: ограничение «MCP не может сам инициировать сообщение» на мульти-аренде острее (админ вне MCP-контура видит всех через Sentry — это ок для админа, но не для уведомления самой конторы). Веб-морда может стать тем каналом push к пользователю, которого нет у MCP — например, уведомить контору о готовности переиндексации.
- ИЗОЛЯЦИЯ ФАЙЛОВОГО ХРАНИЛИЩА ПО ТЕНАНТАМ (добавлено 2026-05-27 по HLE-494 #9): файлы конторы складываются по схеме `/data/{tenant_id}/...` (своя папка на фирму), FastAPI-зависимость инжектит `tenant_id` из JWT в фильтры Qdrant и таблицы PostgreSQL. Это четвёртая развилка мульти-аренды (источник: `bsl-atlas-opensource-research.md` Пробел 9): помимо Qdrant embedded vs server, канарейки и push-канала, нужно отдельно зафиксировать механизм файловой изоляции — иначе в коде это полезет в каждое чтение/запись и поломает локальный режим.
## ПО РЕЗУЛЬТАТАМ ИССЛЕДОВАНИЯ ПОСЛЕДНИХ РЕКОМЕНДАЦИЙ ПО RAG-АГЕНТАМ ОТ 25.05.2026
> Источник: ресёрч «Современный RAG для документации и кода» + разбор в чате. Краткий вывод: текущий стек (BGE-M3 dense+sparse в Qdrant + Cohere Rerank 4 + фолбэк BGE-reranker-v2-m3) совпадает с тем, что тянет вверх все топовые бенчмарки. Ниже — что из ресёрча превратилось в открытые вопросы и решения, требующие проработки.
### Поиск и индексация
- CONTEXTUAL RETRIEVAL (приём Anthropic): добавлять ли к каждому чанку преамбулу 50–100 токенов («откуда этот кусок»), сгенерированную дешёвой моделью (Haiku) перед индексацией? На бенчах даёт −49% провалов поиска top-20 с BM25 и −67% с реранкингом, стоит \~\$1/M токенов с кешем. Кандидат в задачу 4 (индексатор доки). Решить: применяем ли, на каком этапе, чем генерируем преамбулу локально vs облаком.
- ГИБРИД dense+sparse уже в стеке (BGE-M3). Подтвердить параметры слияния: RRF с k=60, размер пула кандидатов (top-50 dense + top-50 sparse → реранк → top-8–12). Откалибровать на своём eval-наборе.
- РЕРАНК-ПОРОГИ как сигнал «не нашёл»: ввести порог уверенности по скору реранкера (ориентир: Cohere rerank \< 0,3, BGE-reranker \< 0,5) → ниже порога агент честно говорит «нет данных» вместо синтеза. Откалибровать пороги по факту.
- QUERY-ТРАНСФОРМАЦИИ (HyDE / multi-query / step-back): не на старте. Кандидат для adaptive-маршрутизации — короткие/расплывчатые → HyDE, неоднозначные → multi-query, конкретные → прямой поиск. На фактологичных запросах добавляют шум и +43–60% задержки. Решать по телеметрии.
- LONG-CONTEXT vs RAG: не валить всю конфу в контекст даже при больших окнах — бенчи (LongBench v2) показывают, что хороший retrieval+реранк обходит «1М токенов в лоб». Подтверждаем стратегию «искать агрессивно, давать модели 8–20 отобранных чанков».
### Обработка кода (стыкуется с ПРИОРИТЕТОМ №1)
- AST-ЧАНКИНГ для кода 1С: резать НЕ по строкам, а по структуре (границы Процедура/Функция/Область, процедура целиком = чанк). Приём cAST даёт +4,3 Recall@5 / +2,67 Pass@1. Инструмент: грамматика BSL Language Server / 1c-syntax (tree-sitter-качество). Прямо отвечает на «как резать длинную ОбработкаПроведения». Завести в задачу 3.
- AST-ВЫВЕДЕННЫЙ ГРАФ vs LLM-граф: ресёрч (arXiv 2601.08773) подтверждает наш приоритет — детерминированный граф из структуры дешевле и надёжнее LLM-извлечения, которое само источник галлюцинаций. Усиливает решение «граф вызовов парсером, не LLM».
- КОД-СПЕЦИАЛИЗИРОВАННЫЕ ЭМБЕДДЕРЫ (SFR-Embedding-Code, voyage-code-3, jina-code) лидируют на CoIR, НО не понимают русские идентификаторы из коробки. Для кода 1С (Цены, Номенклатура, ПолучитьЗначение()) мультиязычный BGE-M3 эмпирически лучше. Вопрос: нужен ли вторичный код-индекс отдельным эмбеддером, или BGE-M3 закрывает оба.
### Эмбеддинги для русского
- BGE-M3 — разумный дефолт для RU (ruMTEB retrieval 74,79, наравне с 7B-моделями при 0,6B + три типа векторов). Открытый вопрос: A/B-тест русских дообучений (FRIDA от ai-forever, USER-bge-m3, GigaEmbeddings от Сбера — текущий SOTA ruMTEB 69,1 ср.) на отложенном 1С-наборе.
- ДООБУЧЕНИЕ BGE-M3 на корпусе ИТС: кандидат после обкатки. Доменный файнтюн обычно даёт +10–30% recall. Рецепт dense+sparse+ColBERT есть в репо FlagEmbedding. Не на старте.
- ОПЦИОНАЛЬНЫЙ АПГРЕЙД на Qwen3-Embedding-4B/8B — только если железо тянет И eval показывает \>5 п.п. улучшения. Иначе не трогаем.
### Eval-набор («золото») и анти-галлюцинации
- ГДЕ БРАТЬ ЗОЛОТО: (1) основной костяк — из статей ИТС (статья = готовая пара «ответ + источник-чанк»), 100–150 кейсов руками; (2) полуавтомат — RAGAS TestsetGenerator по корпусу, затем ручная отбраковка (оставим 40–60%); (3) самое ценное — реальные вопросы Сергея и мамы, копить с первого дня; (4) формулировки вопросов — с форумов (Инфостарт, mista). Цель 150–300 за 3–4 недели.
- NEGATIVE-КЕЙСЫ (агент честно говорит «не нашёл»): обязательны, это половина смысла проекта. Методика ведения требует проработки и заноса в runbook:
- привязывать каждый к версии корпуса (corpus_version) — кейс это утверждение о СЛЕПКЕ индекса на дату, не о реальности;
- не выдумывать, а собирать из реальных промахов агента (телеметрия «не нашёл»/фолбэков) + подтверждать руками: реально нет в корпусе → negative; ответ был, агент тупанул → это проваленный positive, чинить;
- эталон формулировать как «не соврал», а НЕ «промолчал» → обновилась дока, ответ появился, агент нашёл = успех, а не сломанный тест;
- делить на «вечнозелёные» (вне домена / заведомо отсутствующие сущности — стабильны годами) и «текучие» (в домене, но не покрыто — ревизия при каждом обновлении корпуса);
- привязать ревизию negative-набора к регламенту ежемесячного обновления доков (Linear 18).
- RAGAS-HARNESS В CI: 4 метрики (Faithfulness, Answer Relevancy, Context Precision, Context Recall). Блокировать деплой при Faithfulness \< 0,85 ИЛИ Context Precision \< 0,80. Триггер расследования при недельном падении Faithfulness \> 5%. Инструментует канарейку (Linear 0c) и даёт LLM-судье (Linear 12) объективную шкалу. Решить: где хранить золотой набор, кто владелец, частота прогона.
- FAITHFULNESS ≠ CORRECTNESS: система может набрать 0,95 faithfulness и выдать неверный ответ на устаревшем/неверном контенте. Нужен отдельный контроль слоя контекста (свежесть, согласованность определений) — стыкуется с регламентом обновления и метрикой противоречивости (Р1).
- ЗАЗЕМЛЁННЫЙ ПРОМПТ как последний рубеж: если Faithfulness \< 0,80 даже с реранком — переписать системный промпт с мягкого («помоги по документации») на строгий (только из приведённых фрагментов; цитировать chunk_id в каждом утверждении; нет источника → «в документации не описано», додумывать запрещено; разделять найдено/выведено). ВАЖНО: лечит «модель врёт при наличии источников», НЕ лечит «источники не нашлись» — сначала чинить Context Recall, потом промпт. Завести в задачу 7 (поведенческий контракт).
- SELF-RAG (токены рефлексии IsREL/IsSUP перед и после генерации): кандидат для этапа 2. По бенчам снижает галлюцинации до \~5,8% (против 12–14% у обычного агентного). Решать после baseline.
### Архитектурные подтверждения (снимают неопределённость, не открытые вопросы)
- GRAPHRAG / LightRAG по докам: ресёрч подтверждает наш скепсис — окупается ТОЛЬКО на глобальных вопросах «опиши архитектуру подсистемы», а \~95% реальных вопросов по 1С локальны («как закрыть месяц?»), где обычный гибрид выигрывает. Держать как опциональный вторичный индекс, не на старте. LightRAG предпочтительнее полного GraphRAG (дешевле поддержка). Усиливает текущее решение «граф по докам — кандидат после обкатки».
- ADAPTIVE RAG (роутер сложности запроса): ресёрч подтверждает кандидата — агентный RAG обходит статику ТОЛЬКО на сложных многошаговых, на простых это «чистая трата». Маршрут по сложности оправдан. Этап 2.
- ИНДЕКСИРУЕМ, А НЕ GREP: Claude Code работает без эмбеддингов (агентный Glob+Grep) — но это для БЫСТРО МЕНЯЮЩЕГОСЯ кода. Наш случай (статичная RU-документация + стабильная платформа, обновление раз в месяц) — полная противоположность, тут индекс+контекст+гибрид+реранк окупаются максимально. Подтверждает выбор RAG-архитектуры.
### Очерёдность внедрения (из ресёрча, как ориентир для приоритизации задач)
1. Baseline (\~1–2 нед): AST/семантический чанкинг + Contextual Retrieval + гибрид + реранк + цитирование + RAGAS-harness.
2. Агентность (\~1 нед): роутер запросов + ReAct-доискивание при низкой уверенности + Self-RAG.
3. Углубление по коду: граф конфигурации, LSP-разрешение символов, опц. дообучение BGE-M3.
4. Граф знаний (LightRAG) — только если глобальных запросов \> 20% трафика.
5. Оптимизация: апгрейд эмбеддера, Cohere Embed v4 для скринов, кеши.
### Пороги, меняющие план
- Faithfulness \< 0,80 с реранком → заземлённый промпт + Self-RAG.
- Context Recall \< 0,75 → проблема в чанкинге, НЕ в ретривере → перечанкить до смены моделей.
- Глобальные/реляционные запросы \> 20% → инвестировать в LightRAG.
- Корпус \> \~500k чанков → embedded Qdrant → сервер с квантизацией; пересмотреть стоимость ColBERT-индекса.

## 🧪 ЭТАЛОННЫЙ КЕЙС ДЛЯ EVAL — «почему проводка сменила счёт» — от 25.05.2026
> Реальный вопрос не-программиста: **«Почему в апреле этот документ делал проводку по одному счёту, а в мае — по другому?»** Коварен тем, что смешивает три РАЗНЫХ слоя причин — добавить в золотой eval-набор как образец «многослойного» вопроса. Стыкуется с ПРИОРИТЕТОМ №1.
**Причины лежат в РАЗНЫХ местах (не только в коде):**
1. **Изменился код** — поправили алгоритм проведения / подписку. Сюда же входит «**разные ветки одного алгоритма**» — документ в апреле и мае мог попасть в разные условия `Если` из-за отличий в реквизитах, при том же коде. (Ранее это был отдельный 4-й слой; сведено к 3 слоям 2026-05-27 по HLE-494 #6 — ветки всё-таки живут в коде.)
2. **Изменились данные/настройки** — счёт берётся не хардкодом, а из регистра сведений / учётной политики / счетов учёта номенклатуры / договора. Поменяли настройку → код тот же, проводка другая. **Кода это не касается.**
3. **Изменилось расширение/обновление** — приехало обновление ERP или включили/поправили расширение с `&Вместо`/`&После`.
**Что ассистент РЕАЛЬНО должен уметь (цель — не вердикт, а карта «где искать»):**
- найти, где формируется проводка (модуль проведения ИЛИ — чаще — подписка на событие / общий модуль; слепая зона из ПРИОРИТЕТА №1);
- протянуть цепочку до места выбора счёта и честно сказать: «счёт здесь НЕ зашит в коде — берётся из регистра ХХ / настройки → скорее всего поменяли настройку между апрелем и маем, проверьте её»;
- проверить наличие расширения с `&Вместо`/`&После` на методе и предупредить;
- перечислить условия `Если`, влияющие на выбор счёта, и подсказать, что сравнить в двух документах.
**Следствия для архитектуры (влияют на план):**
- **Подписки на событие ОБЯЗАТЕЛЬНЫ в графе** — иначе агент ищет проводку в модуле документа и не находит (она в подписке). Живое подтверждение ПРИОРИТЕТА №1.
- **Навык «это не в коде, это в данных/настройке»** — отдельный пункт поведенческого контракта (Linear 7). Без него агент галлюцинирует объяснение из кода там, где причина в настройке.
- **Точный ответ про КОНКРЕТНЫЕ документы требует двух источников за пределами выгрузки кода:** (1) история изменений конфигурации (git/хранилище, апрель→май — это про gitsync); (2) доступ к живой базе на оба документа — `vladimir-kharin/1c_mcp`, MCP внутри 1С. Решает, входят ли 1c_mcp и gitsync в ядро или остаются на потом.
