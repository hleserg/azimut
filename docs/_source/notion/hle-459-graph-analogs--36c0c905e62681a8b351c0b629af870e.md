> Источник: отчёт Claude Code по задаче [HLE-459](https://linear.app/hleserg/issue/HLE-459). Все факты — из исходников/доков двух репозиториев, по техническим деталям ссылки `файл:строка` сохранены в самом отчёте. Где из кода не следует — помечено «не определить».
## Что разбирали
Два графовых аналога. Вышло зеркально: один умеет код, другой — метаданные, и ни один не закрывает всё.
## 1c-mcp-metacode (Neo4j, Python) — закрытый код
Исходников нет (только Docker-образ + доки), LICENSE отсутствует. Но схема задокументирована в README.
- Есть **граф вызовов**: узлы `Module` и `Routine`, ребро `Routine →[CALLS]→ Routine`.
- Подписки доведены до обработчика: `EventSubscription →[HAS_HANDLER]→ Routine`.
- `Routine` квалифицирован через `Module→MetadataObject` — полная квалификация заложена архитектурно.
- Шаблоны: `resolve_qn` (qualified name → узел), `find_routines_by_name`, `call_graph_subtree` (N уровней).
- **Слепая зона:** сам алгоритм резолва одноимённых процедур (как строится `CALLS` с resolved-ссылкой) — закрыт, «не определить».
- LLM в построении графа нет (TEMPLATE_MODE_ONLY). LLM опционально только для NL→Cypher.
## bsl-graph (NebulaGraph, Kotlin) — открытый, MIT
Полный исходник, переиспользуемость высокая.
- **Граф вызовов НЕ реализован**: нет ни `Module`, ни `Routine`, ни `CALLS` (в roadmap, «Этап 1»).
- Сильные метаданные: единый тег `MDObject`, \~75 типов в свойстве `type` (`NodeType` enum, `MDObjectNode.kt:26-106`).
- Ребро `ATTRIBUTE` с именем реквизита (тип реквизита → объект), ребро `ACCESS` с перечислением прав.
- Подписки доведены только до источника (`EventSubscription → MDObject`), НЕ до обработчика.
- Детерминированный резолв типов реквизитов через `configuration.findChild(MdoReference)` — готовый образец lookup-логики.
- Парсинг через `bsl-mdclasses` (`MDOReader.readConfiguration()`). `bsl-parser` в зависимостях есть, но **не используется**.
- LLM в построении нет; детерминированность подтверждена по коду.
## Резолв одноимённых — ключевой вывод
В открытом коде эту проблему не решил никто. bsl-graph до неё не дошёл; metacode решил, но спрятал. Готового решения мы не унаследуем — проектируем сами.
## Хранилище
NebulaGraph для одних метаданных избыточен (SQLite + adjacency table тянет тот же объём). Графовая БД оправдывается только когда добавится код (\~50K Routine, \~500K CALLS), и тогда логичнее Neo4j (Cypher ближе к metacode, документирован лучше Nebula).

## 📌 Рекомендации к закреплению в дизайне (реш. 2.2)
1. **Ребро ****`CALLS`**** — ссылка на resolved Routine-узел, не плоская строка ****`callee_name`****.** Прямое лекарство от дыры bsl-atlas. Квалификатор узла: `OwnerType.OwnerName.ModuleType.RoutineName`.
2. **Схема метаданных — за основу из bsl-graph (MIT, код переиспользуем):** маппинг `NodeType` 1:1 к MDOType, ребро `ATTRIBUTE` с именем реквизита, ребро `ACCESS`. Точка входа парсинга — `MDOReader.readConfiguration()`.
3. **Схема кода — по образцу metacode (только идея, код закрыт):** узлы `Module`/`Routine`, рёбра `HAS_MODULE`, `DECLARES`, `CALLS`.
4. **Подписки доводить до обработчика** — `EventSubscription →[HAS_HANDLER]→ Routine` (как metacode), не только до источника. Закрывает кросс-объектную слепую зону bsl-atlas.
5. **Хранилище двухфазно:** старт на SQLite (метаданные, BFS в коде) → миграция на Neo4j при подключении кода. NebulaGraph не брать.
6. **Резолв одноимённых — остаётся открытым вопросом** (готового решения нет ни у кого). Проектируем поверх `configuration.findChild()` (детерминированный lookup из bsl-graph) + обход BSL-вызовов через `bsl-parser`. Завязано на HLE-456 (ещё In Progress) — финально 2.2 закрывать рано.
