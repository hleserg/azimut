> Принятые решения и предложения по борьбе с галлюцинациями и стратегии поиска. Синхронизировано с [decisions.md](http://decisions.md) в гите.
## ПРИНЯТЫЕ РЕШЕНИЯ
> 📋 **Действуют:** Р1, Р2, Р3, Р5, Р6, Р7 + предложения П1, П2, П3. **Р4 (был «честный тупик») снят 2026-05-26 — заменён на Р7** (фолбэк = смена режима, не капитуляция). Если ищешь Р4 — его нет осознанно, см. `_resolutions.md` #10.
### Р1. Проверка противоречивости материалов ПЕРЕД выдачей
- Перед выдачей выборки агенту-генератору проверяем извлечённые чанки на взаимную противоречивость.
- При обнаружении противоречия инкрементируем метрику противоречивости в Sentry.
- Уточнено позже (см. Р6): вместо «убрать оба» — иерархия источников.
- ОТКРЫТО: как технически детектировать противоречие, порог, что если их много.
### Р2. Faithfulness как отдельная метрика
- Разводим relevance ретривера (нашли ли релевантное) и faithfulness генерации (опирается ли ответ строго на найденное).
- Не мерить «общее качество» одним числом.
### Р3. LLM-судья со спан-привязкой (Linear задача 12)
- Судья сравнивает спаны ответа со спанами retrieved-контекста, флажит неподкреплённые утверждения.
- Работает с облачным Claude (текстовый метод, не лезет в потроха модели).
### Р5. Контроль ретривинга — на СЕРВЕРЕ, не на агенте
- Управление добором данных на сервере, т.к. там всё ИЗМЕРИМО и КАЛИБРУЕМО:
	- нижняя планка релевантности (отсечка мусора до генерации);
	- триггер добора (если top-N не набрал суммарной релевантности — искать ещё);
	- потолок окна (добираем пока влезаем в бюджет токенов);
	- всё логируется/мониторится/калибруется.
- Агенту оставляем только семантическую переформулировку запроса для 2-го прохода.
- Обоснование: у агента нет измеримых критериев «надо ещё / хватит» и нет управляемости; он может проигнорить даже хорошие данные.
- Соответствует Self-RAG 2026: сервер держит измеримые гейты, агент помогает по смыслу.
- КОГДА добор ВРЕДЕН: вопрос уже закрыт; риск выйти за окно (lost-in-the-middle); добор тащит низкорелевантный шум.
### Р6. Иерархия источников при конфликте
- Приоритет: 1) КОД в базе  2) доки из конфига (встроенная справка)  3) официальные доки с ИТС.
- НЕ выкидываем конфликтующие — ОБЪЯСНЯЕМ все варианты по порядку приоритета, помечая первый как наиболее вероятный.
- Противоречие всё равно инкрементит метрику в Sentry.
### Р7. Фолбэк = переключение режима, НЕ капитуляция
- «Не нашёл в индексе» — НЕ конец ответа, а триггер перехода в дип-ресёрч в интернете.
- «Копай сам» = обезьянка из колл-центра; бот бесполезен. Нормально: «не знаю» → иду разбираться.
- НЮАНС: интернет по 1С — минное поле. Дип-ресёрч тащит ТОТ ЖЕ КОНТРАКТ: показать источник, пометить «из интернета, не из базы», не выдавать за истину.
- Заменяет снятый Р4 («честный тупик» оказался недостаточным).
## ПРЕДЛОЖЕНИЯ (не финальные)
### П1. Детектор relevance-высокая / groundedness-низкая (бот красиво врёт при хороших данных) — 3 слоя действий
- Уровень 1 (сильный, «лишение возможности»): судья проверяет ответ ДО выдачи. Ниже порога → ответ НЕ уходит, агенту возврат «перепиши строго по чанкам или скажи что не нашёл». Стенка, не шлепок.
- Уровень 2 (страховка): после переписи всё равно низко → ответ с ПЛАШКОЙ «частично из общих знаний, в источниках не подтверждено».
- Уровень 3 (канарейка): доля негрунтованных копится в Sentry.
### П2. Второй проход ретривера (re-retrieval) при неуверенности
- Инициатор — агент, исполнитель — ретривер. ОТКРЫТО: триггер.
### П3. Оценка достаточности запроса + помощь агенту в диалоге
- Измерение, которого нет в Р5: оценка САМОГО ЗАПРОСА на входе («хорошо ли нашли» ≠ «отвечабелен ли вопрос»).
- (а) размазанный вопрос не молотить поиском вхолостую, вернуть «слишком общий»;
- (б) НАМЁКИ агенту что переспросить на основе РЕАЛЬНОГО индекса (агент сам не знает что в данных, иначе уточняет про типовую 1С = издевательство);
- (в) проверка что переформулированный агентом запрос не уехал от исходного.
- Коррелирует с Adaptive RAG: классифицируем сложность И отвечабельность.
## ФОН: причины галлюцинаций при хороших данных
- Внутреннее знание перебивает контекст (Claude видел тонны ТИПОВОЙ 1С, тянет привычное вместо кастомного).
- Чанк есть, но неудобно собран → модель «дорисовывает».
- Соблазн полноты: чанк отвечает на 80%, модель достраивает 20% из головы (самый коварный).
## ТЕРМИНЫ (чтобы не путаться)
- **Ретривер** — тупой механизм поиска (BGE-M3 → Qdrant → top-N). НЕ агент, ничего не решает.
- **Агент-генератор** — облачный Claude. Пишет ответ, решает «мало данных, искать ещё».
- **MCP-сервер** — посредник: дёргает ретривер, упаковывает, отдаёт агенту.
