## Что разбирали
`FSerg/mcp-1c-v1` — chunking-стратегия и Qdrant-схема референсного MCP-сервера (TS-оболочка + Python-ядро). Цель — вытянуть идеи для нашего чанкинга кода (тема 2) и поискового стека (тема 3). **Код не берём** (и по архитектуре — реш. 1.3, и до разрешения лицензионного конфликта выше).
## Их chunking (как устроено)
- **Один объект конфигурации = одна точка в Qdrant.** Без нарезки внутри объекта, без overlap, без sliding window. Вход — ZIP с `objects.csv` (колонки: Имя/Тип/Синоним/Файл) + markdown-описания. Каждая строка CSV → чанк (`loader.py:84-116`).
- **Два вектора на объект, эмбеддятся только имена:** `object_name` = точное имя («Справочник.Номенклатура», `loader.py:107`); `friendly_name` = `"{тип}: {синоним}"` («Справочник: Номенклатура», `loader.py:110`).
- **Полное описание (****`doc`****) НЕ эмбеддится** — лежит в payload и возвращается при поиске. Причина (`article.md:72`): векторизация полного описания давала хуже — семантически похожие объекты (расходный ордер ≈ расходная накладная) обгоняли точные совпадения по имени.
- Батчи: ROW_BATCH_SIZE=250, EMBEDDING_BATCH_SIZE=50.
## Их Qdrant-схема
- Одна коллекция на конфигурацию (дефолт `1c_rag`); имя переопределяется HTTP-заголовком `x-collection-name` → несколько конфигураций в одном Qdrant (`mcp_server.py:166-170`).
- **Named vectors (два на точку):** `object_name` и `friendly_name`, оба Distance.COSINE, **`on_disk=True`** (экономия RAM). Размерность динамически от сервиса (384 дефолт, 1024 у jina/Qwen). Явных HNSW/quantization нет — дефолты Qdrant.
- **Payload:** `object_name`, `object_type`, `doc` (полный markdown), `file_name`, `friendly_name`. ID точки — UUID4.
## Их поиск (RRF)
- **Сливают ДВА dense-вектора** (`object_name` + `friendly_name`) — **НЕ dense+sparse, НЕ BM25**. Один и тот же query-вектор идёт в оба prefetch.
- **Реализация — Qdrant native RRF** через `Prefetch[...] + FusionQuery(fusion=Fusion.RRF)` (`mcp_server.py:110-128`). Точный API уже показан — готовый образец реализации.
- `PREFETCH_LIMIT_MULTIPLIER=3` (рекомендуют 2–5): при limit=5 берётся 15 кандидатов на вектор → RRF → топ-5.
- Разделение задач эмбеддинга: индексация `task=retrieval.passage`, поиск `task=retrieval.query` (для моделей типа jina).
## Эмбеддер
Дефолт `all-MiniLM-L6-v2` (384d). Альтернативы в конфиге: jina-embeddings-v3 (1024d, task), Qwen3-Embedding-0.6B (1024d). FastAPI-сервис + SentenceTransformer, замена модели — правка config.json + перезапуск.
