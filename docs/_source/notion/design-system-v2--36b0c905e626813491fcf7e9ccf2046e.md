## Зачем эта версия (главный сдвиг)
ТЗ v1.1 проектировало почти всё **с нуля**: свой парсер XML-конфигурации, свой индексатор документации, свой поисковый слой, свой реестр файлов, свой OCR-конвейер. Это много кода, рисков и времени до первого работающего прототипа.
**Решение Сергея:** по максимуму опереться на **готовые решения** (форки, библиотеки, MCP-серверы из экосистемы 1С и RAG). Цель v2.0 — собрать систему из проверенных «кирпичей», а писать руками только то, чего нет в опенсорсе или что является нашим конкурентным ядром.
Исследования (см. «Открытые вопросы») уже нащупали реалистичную основу: **форк ****`bsl-atlas`** (AGPL-3.0, ✅ проверено по LICENSE репо — форк разрешён лицензией) + \~10 готовых библиотек под каждый пробел. Эта версия дизайна — про то, чтобы осознанно это подтвердить и достроить.
## Что НЕ меняется (наследуем из v1.1 без пересмотра)
- **Назначение.** Не «улучшенный поиск», а компетентный спец: собирает контекст, копает в источниках, отвечает ТОЛЬКО из найденного, сам определяет нехватку данных и идёт искать. Планка — «почти всегда сам в источники, по делу», не 100%.
- **Пользователи и приватность.** Сергей и мама; локальный запуск; персональные данные маскируются до отправки в облако.
- **OSS под AGPL-3.0.** Проект открытый, лицензия AGPL-3.0 (наследуется от bsl-atlas, форк официально разрешён лицензией). Цель монетизации сейчас НЕ стоит — готовы делиться с программистским миром. Идём по best-practice, а не прячем ноу-хау. Если монетизация когда-то появится — через услуги (внедрение, настройка под кейс клиента, поддержка) поверх открытого кода, не через закрытость. Этот принцип задаёт тон всем темам: открытие кода — не риск, а цель.
- **Источники знаний (легальные).** Выгрузка конфигурации в XML, офлайн-ИТС, портал техдокументации платформы, личные документы (по желанию).
- **Базовый стек как ориентир.** Python ≥3.11, uv, FastMCP; разговорная модель — облачная и **подключаемая через адаптер** (Claude / DeepSeek / Qwen — выбор по умолчанию открыт, см. реш. 1.8); эмбеддинги BGE-M3 (локально); хранилище Qdrant; реранкинг — по умолчанию локальный BGE-reranker, Cohere Rerank v4 как опциональный апгрейд (грант по умолчанию не тратим — см. реш. 1.8). Локальные **разговорные** LLM исключены (рабочие ПК не тянут). Конкретику по каждому пункту пересматриваем в темах ниже.
- **Принцип «дотошность — в сервере, не в модели».** Сервер отдаёт полные разделы, разворачивает вложенность, сигналит «есть ещё N разделов».
## Уже принятые решения (фон для всех чатов)
Это решения из дока «Решения (decisions) — борьба с галлюцинациями и поиск». Их НЕ переоткрываем без причины — используем как отправную точку.
- **Р2.** Faithfulness (опора ответа на найденное) и relevance ретривера — РАЗНЫЕ метрики, не мерить одним числом.
- **Р3.** LLM-судья со спан-привязкой: сравнивает спаны ответа со спанами контекста, флажит неподкреплённое (облачный Claude, текстовый метод).
- **Р5.** Контроль добора данных — на СЕРВЕРЕ (измеримо/калибруемо): нижняя планка релевантности, триггер добора, потолок окна. Агенту — только семантическая переформулировка запроса для 2-го прохода.
- **Р6.** Иерархия источников при конфликте: 1) код в базе → 2) встроенная справка из конфига → 3) официальные доки ИТС. Не выкидывать конфликт — объяснять все варианты по приоритету, помечать первый как вероятный; инкремент метрики противоречивости в Sentry.
- **Р7.** Фолбэк = смена режима, не капитуляция. «Не нашёл в индексе» → дип-ресёрч в интернете С ТЕМ ЖЕ контрактом (показать источник, пометить «из интернета», не выдавать за истину).
- **П1 (предложение).** Детектор «релевантность высокая / groundedness низкая» — 3 слоя: (1) судья режет ответ до выдачи; (2) плашка «частично из общих знаний»; (3) канарейка доли негрунтованного в Sentry.
- **П3 (предложение).** Оценка достаточности самого запроса + подсказки агенту что переспросить на основе РЕАЛЬНОГО индекса; проверка что переформулировка не уехала от исходного вопроса.
- **Р1 (открыто).** Метрика противоречивости — как технически детектировать, порог, поведение при множестве конфликтов.

## 1. Фундамент: готовая основа и стратегия «форк vs сборка»
*Статус: ✅ зафиксировано · Linear: тема 1 (фундамент, блокирует остальные)*

### ✅ Реш. 1.1 — Основа: форкаем `bsl-atlas`
Подтверждено (2026-05-26). Форкаем `bsl-atlas` как ядро системы. **Лицензия: AGPL-3.0** (проверено напрямую в репо — LICENSE на ветке master содержит GNU Affero GPL v3). Форк официально разрешён лицензией; «разрешение автора / некоммерческое» из ранней версии фундамента — НЕВЕРНО, действует публичная AGPL-3.0. Наш форк (Азимут) сам идёт под AGPL-3.0 — это согласуется с целью проекта быть OSS (см. «Что НЕ меняется»).
Почему именно так:
- Единственный зрелый проект, который уже понимает структуру кода 1С: парсит **текст** BSL + строит детерминированный **граф вызовов** (узлы — процедуры/функции, рёбра — кто кого зовёт). Это самая дорогая и рискованная часть — ровно то, ради чего затевалась v2.0 «по максимуму готовое».
- Большие RAG-платформы (RAGFlow, kotaemon, Onyx) BSL не понимают — годятся только как UI-морда поверх нашего MCP, не как мозг.
- `FSerg/mcp-1c` (ближайший аналог) индексирует только метаданные структуры конфигурации, **не тексты модулей** → в ядро не годится; рассматриваем отдельно как референс (реш. 1.3).
- Стек-сосед правильный: Python + MCP + русский/Ollama. Лицензия AGPL-3.0 (copyleft с сетевым пунктом §13). Для нас это не ограничение, а естественный выбор: проект сам OSS, монетизации-через-закрытость не планируем.
На что закладываемся (раскрывается в решениях ниже):
- миграция стека форка ChromaDB + SQLite → наш Qdrant + BGE-M3 + Cohere (реш. 1.4);
- нет дружелюбного UI «для мамы» (реш. 1.7);
- проверить покрытие расширений и внешних обработок, а не только основной конфы (детали — тема 2);
- некоммерческий триггер пересогласования с автором → лицензионный чек-лист (реш. 1.10). **Снят:** AGPL-3.0 публична, отдельного разрешения/пересогласования не требует.
### ✅ Реш. 1.2 — Роль `bsl-atlas`: только «движок понимания кода»
**Решение (2026-05-26).** Форк `bsl-atlas` («Азимут») используется строго как движок понимания кода 1С, а не как готовый продукт. Делим компоненты форка на три категории.
**Что оставляем (ради этого и форк):** парсер текста BSL (разбор модулей в структуру); детерминированный граф вызовов (кто кого вызывает); каркас MCP-сервера (точка подключения к Claude); docker-упаковку; поддержку русского и локального инференса через Ollama.
**Что меняем/выкидываем:** хранилище ChromaDB → Qdrant (см. реш. 1.4); эмбеддер → BGE-M3 (тема 3); добавляем реранкер (Cohere v4 + локальный BGE-фолбэк) — в форке его нет вообще.
**Что дописываем своим кодом** (конкурентное ядро, детально — реш. 1.9): поведенческий контракт; серверный контроль добора (Р5); сборку полных разделов + «сигнал полноты»; иерархию источников при конфликте (Р6); eval / LLM-судью.
**Почему так.** Ценность `bsl-atlas` для нас — ровно один дорогой и рискованный кусок: парсер BSL + граф вызовов, который мы НЕ хотим писать с нуля. Остальное в форке либо не дотягивает до нашего стека (хранилище, эмбеддер), либо отсутствует (реранк, анти-галлюцинационная логика). Поэтому относимся к форку как к двигателю, на который ставим свой кузов, а не как к машине под ключ.
**Последствия.** Объём «нашего» кода смещается с парсинга/графа (берём готовое) на оркестрацию поиска и поведение агента (пишем сами). Это и есть смысл v2.0: не строить заново то, что уже есть, и вкладываться только в конкурентное ядро.
### ✅ Реш. 1.3 — `FSerg/mcp-1c-v1`: референс архитектуры
**Решение (2026-05-26).** Используем `FSerg/mcp-1c-v1` как референс архитектуры, не как кодовую основу.
**Что это.** Ближайший публичный аналог нашего проекта: MCP-сервер с RAG-описанием структуры конфигурации 1С (TypeScript, \~152★, Qdrant + RRF, Docker Compose).
**Почему не скелет.** (1) Язык — TypeScript, у нас Python ≥3.11 + FastMCP: код напрямую не переиспользуется. (2) Индексирует только **метаданные структуры** конфигурации, а не тексты модулей BSL/расширений/обработок — не закрывает наш Приоритет №1 (понимание кода). По умолчанию all-MiniLM (не BGE-M3), реранкера нет. Как ядро бесполезен.
**Почему не мимо.** У него уже собрано то, что полезно подсмотреть: рабочий гибрид **RRF** (Reciprocal Rank Fusion — слияние плотного и разреженного списков результатов в один рейтинг) поверх Qdrant; docker-оркестрация (Qdrant + эмбеддер + FastMCP + Streamlit); и **схема payload по метаданным конфигурации** — метаданные нужны и нам, как дополнение к графу кода. Плюс это живой ориентир «чем мы от него отличаемся»: мы добавляем тексты модулей + граф вызовов + реранк + поведенческий контракт.
**Что берём.** Идеи и архитектурные решения (структура payload, склейка Qdrant+эмбеддер+MCP, RRF-слияние). **✅ ПО ЛИЦЕНЗИИ (исправлено 2026-05-26):** у `FSerg/mcp-1c-v1` **ЕСТЬ LICENSE с MIT** (Copyright (c) 2025 Sergey Filkin) — подтверждено отчётом HLE-460 и согласовано Сергеем. Ранняя пометка «LICENSE НЕТ» была НЕВЕРНА (это ровно тот класс ошибок, против которого введено правило источников). MIT разрешает использование/копирование/модификацию кода при сохранении уведомления об авторстве, т.е. формально код брать можно. Но **архитектурно код всё равно не берём** — берём только идеи/подходы, реализация с нуля на Python под наш стек (причины ниже: TS-стек, индексирует только метаданные структуры, а не тексты модулей).
### ✅ Реш. 1.4 — Миграция стека: гибрид (дымовой прогон на Chroma → сразу Qdrant)
**Решение (2026-05-26).** Мигрируем хранилище и эмбеддер форка на целевой стек (Qdrant + BGE-M3 + Cohere) — но не на старте, а сразу после одного дымового прогона форка в его родном виде.
**Контекст.** `bsl-atlas` ездит на ChromaDB (векторное хранилище) + SQLite (метаданные). Наш целевой стек — Qdrant (embedded локально) + BGE-M3 (эмбеддер) + Cohere Rerank (реранк).
**Почему миграция обязательна.** ChromaDB не держит нативно то, на чём стоит наш поиск: гибрид dense+sparse от BGE-M3 в одной коллекции и несколько векторов на точку (multi-vector — нужно для shadow-эксперимента и late-interaction/ColBERT). Это уже зафиксировано в ADR. Оставить ChromaDB = упереться в потолок ровно там, где начинается тема 3. Значит вопрос не «мигрировать ли», а «когда».
**Почему гибрид по времени (не сразу и не потом).** (1) Решение 1.1 стоит на допущении «bsl-atlas реально парсит нашу ERP и строит граф на нашем объёме». Это допущение дешевле всего проверить на форке as-is, ДО вложений в миграцию: один дымовой прогон на ChromaDB — убедиться, что движок не врёт (парсит модули, строит граф, не давится на объёме ERP — десятки тысяч модулей). (2) Но «этапом» в смысле «писать свой код под ChromaDB» — нет: двойная работа + риск, что «временное» хранилище присосётся. Правило: **ни строчки нового кода под Chroma**. Прошёл дымовой прогон → миграция на Qdrant+BGE-M3 становится первой задачей фазы, дальше всё на целевом стеке.
**Риск к проверке на том же прогоне.** Насколько жёстко `bsl-atlas` сцеплен со своим хранилищем: если слой storage размазан по коду, миграция дороже, чем кажется. По докам не видно — оценить руками. Замер влияет на оценку трудозатрат фазы.

### ✅ Реш. 1.5 — MCP-шлюз `onec-mcp-universal`: отложить до темы 7
**Решение (2026-05-26).** В фундамент не берём. Логируем как кандидата на тему 7 (онлайн-деплой / мульти-аренда). На локальном сценарии — один наш MCP-сервер (Азимут), без агрегатора.
**Что это.** `AlekseiSeleznev/onec-mcp-universal` — \~400 строк Python (Starlette), MCP-гейтвей-агрегатор: один MCP-вход, маршрутизирующий запросы на несколько backend-1С-MCP-серверов. Предлагался как шлюз перед Азимутом, чтобы «сократить свой MCP-сервер».
**Почему откладываем.** (1) Экономия мала: сам гейтвей крошечный, а «свой MCP-сервер» (задача 6) дорог не транспортом, а нашей оркестрацией (Р5, сигнал полноты, Р6) — её гейтвей не делает. (2) Наше ядро — это и есть оркестрация запроса; прятать её за чужим дженерик-агрегатором архитектурно неверно: контроль добора и иерархия источников должны жить в нашем сервере. (3) Технически не нужен локально: Claude Desktop сам подключает несколько MCP-серверов (перечисление в конфиге), так что держать Азимут + справочник платформы рядом можно без шлюза.
**Когда вернёмся.** Гейтвей осмыслен, когда (а) backend-серверов реально много и/или (б) на VDS нужен единый вход с авторизацией по конторе (tenant_id в JWT). Тогда решим: брать `onec-mcp-universal` за основу или писать тонкий свой. Прямо стыкуется с темой 7 (авторизация фирмы, мульти-аренда).
### ✅ Реш. 1.6 — `alkoleft/mcp-bsl-platform-context`: берём в фундамент
**Решение (2026-05-26).** Подключаем как отдельный MCP-сервер рядом с Азимутом (в Claude Desktop). Бесплатно, без написания кода.
**Что это.** MIT, \~130★. Отдаёт через MCP справочник синтаксиса платформы 1С — содержимое `shcntx_ru.hbk` (встроенная справка: глобальные методы, API объектов типа `Запрос`, `ТаблицаЗначений` и т.д.).
**Почему берём — закрывает отдельный пласт знания и вектор галлюцинаций.** Азимут понимает код *твоей конфигурации* (кастомный BSL), но не знает встроенный **API платформы** (сигнатуры и поведение `ПолучитьОбщийМакет()`, `Записать()` и пр.). Это два разных слоя: «что написано в этой ERP» vs «что вообще умеет язык/платформа». Именно по платформенным методам облачный Claude склонен галлюцинировать из памяти (видел массу типовой 1С — прямо причина из раздела «причины галлюцинаций при хороших данных»). ### ✅ Реш. 1.8 — Разговорная модель, анти-галлюцинации и где крутится LLM
**Решение (2026-05-26).** Несколько связанных правил.
**(1) Разговорной LLM внутри MCP-сервера НЕТ.** Разговорный мозг — это модель клиента (облачная). Сервер свою генеративную модель не держит.
**(2) Разговорная модель — облачная и подключаемая (адаптер), с самого старта.** Локально не тянем (см. 1.7). Из-за доступа из РФ (Claude = VPN) закладываем замену Claude на DeepSeek/Qwen сразу — поэтому адаптер к любой разговорной модели входит в фундамент, а не откладывается. Claude — один из вариантов, не захардкоженный.
**(3) Анти-галлюцинации — модель-независимые, «для клиента, не для Claude»:** измеримые гейты на сервере (Р5 добор, Р6 иерархия, сигнал полноты) — модель-агностичны (наш принцип «дотошность в сервере»); поведенческий контракт = портативный системный промпт под любую модель, без Claude-специфичных трюков; судья (Р3/П1) — независимый компонент, не «модель проверяет сама себя».
**(4) Политика «где крутится LLM» — по умолчанию НЕ тратим грант Cohere на пользователя:** лёгкие модели (эмбеддер BGE-M3, реранкер BGE-reranker-v2-m3, лёгкая проверка заземлённости для судьи) — **локально по умолчанию** (помещаются на CPU/слабую машину; грант не расходуется); тяжёлую генеративную работу, которую локально не потянуть (разговор; при необходимости — тяжёлый LLM-судья, преамбулы Contextual Retrieval), отдаём облаку — **по умолчанию модель самого клиента** (её пользователь и так оплачивает), Cohere опциональный апгрейд по решению Сергея, не дефолт; грант Cohere (\$1000) — рычаг Сергея, а не автоматическая зависимость каждого пользователя.
**Честная оговорка.** Слабая разговорная модель будет следовать контракту хуже фронтира; серверные гейты ловят плохой поиск, но «модель проигнорила хороший контекст» — отчасти качество модели. Какие модели «достаточно умны» — открытый вопрос ниже, валидируем eval-ом (тема 6).
**Ресурсная развилка.** Дефолт-локально для лёгких моделей грузит слабый ПК (BGE-M3 + BGE-reranker + Qdrant одновременно) — риск RAM/латентности, под мониторингом канарейки; для слабых машин это аргумент в пользу онлайн/VDS-варианта (тема 7) либо выборочного выноса реранка в Cohere.

Что решаем: чанкинг модулей без разрыва цепочек проведения, AST-чанкинг, детерминированный граф вызовов, подписки на события, слепая зона внешних обработок, сбор кода (конфа + расширения + .epf/.erf), чем парсим BSL.
> **Метод темы 2 (важно):** все решения опираются на ФАКТЫ из исходников `bsl-atlas` (read-only анализ Claude Code), а не на догадки. Где факта из кода нет — помечаем «проверить на дымовом прогоне», не выдумываем.
### 📂 Что `bsl-atlas` реально делает (факты из кода, не из README)
Прочитаны исходники форка. Зафиксировано как отправная точка:
- **Чанкинг — структурный**, по функциям: одна процедура/функция = один чанк, если тело ≤ **2000 символов** (`vector_indexer.py:499–503`). Функции \< 5 строк пропускаются. Для тел \> 2000 символов — `RecursiveCharacterTextSplitter` (chunk_size=1000, overlap=100, сепараторы `\n\n / \n / . / пробел`), т.е. **символьная резка, НЕ по структуре**. Fallback при падении парсера — весь файл тем же сплиттером.
- **Граф вызовов есть в SQLite**: таблица `calls(caller_id, callee_name)` с индексами по обоим полям. Это настоящие рёбра. НО `get_function_context` делает **лукап на 1 уровень** (прямые «кто зовёт / кого зовёт»), многоуровневого обхода (recursive CTE / итерации) в коде НЕТ.
- **Парсер BSL — двойной**: tree-sitter (грамматика `alkoleft/tree-sitter-bsl`, `bsl.so`) первичный; regex-fallback если `.so` нет. В Docker `.so` компилируется → tree-sitter активен по умолчанию.
- **Подписки на события — частично**: `ПодпискаНаСобытие` индексируется как ТИП объекта метаданных (TXT-парсер), но кросс-объектных рёбер «событие → обработчик в другом модуле» НЕТ (нет в XML-парсере, нет таблицы связи в SQLite). **Слепая зона для проводок.**
- **Внешние обработки ****`.epf/.erf`**** — НЕ читаются** вообще: индексатор берёт только `*.bsl` + метаданные `*.txt/*.xml`. Подтверждено grep'ом.
- **Асинхрон — полная слепая зона**: `НачатьВыполнение`, `ОписаниеОповещения` попадают в `calls` как обычные вызовы, семантика «это callback» не хранится, цепочка через них рвётся.
- **Источник чанка (конфа vs расширение)** — явного поля НЕТ. Определяемо косвенно по наличию `cfe/` в пути (`source_file`). Парсер метаданных обходит `cfe/<Ext>/` отдельно.
### ✅ Реш. 2.1 — Чанкинг кода: достраиваем структурную резку поверх Азимута
**Решение (2026-05-26).** Базовый структурный чанкинг Азимута (функция = чанк) берём. Но символьный fallback для длинных тел НЕ годится под Приоритет №1 и заменяется на детерминированную структурную резку. LLM для РЕЗКИ не используем (недетерминированность = источник галлюцинаций, противоречит ядру проекта); LLM остаётся кандидатом на ОБОГАЩЕНИЕ смыслом (Contextual Retrieval, тема 3/5).
**Почему символьный fallback — проблема.** Порог 2000 символов ≈ 25–30 строк кода 1С. Реальная `ОбработкаПроведения` в типовой ERP — сотни строк → почти всегда срабатывает символьная резка → цепочка рвётся посреди `Если`, теряется заголовок процедуры, агент видит обрывок. Это ровно сценарий Приоритета №1. Вывод: «чанкинг as is» под длинные процедуры НЕ проходит.
**Что делаем (детерминированная структурная резка, границы даёт tree-sitter):**
1. **Функция ≤ порога → один чанк** (как в форке).
2. **Функция \> порога → режем по top-level блокам** (`Если`/`Цикл`/`Попытка`/`Область`), НИКОГДА не разрывая блок посередине. Цель \~**1000–1500 токенов** (считаем в ТОКЕНАХ, не символах — русские идентификаторы в символах/токенах считаются по-разному, окно модели в токенах).
3. **Сам блок больше порога → рекурсивно заходим внутрь** и режем по его вложенным блокам. Метки иерархические: блок `4`, внутри — `4.1`, `4.2`. **Стоп на глубине 1–2 уровня** (глубже плодим бессмысленные микрочанки).
4. **«Шапка контекста» в КАЖДОМ подчанке**: процедура + путь блока + краткое «внутри чего» (напр. «ОбработкаПроведения, блок 4.2: внутри Если ВалютаУчёта \<\> ВалютаДокумента»). Без шапки подчанк — сирота.
5. **Крайний fallback** (дерево не построилось / блок неделим) — символьный сплит, но это редкий край, не базовый путь.
**Запросы в строках (****`Запрос.Текст = "..."`****) — отдельный тип контента.** Внутри BSL-строки сидит язык запросов 1С; tree-sitter видит её как атом и не разбирает. Для длинных запросов (бывают по 1000+ строк, пакеты с временными таблицами):
- **Режем по ****`|;`** — официальный разделитель пакетов запроса (значащий символ — `;` после `|`; слэши/`|`-префиксы — оформление, чистим). Каждый пакет (`ВЫБРАТЬ`/`ПОМЕСТИТЬ`) = логическая граница.
- Один пакет всё ещё \> порога → символьный fallback этого пакета (редкий край).
- **Метка чанка-запроса обязательна**: «блок n из N, процедура X, модуль Y».
- **Лёгкие метки временных таблиц (одна регулярка, не парсер запросов):** что пакет `создаёт` (`ПОМЕСТИТЬ X`) и `читает` (имена из `ИЗ`/соединений). Помогает поиску «откуда берётся таблица Z». Если читаемой таблицы нет ни в одном `ПОМЕСТИТЬ` этого же запроса → флаг `external_temp_table` (пришла извне).
- **Связи временных таблиц между пакетами в индексе НЕ строим** — восстанавливаются на этапе ответа: агент, найдя любой пакет, по метке тянет процедуру целиком и разматывает запрос сам (принцип «дотошность в сервере»).
**`МенеджерВременныхТаблиц`**** — честная слепая зона (runtime, не статика).** Менеджер течёт через процедуры/модули как параметр/свойство структуры; где он наполнен — определяется трассой выполнения, статически не доказуемо (природа как у асинхрона/подписок: поток данных, а не вызовов). Граница покрытия:
- **Берём (дёшево, детерминированно):** Уровень 0 — флаг `external_temp_table` на чанке (см. выше); Уровень 1 — пометить процедуры, которые создают `Новый МенеджерВременныхТаблиц` или принимают его параметром (пул кандидатов «где искать наполнение»).
- **НЕ строим:** Уровень 2 — межмодульный data-flow трекинг менеджера (исследовательская задача, недетерминированно, утягивает в «свой сложный анализатор» — против v2.0). Записано в слепые зоны; поведенческий контракт (тема 5) должен честно говорить «источник статически не доказуем, вот кандидаты». Достоверный путь для конкретного документа — живые данные базы (`1c_mcp`, см. вопрос ниже).
**Последствия / зависимости.** Структурный сплиттер + резка запросов + метки ВТ — это НАШ код поверх форка (не было в Азимуте). Параметры порога окончательно калибруем на дымовом прогоне (реш. 1.4) и eval (тема 6). Размер в токенах стыкуется с темой 3 (эмбеддер BGE-M3).

> ⚠️ **Честная оговорка (правило про код 1С).** Формулировка из ресёрча «канонические образцы платформенных событий» НЕ проверена по исходникам и принята с оговорками: (1) БСП — это библиотека *подсистем* (инфраструктура), а не учётная конфа — есть ли там проведение с проводками в товарном смысле, НЕ подтверждено, надо проверять. (2) «Справочник платформенных событий» из БСП НЕ выводится: БСП показывает *примеры реализации* обработчиков, но *перечень* событий платформы — это синтакс-помощник (`bsl-help-toc-parser` / `.hbk`), другой источник. Поэтому БСП = возможный источник ПРИМЕРОВ кода, НЕ источник перечня событий.
