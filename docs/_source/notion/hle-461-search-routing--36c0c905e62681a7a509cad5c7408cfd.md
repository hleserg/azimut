## Что это
Не граф и не индекс, а набор **правил поведения ИИ-агента-разработчика 1С**. Ценность — это чужой обкатанный ответ на наш открытый вопрос темы 2: в каком порядке дёргать инструменты поиска по коду.
## Fallback-цепочка (главное)
Официальная формулировка (`tooling-playbooks.md`): `graph → code-metadata → grep=true retry → Grep`. Применяется **только** к поиску по исходникам проекта (не к внешним знаниям).
1. **Граф** (Neo4j/Cypher) — высший приоритет, самый мощный.
2. **code-metadata** (FTS + semantic индекс) — когда граф недоступен или вернул пусто/нерелевантно.
3. **code-metadata с ****`grep=true`** — substring-retry **внутри MCP-индекса** (НЕ файловый grep). Для точных идентификаторов, фрагментов запроса, путей метаданных, имён обработчиков, текста ошибок.
4. **файловый ****`Grep`****/****`rg`** — только когда всё MCP исчерпано, с обязательной пометкой «что пробовал и почему не помогло».
**Ключевой триггер перехода вниз:** не «пусто один раз», а пусто/non-actionable **дважды на существенно разных запросах** по одной цели. Защита от соскакивания с мощного инструмента из-за одной неудачной формулировки.
**Принцип остановки:** стоп, как только собранных данных достаточно; каждый вызов обязан закрывать конкретный пробел и не дублировать прошлый. No-change repeats запрещены.
## Диспетчер — два независимых вектора
- **Поиск по проекту** — каскад выше, с файловым grep в хвосте.
- **Внешние знания** (платформа, БСП, шаблоны, живая ИБ) — без grep-эквивалента, вызываются только по нужде.
На наш «зоопарк»: Азимут = их граф-сервер, справочник платформы = их docs-сервер. Сервер считается доступным только если его инструменты реально экспонированы в сессии (наличие в конфиге ≠ доступность).

## 📌 Рекомендации к закреплению — реш. 2.3 «Роутинг поиска по коду»
1. **Каскад ****`граф → индекс(code-metadata) → индекс+grep → файловый grep`** как основная стратегия поиска по исходникам. Файловый grep — последний резерв, не первый инструмент.
2. **Триггер перехода вниз** — пусто/нерелевантно **дважды на разных запросах** по одной цели, а не с первой неудачи. Записать явно.
3. **`grep=true`**** как промежуточный уровень** — substring-retry внутри индекса (точные идентификаторы, имена обработчиков, текст ошибок), отдельно от файлового grep. Ляжет на наш AST/индекс-слой.
4. **Разделить диспетчер на два вектора:** «поиск по проекту» (каскад с grep) и «внешние знания» (платформа/БСП/живые данные — по нужде, без grep-хвоста). Не смешивать.
5. **Уровни детализации ответа (аналог L0–L3)** — управление токенами, от «только имя+score» до «полный код». Завязать на чанкинг (реш. 2.1).
6. **Принцип «стоп когда достаточно» + запрет no-change repeats** — в поведенческий контракт инструмента.
**Связь с темой 5:** блок `Context sources` в ответах с фактами — усиление нашего сквозного анти-выдумка-правила (явно перечислять, что проверено через Азимут/справочник).
> ⚠️ **Зависимость:** у них каскад завязан на граф вызовов, которого у нас пока нет (реш. 2.2, ждёт HLE-456). Полностью каскад заработает только когда граф будет готов. До того фактически `индекс → grep`, верхний уровень — задел.
