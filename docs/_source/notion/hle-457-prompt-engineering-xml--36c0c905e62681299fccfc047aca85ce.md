## Слой 1 — промт-инжиниринг → тема 5 (поведенческий контракт)
Главная ценность репо. Приёмы, бьющие прямо в наши болевые точки (риск «агент делает вид, что MCP нет» и сквозное правило «отвечай только из источника»).
### Приём А — анти-выдумка (источник: `meta-compile/SKILL.md`)
Формулировка скилла: инструкция и reference-файлы — это *полная* документация, «не ищи примеры XML в выгрузках конфигураций». Прямой аналог нашего «отвечай только из результатов инструментов; структуры, имена атрибутов и форматы — не из головы». **Применение:** адаптировать формулировку почти дословно в системный промпт. Это ровно про тот failure mode, против которого строится агент.
### Приём Б — описания инструментов через «вместо» + «когда» (источник: `description` скиллов form-info, meta-info, form-patterns, cf-info)
В репо `description` скилла говорит не *что делает инструмент*, а *когда его вызвать* и *что он заменяет*: «заменяет чтение тысяч строк XML», «используй вместо чтения XML-файлов напрямую», «используй как подготовительный шаг при написании запросов и кода», «вызывай ПЕРЕД проектированием формы». **Применение:** наши обязывающие описания MCP-инструментов писать в той же логике — с позиции «когда агент должен это вызвать», с явным «вместо чего» и контекстом «что предшествует / что следует». Решает риск №1 (агент не тянется к MCP).
### Приём В — явные воркфлоу в порядке шагов (источник: `docs/meta-guide.md`)
Модели показывается не перечень инструментов, а их **порядок**: «текст → JSON DSL → /meta-compile → XML → /meta-validate». **Применение:** зафиксировать в контракте типовые цепочки наших сценариев (напр. «search_metadata → get_object → ответ»).
### Приём Г — уровни детализации + drill-down + пагинация (источник: `meta-info/SKILL.md`)
Три уровня detail (`overview` default / `brief` / `full`) + `-Name` для drill-down в элемент + `-Limit/-Offset` для пагинации. Управление токенами и фокусом. **Применение:** заложить в дизайн MCP-инструментов явные уровни детализации и drill-down; не отдавать всё сразу. Стыкуется с нашим «сигналом полноты» («есть ещё N разделов» = их `-Offset`).
### Приём Д — allowed-tools на скилл (источник: frontmatter [SKILL.md](http://SKILL.md))
Каждый скилл явно ограничивает набор инструментов (meta-info: только Bash/Read/Glob; web-test: всё). **Применение:** аргумент к тому, чтобы режим консультанта физически НЕ включал пишущие инструменты, а не «просил не использовать».
### Приёмы для генерации (нам как читающему консультанту — слабо применимы)
Паттерны-архетипы форм (`form-patterns`), shorthand-синтаксис вместо JSON, batch через `;;`, русские синонимы ключей DSL — всё это про генерацию/редактирование объектов. Прямой пользы консультанту мало. Единственное пересечение: **русские синонимы рус↔англ** перекликаются с уже принятым в реш. 2.4 «BSL-синонимы из mcp-1c» — отметка как подтверждение приёма, не новость.

## Слой 2 — категоризационные таблицы → справочник типов
Отчёт отделяет: эти артефакты — про **имена и группировку** типов (не про XML-форматы). Переиспользуемы независимо.
- **Таблица 44 типов в 8 группах** (`docs/1c-specs-index.md`): XML-элемент → каталог выгрузки → русское название → ссылка на спеку. Готовая основа справочника типов.
- **42 каталога верхнего уровня** выгрузки (`1c-config-objects-spec.md`, строки 18–64).
- **Матрица модулей по типам** (кто имеет ObjectModule / ManagerModule / RecordSetModule) — готовая карта «где искать BSL-код».
- **Структура каталога объекта** (`<MetaType>/<ObjectName>/ ... Ext/ObjectModule.bsl`, Forms/, Templates/, Commands/) — где физически лежат модули в выгрузке.
- **Русская классификация 23 «рабочих» типов** (4 группы; термин «Ссылочные» вместо «Прикладные») — subset самых ходовых, удобен для объяснения пользователю.
8 групп из `1c-specs-index.md`: Служебные и интерфейсные · Общие объекты · Интеграция и сервисы · Поведение и параметризация · Прикладные объекты · Регистры · Планы · Бизнес-процессы.
