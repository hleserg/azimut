> Источник: отчёт Claude Code по задаче [HLE-456](https://linear.app/hleserg/issue/HLE-456) — сравнение 4 реализаций индекс/граф. Факты из исходников, ссылки `файл:строка` — в заметках задачи. Где из кода не следует — «не определить».
Разбирались 4 репо по осям (модель графа, обход, чанкинг, парсинг, скорость, расширения, лицензия): **bsl-atlas** (наш upstream), **1c-mcp-metacode**, **bsl-graph**, **mcp-1c** (feenlace). Каждый силён в чём-то одном, никто не закрывает всё.
## Сводка по репо
**bsl-atlas (наш upstream, AGPL-3.0, Python).** Единственный из открытых, кто реально режет код на функции через tree-sitter (fallback regex) и делает функциональный чанкинг (≤2000 симв → 1 чанк, иначе сплиттер, функции \<5 строк пропуск). Инкрементальность на уровне функций (function-hash), поддержка `.cfe`. Слабости (известны): callee как строка `callee_name TEXT` без резолва, обход 1 уровень, подписки упомянуты но рёбра не строятся. Деталь: в тестах есть `ReverseCallIndex`, но в исходниках его нет (выпилен/не смержен).
**1c-mcp-metacode (закрытый, Neo4j).** Эталон графа: Routine-узлы, `CALLS*1..N`, подписки до обработчика. Код закрыт — только архитектурный ориентир. Тяжёлый (\~30 мин на БУХ без эмбеддингов, 4GB heap).
**bsl-graph (MIT, Kotlin/NebulaGraph).** Лицензия чистая, метаданные через bsl-mdclasses хорошие, подписки строятся. Но граф вызовов 0% (BSL Parser не интегрирован, `searchMetadata()` закомментирован). Сырой, тяжёлый JVM-стек.
**mcp-1c (feenlace, MIT, Go, Bleve) — НОВЫЙ донор, в прошлых отчётах его не было.** Не граф, полнотекстовый поиск. Но **скорость \~7 сек на 13000+ модулей** против 30 мин у metacode. Техники: GC-off на сборку, FNV-шардирование, batch+`unsafe_batch`, дисковый кеш по SHA-256 пути, манифест-diff, неблокирующий старт. Плюс **BSL-синонимный анализатор** (рус↔англ, \~40 ключевых слов + 180 функций, `СтрНайти ↔ StrFind`) и интеграция с живой 1С (HTTP). Минусы: чанкинг на уровне модуля (грубо), графа и подписок нет, `.cfe` и живой граф зависимостей — только в платной версии.
## Резолв одноимённых — узел разблокирован
Отчёт подтверждает: callee должен быть **узлом, не строкой** — единственный способ различить одноимённые процедуры. Готовой открытой реализации алгоритма нет (metacode закрыт, bsl-graph не дошёл). НО отчёт даёт конкретную SQLite-схему как стартовый каркас:
```sql
CREATE TABLE routines (id, name, module_id, is_export, ...);
CREATE TABLE calls (caller_id REFERENCES routines, callee_id REFERENCES routines);
-- callee_id = NULL для неразрешённых вызовов (внешние/динамические)
```
То есть HLE-456 не «решил» резолв алгоритмически (этого никто в открытую не сделал), но **снял неопределённость по схеме хранения** — блокер финализации реш. 2.2 снят.

## ⚠️ Лицензия — расхождение трактовок (важно)
Отчёт HLE-456 подаёт **AGPL-3.0 у bsl-atlas** как «критическую проблему» (якобы форк невозможен). Факт лицензии (AGPL-3.0) совпадает с ранее зафиксированным в дизайне. Расходится трактовка.
**Статус в дизайне (не пересматриваем):** действует публичная AGPL-3.0, наш форк (Азимут) сам идёт под AGPL-3.0 — это осознанно принято. Трактовка отчёта «AGPL = блокер» — ЭТО НЕ НОВОЕ РЕШЕНИЕ, а уже закрытый вопрос. Пересмотр только если появится цель коммерциализации под закрытый сервис — это отдельный звонок Сергея.
Практическое следствие для заимствований: из bsl-atlas (AGPL) берём **логику/алгоритмы**, не форкаем код; mcp-1c и bsl-graph (MIT) — код можно переиспользовать.

## 📌 Рекомендации к закреплению в дизайне
### Финализация реш. 2.2 (граф вызовов) — теперь можно
1. **Конкретная SQLite-схема** `routines` + `calls(caller_id, callee_id)` с `callee_id=NULL` для неразрешённых вызовов. Даёт различение одноимённых на уровне хранения.
2. **Резолв одноимённых** остаётся алгоритмически нашим (готового нет), но схема под него зафиксирована — узел разблокирован.
### Новое реш. 2.4 «Скорость и пайплайн индексации» (отвечает на вопрос «пайплайн сбора кода»)

> ✏️ **Переформулировано 2026-05-27 (HLE-494 #2).** Ранее реш. 2.4 предлагало «переписать на Go» и называло `mcp-1c (feenlace)` основным донором; это был тихий разворот стека от фундамента Python+FastMCP (см. `solutions-registry-summary.md` противоречие №3). Теперь сформулировано так: **портировать технику в Python, второй язык в рантайме не вводим**. `mcp-1c (feenlace)` остаётся как референс техник, не как зависимость и не как язык.

1. **Техники скорости из mcp-1c (MIT) — портировать в Python:** GC-off-аналог (отключение/настройка сборщика на этапе массовой индексации), шардирование по CPU (`multiprocessing` или `concurrent.futures`), batch + неблокирующий старт, дисковый кеш по хешу пути (SHA-256), манифест-diff. Идеи переносим, код Go не копируем.
2. **BSL-синонимный анализатор (рус↔англ) из mcp-1c** — портировать как Python-модуль; критично для поиска (разработчики мешают термины).
3. **Чанкинг — логика bsl-atlas, на нашем Python-стеке.** Пороги и метаданные чанка — как идея (согласуется с реш. 2.1). НЕ форкать Python в Go — фундамент (реш. 1.1, 1.2) фиксирует Python+FastMCP, второй язык в рантайме противоречит «маме не ставить лишнее».
4. **Инкрементальность:** старт на file-level манифесте (mcp-1c, проще), function-hash (bsl-atlas) как апгрейд позже.
### Реестр доноров
1. **Добавить mcp-1c (feenlace)** — раньше его в реестре не было, теперь основной донор по скорости и синонимам (MIT).
### Подписки на события
1. Подтверждается из двух источников (metacode + bsl-graph): читать `EventSubscriptions/*.xml`, строить `ПодпискаНаСобытие.ИмяОбработчика → Модуль.Процедура`. Усиливает реш. 2.2.
