## Что разбирали
Пять репо про доступ к **живой** базе 1С (runtime), а не к выгрузке XML. Классификация: **группа A** — живые данные 1С (релевантно теме 7), **группа B** — не живые данные (внешний AI-сервис или чистая инфраструктура).
## Группа A — живые данные (3 репо)
### 1. SteelMorgan/1c-log-checker — MIT ✅
Только журналы. Go + ClickHouse. Читает бинарные файлы журналов с ФС (`.lgf`/`.lgp` — журнал регистрации, `.log` — техжурнал), складывает в ClickHouse (`logs.event_log`). 8 MCP-инструментов (logc_get_event_log, logc_get_tech_log и др.). База идентифицируется через `cluster_map.yaml` (GUID кластера/инфобазы). Ничего в 1С ставить не надо, данные всегда актуальные. Доступа к бизнес-данным нет.
### 2. vladimir-kharin/1c_mcp — MIT ✅ — КАНДИДАТ №1 для темы 7
Метаданные + расширяемость. BSL-расширение (.cfe) ставится в живую базу и регистрирует HTTP-сервис `mcp_APIBackend` (JSON-RPC); BSL-код выполняется внутри 1С-сервера. Python-прокси (опц.) решает аутентификацию (OAuth2→Basic) и stdio-транспорт. 2 встроенных инструмента (`list_metadata_objects`, `get_metadata_structure` через глобальный `Метаданные[МетаТип]`). **Расширяемость:** любая обработка в подсистеме `mcp_КонтейнерыИнструментов` автоматически становится MCP-инструментом. Расширение НЕ требует изменения конфигурации. **Почему кандидат №1:** ближе всех к задаче Азимута (доступ к метаданным через BSL в runtime); паттерн «1С HTTP-сервис + BSL внутри 1С» позволяет агенту работать с живой конфигурацией без внешнего разбора XML.
### 3. ROCTUP/1c-mcp-toolkit — ⚠️ ЛИЦЕНЗИИ НЕТ — максимум мощи
Полный доступ. Внешняя обработка (EPF) работает в сессии 1С-клиента — полный доступ к данным и метаданным. Два режима: встроенный HTTP-сервер (нативный C++ компонент, порт 6003) или Python FastAPI + long polling. 13 MCP-инструментов, среди них `execute_query` (`Новый Запрос` + `Запрос.Выполнить()`) и `execute_code` (`Выполнить(Код)`) — доступ к ЛЮБЫМ данным базы. Плюс get_metadata, get_event_log, find_references_to_object, get_bsl_syntax_help, get_screenshot и др. Анонимизация ПДн (токены `[ORG-00001]`, `[PER-00001]`, `[INN-00001]`). **Мощнейший из пяти**, но: `execute_code` = полный доступ (есть чёрный список + режим подтверждения). **⚠️ файл LICENSE отсутствует (по отчёту) → все права у автора, код не копируем без явного ответа автора; изучаем только подходы.**
## Группа B — НЕ живые данные (2 репо, не релевантны теме 7)
- **SteelMorgan/spring-mcp-1c-copilot** (Custom OSS, некоммерческое свободно): Kotlin/Spring прокси к облачному AI-сервису **1С:Напарник** (`code.1c.ai`). Инструменты (ask_1c_ai, explain_1c_syntax, check_1c_code) бьют во внешний AI — данные конкретной базы не затрагиваются.
- **SteelMorgan/1c-ai-sandbox-client-server** (MIT): инфраструктурная песочница (Dev Container + Hyper-V VM + Docker Compose). Собственных MCP-инструментов нет — платформа для запуска других (напр. toolkit).

## Наблюдения для Азимута
1. **Три уровня доступа:** только логи (1c-log-checker) → только метаданные + custom (1c_mcp) → полный доступ (1c-mcp-toolkit).
2. **Три механизма доступа:** ФС (самый низкоуровневый, ничего ставить не надо) → 1С HTTP-сервис (стандарт, но нужно расширение) → BSL через EPF (внутри живого сеанса, максимальная мощь).
3. **Для Азимута** (анализ кода и метаданных) ближе всех — **1c_mcp** (метаданные через BSL в runtime, аналог граф-реп, но в живой базе). Паттерн «1С HTTP-сервис + BSL внутри» — способ взаимодействовать с живой конфигой без внешнего разбора XML.
4. **1c-mcp-toolkit/execute_query** — ближайший аналог «произвольные запросы к данным живой базы» (как metacode через Cypher/Neo4j, но без граф-модели — только табличные запросы).
5. **Связь с реш. 2.1:** живые данные базы — достоверный путь для того, что статически не доказуемо (МенеджерВременныхТаблиц, асинхрон, подписки). 1c_mcp — кандидат на этот путь.
