## Контекст

Без объективной шкалы все остальные решения — на глаз. Эта тема даёт «золото» (эталонный набор), метрики качества, пороги-блокеры в CI, и инструменты наблюдаемости/«канарейку». Здесь же — операционный регламент (кто чинит, когда «пожар»), потому что пороги пожара = те же метрики качества.

## Вопросы темы

### Эталонный набор («золото»)

* **Откуда брать:** (1) костяк — из статей ИТС (статья = готовая пара «ответ + источник-чанк»), 100–150 кейсов руками; (2) полуавтомат — RAGAS `TestsetGenerator` по корпусу + ручная отбраковка (оставить 40–60%); (3) самое ценное — реальные вопросы Сергея и мамы, копить с первого дня; (4) формулировки — с форумов (Инфостарт, mista). Цель 150–300 за 3–4 недели.
* Занести эталонный многослойный кейс из темы 2 («почему проводка сменила счёт») как образец.
* **Где хранить золото, кто владелец, частота прогона** — решить.

### Negative-кейсы (агент честно «не нашёл») — половина смысла проекта

* Привязывать каждый к версии корпуса (`corpus_version`) — кейс это утверждение о СЛЕПКЕ индекса на дату, не о реальности.
* Не выдумывать, а собирать из реальных промахов (телеметрия «не нашёл»/фолбэков) + подтверждать руками: реально нет в корпусе → negative; ответ был, агент тупанул → проваленный positive, чинить.
* Эталон формулировать как «не соврал», а НЕ «промолчал» (обновилась дока, ответ появился, нашёл = успех).
* Делить на «вечнозелёные» (вне домена — стабильны годами) и «текучие» (в домене, не покрыто — ревизия при обновлении корпуса).
* Привязать ревизию negative-набора к регламенту ежемесячного обновления доков.

### RAGAS-harness и метрики

* `ragas` (Apache-2.0) + `adapt_prompts(language="russian")` + `TestsetGenerator`; Claude как модель-судья — закрывает Р3 (LLM-судья).
* 4 метрики: Faithfulness, Answer Relevancy, Context Precision, Context Recall.
* **Пороги-блокеры в CI** (GitHub Actions): PR/деплой падает при **Faithfulness < 0,85 ИЛИ Context Precision < 0,80**; триггер расследования при недельном падении Faithfulness > 5%. Инструментует канарейку и даёт судье объективную шкалу.
* **Faithfulness ≠ correctness:** система может набрать 0,95 faithfulness и выдать неверный ответ на устаревшем/неверном контенте. Нужен отдельный контроль слоя контекста (свежесть, согласованность определений) — стыкуется с регламентом обновления и метрикой противоречивости Р1.
* Пороги, меняющие план: Faithfulness < 0,80 с реранком → заземлённый промпт + Self-RAG; Context Recall < 0,75 → проблема в чанкинге, не в ретривере.

### Наблюдаемость и «канарейка»

* **Langfuse self-hosted** (MIT) — трейсы/стоимость/оценка, тегирование тенанта. Дублировать в **Sentry** (уже в стеке) через OTel → пороговые алерты. Решить состав стека наблюдаемости.
* Канарейка (была задача 0c): метрики латентность p95, здоровье индекса, RAM, рост фолбэков/«не нашёл»/переспросов — собираются только в потоке работы и на healthcheck, без фонового демона (для локального режима; для VDS — см. тему 7). Алерты Сергею в Sentry (админ вне MCP-контура — обходит «нет push пользователю»).

### Операционный регламент (из открытых вопросов «Эксплуатация»)

* Телеметрия: автоматизировать без ручного участия Сергея.
* Авто-дёрг кодеров на пожар (Sentry → Linear)? Кто «дежурный»?
* Регламент несрочных ошибок: периодичность, кто, как накапливаем.
* Пороги пожара (конкретные числа) — увязать с метриками выше.
* Калибровка: периодичность, что смотрим, кто.
* Связь Sentry ↔ Linear: дубль или ссылки?
