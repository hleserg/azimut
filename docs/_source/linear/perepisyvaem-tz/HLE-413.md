## Вопросы темы

* **Подтвердить выбор основы.** Ресёрч («Открытые вопросы» → «Опенсорсные компоненты для форка bsl-atlas») формулирует вывод как почти решённый: **форкаем** `bsl-atlas` (Python + MCP + SQLite + ChromaDB + граф вызовов BSL; разрешение автора получено, некоммерческое) + \~10 готовых библиотек под пробелы. Ни RAGFlow, ни kotaemon, ни Onyx не понимают BSL — большие платформы годятся только как UI-фронт поверх нашего MCP. Нужно осознанное подтверждение Сергея и запись «почему именно так».
* **Код-ревью** `FSerg/mcp-1c-v1` (TypeScript, \~152★, Qdrant + RRF, Docker Compose) — ближайший существующий аналог нашего проекта. ВАЖНО: из коробки индексирует только МЕТАДАННЫЕ структуры конфигурации, не тексты модулей BSL/расширения/обработки; по умолчанию all-MiniLM (не BGE-M3), реранкера нет. Решить: берём как скелет, как референс архитектуры, или мимо.
* **Роль** `bsl-atlas` **как ядра.** Он ровно про код (текст BSL + call graph), `docker compose up`, \~500 МБ. «Маме» сам по себе ❌ (нет дружелюбного UI). Подтвердить, что форкаем именно его, и очертить, что в нём переделываем.
* **Стек форка vs наш целевой стек.** bsl-atlas использует ChromaDB + SQLite; мы хотим Qdrant (embedded) + BGE-M3 + Cohere rerank. Решить: мигрировать хранилище/эмбеддер сразу или этапом; что это значит для объёма работ.
* **MCP-шлюз** `AlekseiSeleznev/onec-mcp-universal` (\~400 строк Python, Starlette) — готовый агрегатор нескольких 1С-MCP-серверов. Кандидат на роль шлюза перед bsl-atlas; может сократить объём «своего» MCP-сервера.
* `alkoleft/mcp-bsl-platform-context` (MIT, \~130★) — справочник синтаксиса платформы (shcntx_ru.hbk) через MCP. Можно запустить рядом в Claude Desktop бесплатно, без кода — агент сразу получает API платформы. Включаем в фундамент?
* `mini-ai-1c` (десктоп Tauri, installer, релиз 1.2.7) — лучший готовый вариант «для мамы» (захват кода Конфигуратора, пред-настроенные MCP, провайдеры). Решить: это отдельный клиент для мамы или мы делаем свой UX.
* **Что остаётся НАШИМ кодом (конкурентное ядро).** Поведенческий контракт; серверный контроль добора (Р5); сборка полных разделов + «сигнал полноты»; иерархия источников при конфликте (Р6); eval/LLM-судья. Зафиксировать границу «форк даёт X / мы пишем Y».
* **Adapter-агностичность** (была задача 0i в v1.1 — подключение к любому LLM через адаптер): нужна ли в v2.0 или форк уже задаёт ответ.

## Лицензии (зафиксировать как чек-лист фундамента)

* `bsl-atlas` — разрешение автора (Arman Kudaibergenov) получено, **некоммерческое**; триггер повторного согласования — перед любой монетизацией (мульти-аренда на VDS = коммерция). TODO: сохранить переписку вне Telegram; поправить README форка + файл NOTICE/CREDITS.

* `alkoleft/tree-sitter-bsl` — лицензию проверить напрямую в репо перед использованием.
* Транзитивные зависимости (tree-sitter-bsl, 1c-syntax = LGPL-3.0) — перед коммерческим запуском разложить pyproject форка по лицензиям.
