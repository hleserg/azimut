## Контекст: почему это Приоритет №1

По словам Сергея — возможно важнейший вопрос проекта. Цепочки проведения в 1С не должны разваливаться при чанкинге, иначе агент видит обрывки кода и галлюцинирует. Перекликается с Р6 (код базы — приоритет №1 при конфликте источников).

Эталонный многослойный кейс для проверки (занести в eval, тема 6): **«Почему в апреле документ делал проводку по одному счёту, а в мае — по другому?»** Причина может лежать в коде, в данных/настройках, в расширении (`&Вместо`/`&После`) или в ветке `Если` — и часто проводка пишется НЕ в модуле документа, а в подписке на событие. Цель агента — не вердикт, а карта «где искать», включая честное «это не в коде, это в настройке» (отдельный навык поведенческого контракта, тема 5).

## Вопросы темы

* **Чанкинг модулей.** Как резать длинную `ОбработкаПроведения`: один большой чанк плохо ранжируется, мелкие теряют контекст. Базовый ответ из ресёрча — **AST-чанкинг (cAST)**: резать по структуре (границы Процедура/Функция/Область, процедура целиком = чанк), не по строкам. Даёт +4,3 Recall@5 / +2,67 Pass@1. Подтвердить и определить параметры (целевой размер чанка \~1000–1500 токенов, fallback-сплиттер).
* **Граф вызовов** (узлы = процедуры, рёбра = кто кого вызывает) — детерминированный, парсером, НЕ через LLM (ресёрч arXiv:2601.08773 подтверждает: LLM-граф сам источник галлюцинаций). Как связать чанк кода с его местом в графе, чтобы агент шёл по цепочке, а не видел обрывки.
* **Подписки на событие — ОБЯЗАТЕЛЬНЫ в графе.** Движение часто пишется в подписке другого объекта, а не в проведении документа. Без этого агент ищет проводку в модуле документа и не находит.
* **Внешние обработки (.epf/.erf) — слепая зона.** В XML-выгрузке конфигурации их НЕТ (бинарники в данных базы, справочник `ДополнительныеОтчётыИОбработки`). Как собирать и как обозначать пробел пользователю.
* **Сбор кода (пайплайн).** `DESIGNER /DumpConfigToFiles -update` (конфа, инкрементально) + цикл по расширениям (НЕ `-AllExtensions` — баг на 8.3.18) + обработка на 1С/OneScript для выгрузки .epf/.erf из справочника + распаковка (onepack/v8unpack). Подтвердить схему.
* **Чем парсим BSL и строим граф.** Кандидаты: `alkoleft/tree-sitter-bsl` (py-tree-sitter, быстрый — символы 1–23 мс на 20+ ГБ; лицензию проверить) как основной; JSON-репорты `BSL Language Server` (`analyze -r json`) как subprocess; `mdclasses` для графа метаданных. Граф вызовов — каннибализировать из bsl-atlas / `alkoleft/bsl-graph` (без NebulaGraph, через bsl-mdclasses-обёртку).
* **Асинхронность — слепая зона всех инструментов** (`НачатьВыполнение`, `ОписаниеОповещения`, `ПодключитьОбработчикОжидания`, фоновые задания). Принять эвристику (имя обработчика-строки → ребро на метод) или отложить.
* **Код-специализированные эмбеддеры** (SFR-Embedding-Code, voyage-code-3, jina-code) лидируют на CoIR, НО не понимают русские идентификаторы (`Цены`, `Номенклатура`, `ПолучитьЗначение()`). Эмпирически мультиязычный BGE-M3 для кода 1С лучше. Вопрос: нужен ли отдельный код-индекс другим эмбеддером или BGE-M3 закрывает оба (стыкуется с темой 3).
* **grep+AST как параллельные MCP-tools** (CodeRAG-Bench: для кода чистый вектор проигрывает grep+AST). Делать ли с самого начала.
* **Живые данные базы для точного ответа про конкретные документы** — `vladimir-kharin/1c_mcp` (MCP внутри 1С) + история изменений (gitsync, апрель→май). Решить: входят в ядро или остаются на потом.
