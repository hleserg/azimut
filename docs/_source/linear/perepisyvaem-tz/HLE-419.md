## Контекст

Расширение за пределы локального деплоя: размещение на VDS с обслуживанием нескольких контор (мульти-аренда). **Локальный деплой (для Сергея и мамы) НЕ убираем — должны жить оба варианта.** Онлайн/мульти-аренда — дополнительный режим, не замена. ВАЖНО: это коммерческий триггер по лицензии bsl-atlas (см. ниже).

## Вопросы темы

### Деплой и общая кодовая база

* VDS-вариант рядом с локальным: что общего в кодовой базе, а что — развилка по конфигу (режим `local` / `online`).

### Мульти-аренда (несколько фирм на одном VDS)

* У каждой фирмы — отдельная конфигурация 1С и личная папка документов; **изоляция данных обязательна** (одна не видит индекс/данные другой).
* Qdrant: **одна коллекция + поле** `group_id` **с индексом** `is_tenant=True` (Qdrant 1.16 Tiered Multitenancy, ACORN-фильтрация) — НЕ отдельная коллекция на тенанта (лимит \~1000). Подтвердить.
* Файловое хранилище `/data/{tenant_id}/...`; FastAPI-зависимость инжектит `tenant_id` из JWT в фильтры Qdrant и таблицы.
* Где границы изоляции: индекс, папки, ключи.

### Веб-морда

* Заливка и просмотр файлов конфигурации и документов; кто заливает (сотрудник/админ), что видит, как привязано к фирме.
* **Кандидаты:** старт с **kotaemon** (Apache-2.0, Gradio, лёгкий); **RAGFlow** (Apache-2.0, мульти-аренда из коробки, тяжёлый — MySQL+Redis+MinIO+ES) как UI поверх нашего MCP; Open WebUI — перечитать LICENSE (ужесточение для коммерции >50 польз.).
* Веб-морда может стать каналом push к пользователю, которого нет у MCP (напр., «переиндексация готова»).

### Авторизация

* На MCP и на веб-морде: как фирма идентифицируется при запросе к MCP, как переиспользуются/различаются креды между MCP и вебом, где хранятся секреты.
* **Кандидаты:** FastMCP OAuth 2.1 встроен (Authorization Code + PKCE, DCR); **Authentik** (MIT, Docker-native) как IdP (легче Keycloak); `fastapi-keycloak-middleware` через OIDC; **JWT с claim** `tenant_id` → один реалм для MCP и веб-морды.

### Мониторинг по тенантам

* Метрики канарейки и алерты в Sentry размечать по `tenant_id`, иначе деградация одной конторы утонет в усреднении. Langfuse тегирует тенанта через `metadata.tenant_id`.

### Влияние на ранее принятые решения (пересмотреть в ADR)

* **Qdrant embedded vs server** становится развилкой по режиму: embedded — для локального (маме не ставить вторую сущность); server с разделением по тенантам — для VDS.
* **Канарейка «без фонового демона»** под вопросом на VDS: «протухшее в покое» у неактивной конторы некому разбудить, пока вторая ждёт корректных метрик → фоновый сторож может быть оправдан. Решение «без демона» оставить для локального режима.
* **Push пользователю:** ограничение «MCP не инициирует сообщение» острее на мульти-аренде; веб-морда — кандидат на канал push.

### Лицензия / коммерция

* Мульти-аренда на VDS / платные подписки / продажа доступа = **коммерция → СНАЧАЛА согласовать с автором bsl-atlas** (Arman Kudaibergenov). Триггер из темы 1. Также: разложить транзитивные зависимости по лицензиям; перечитать LICENSE Open WebUI; Arize Phoenix (ELv2) нельзя перепродавать как hosted.
