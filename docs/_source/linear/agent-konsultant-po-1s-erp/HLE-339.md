Два уровня кеширования поисковых запросов для снижения латентности и стоимости повторных вопросов.

## Уровень 1: Exact-match кеш

Точное совпадение текста запроса → возвращаем кешированный результат поиска без обращения к Qdrant и реранкеру.

```python
cache_key = hashlib.sha256(query.strip().lower().encode()).hexdigest()
```

TTL: `semantic_cache_ttl` секунд (дефолт 3600). Хранилище: в памяти (dict) с LRU-вытеснением, лимит `semantic_cache_max_size` записей.

## Уровень 2: Семантический кеш

Если exact-match промах — проверяем семантическую близость к ранее отвеченным запросам. Если косинусная близость эмбеддинга нового запроса к кешированному ≥ `semantic_cache_threshold` (дефолт 0.95) — возвращаем тот же результат.

Реализация: держим небольшой список `(embedding, cache_key)` последних N запросов. При промахе exact-match — вычисляем близость к ним.

**Важно:** семантический кеш не должен быть тяжелее самого поиска. Если список кешированных запросов большой — лучше отключить.

## Конфиг

```yaml
cache:
  enabled: true
  exact_match: true
  semantic: true
  threshold: 0.95       # порог семантического совпадения
  ttl_seconds: 3600     # время жизни записи
  max_size: 200         # макс. записей в кеше
  semantic_max_size: 50 # макс. запросов для семантического сравнения
```

При `cache.enabled: false` — кеш полностью отключён, нет никаких побочных эффектов.

## Телеметрия кеша

Логируем: `cache_hit_exact`, `cache_hit_semantic`, `cache_miss`. Это нужно для оценки эффективности в задаче 13.

## Инвалидация

При запуске переиндексации — кеш полностью сбрасывается (`cache.clear()`). Иначе может вернуть устаревшие результаты.
