Обкатка нескольких стратегий поиска на реальных запросах с накоплением телеметрии — чтобы выбор стека делать по данным.

## Два режима (рубильник в конфиге)

* `experiment` — на каждый запрос гоняются несколько стратегий (dense / hybrid / hybrid_rerank / \[мультимодал картинок\]). Пользователю — лучшая по оценке, остальные логируются в фоне. Медленнее, больше векторов — осознанный компромисс.
* `production` — одна выбранная стратегия, быстро. Переключение одной настройкой.

## Телеметрия (на запрос в experiment)

* Запрос, какие стратегии запускались, что каждая вернула (id разделов, скоры, тайминги, стоимость Cohere).
* **Три источника оценки:**
  1. **Явный фидбэк** — лёгкий «в точку / мимо / частично» (золотой стандарт).
  2. **Неявные сигналы** — переспрос, число итераций поиска, использовал ли найденное.
  3. **LLM-судья** — офлайн (задача 12).
* Лог в локальное хранилище (вне гита), структурированный, под офлайн-анализ (задача 13).

## Definition of done

* Рубильник experiment/production работает; production не тратит ресурсы на лишние стратегии.
* В experiment на запрос логируются все стратегии с таймингами и стоимостью.
* Явный фидбэк захватывается (механизм лёгкий, не раздражающий).
* Неявные сигналы фиксируются автоматически.
* Лог готов для анализа (задача 13).
