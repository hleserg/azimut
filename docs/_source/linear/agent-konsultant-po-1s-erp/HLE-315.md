## Логирование

* Структурные логи (JSON) с уровнями; контекст: session_id, config_name, стратегия поиска, источник.
* EventScrubber: вычищать ключи Cohere/OpenAI, пути пользователя, содержимое доки из payload (PII/секреты) — как в atman.

## Трейсинг (распределённые спаны)

* Спаны на полный путь запроса: получен запрос → embed (BGE/Cohere) → поиск (dense/sparse) → rerank → сбор разделов → ответ.
* `gen_ai.*` спаны для всех LLM/embedding/rerank вызовов (provider, model, токены, стоимость) — паттерн atman.
* Спаны индексации: парсинг XML, OCR, эмбеддинг, запись в индекс — с таймингами по этапам.
* Спаны MCP-инструментов: каждый tool call (search_documentation, get_section, healthcheck, ...).

## Метрики (максимум полезного)

* Поиск: latency по стратегиям, hit rate, глубина ресёрча (сколько разделов просмотрено), частота срабатывания «сигнала полноты».
* Модель BGE: время загрузки/выгрузки, частота прогревов, попадания в «тёплую».
* Стоимость Cohere: $ на запрос/в сумме, по типам (rerank/embed/command).
* Индексация: длительность, чанков/сек, объём индекса, частота переиндексаций.
* Поведение агента: доля ответов с предварительным поиском vs из памяти (контроль «не фантазирует»), частота уточняющих вопросов, доля доведённых до удовлетворённости.
* Онбординг: успешность validate_path, частота починки процедур (задача 17).
* Телеметрия стратегий (задача 11) — туда же.

## Канарейка по Qdrant и качеству (стучать админу ДО раздражения пользователя)

**Цель: система сама сигналит Сергею о деградации до того, как она начнёт бесить пользователя.** Метрики собираются В ПОТОКЕ реальной работы (во время запросов и на healthcheck) — фонового демона-сторожа НЕТ (в покое показатели не меняются, тикать вхолостую незачем; «протухшее в покое» — истёкший ключ Cohere, недоступный индекс — ловится проверкой на ближайшем healthcheck).

* **Производительность Qdrant:** латентность запроса p50/p95/p99. Канарейка: p95 пробил порог (например, 2× от базовой) → алерт.
* **Здоровье индекса:** размер коллекции (число точек), объём на диске; аномалия после переиндексации (полупустой индекс = парсинг отвалился молча).
* **Память:** RAM под Qdrant + BGE суммарно; приближение к лимиту ОЗУ → упреждающий алерт (риск свопа/OOM на слабой машине, напр. у мамы).
* **Качество поиска:** доля срабатываний фолбэка (Cohere недоступен → локальный reranker), доля «не нашёл», частота переспросов — просадка тренда сигналит, пока пользователь ещё терпит.
* **Сбои:** ошибки Qdrant (битая коллекция, не читается), таймауты → issue в Sentry сразу.
* **Алерты:** настроить в Sentry пороговые metric-alerts на имя Сергея (письмо/уведомление). Это обходит ограничение MCP «нет push пользователю» — админу Sentry написать может, он вне MCP-контура.

## Дополнение: opt-in телеметрии (GDPR-compliant)

По итогам обсуждения — сбор телеметрии требует явного согласия пользователя (GDPR, Art. 6(1)(a)). Дефолт — **выключено**. Включается только активным действием.

### Что добавить в задачу 0c

**Флаги конфига:**
```yaml
sentry_enabled: false           # false до явного согласия
sentry_consent: null            # null | "granted" | "denied"
sentry_consent_date: null       # ISO дата согласия
```

**Sentry инициализируется ТОЛЬКО если `sentry_enabled: true` И `sentry_consent: "granted"`.**

При любом другом состоянии — Sentry SDK не подключается вообще, никаких «тихих» вызовов.

### Текст вопроса при онбординге (задача 10)

Агент спрашивает один раз, при первом запуске:

> «Если что-то пойдёт не так, могу автоматически отправить отчёт об ошибке разработчику — без ваших документов и личных данных, только технические детали. Это поможет быстрее находить и чинить баги. Разрешить?»

- Ответ «да» → `sentry_enabled: true`, `sentry_consent: "granted"`, инициализировать Sentry
- Ответ «нет» → `sentry_consent: "denied"`, Sentry молчит, **больше не спрашиваем**
- Закрыл без ответа / проигнорировал → считается «нет», `sentry_consent: "denied"`

### Отключение по запросу пользователя

Пользователь в любой момент: «отключи отчёты об ошибках» / «убери телеметрию» / «не отправляй ничего никуда» → агент вызывает `update_settings('sentry_enabled', False)`, подтверждает: «Готово, отчёты об ошибках отключены».

Отзыв должен быть таким же простым, как дача согласия — это требование GDPR.

### DSN

DSN хранится НЕ в коде и НЕ в гите. Поставляется скриптом установки (задача 15) — скачивается с нашего endpoint'а при первом запуске, кладётся в `~/.1c-consultant/.env`. Пользователь его не видит и не пишет вручную.

## Дополнение к задаче: механизм DSN и условная отправка

### DSN без хардкода в гите

DSN Sentry **не хранится в репозитории**. Механизм: при установке (скрипт `install.sh` / `setup.py`) DSN скачивается с нашего endpoint'а (`https://api.1c-consultant.ru/config` или аналог) и кладётся в `~/.1c-consultant/.env`. Пользователь не видит DSN и не должен его знать. В репозитории — только `.env.example` с `SENTRY_DSN=your_dsn_here`.

### Условная отправка по флагу

Sentry инициализируется **только если** `sentry_enabled: true` в конфиге. Дефолт — `false` до явного согласия пользователя (задача 10 — онбординг). Если `false` — Sentry SDK не инициализируется вообще, никаких фоновых подключений.

```python
import sentry_sdk
from config import settings

if settings.sentry_enabled and settings.sentry_dsn:
    sentry_sdk.init(
        dsn=settings.sentry_dsn,
        traces_sample_rate=0.1,
        # НЕ включаем send_default_pii — пути к файлам могут содержать имя пользователя
    )
```

### Что НЕ логируем в Sentry (конфиденциальность)

- Содержимое документов и чанков
- Пути к файлам пользователя (могут содержать имя)
- Тексты запросов пользователя
- IP-адреса (Sentry собирает автоматически — отключаем через `send_default_pii=False`)

Логируем только: тип ошибки, стек трейс (без данных пользователя), метрики (латентность, RAM, статусы индекса).

### Канарейка при sentry_enabled=false

Если пользователь не согласился на телеметрию — Sentry-алерты на Сергея не работают. Это ок: мониторинг в этом случае только локальный (логи). Документируем в README.
