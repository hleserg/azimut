Индексатор текстовой документации: офлайн-ИТС (дока по конфе) и портал техдокументации платформы. Источники — пути в конфиге, данные вне гита.

## Что сделать

* Парсер HTML/htm офлайн-ИТС: заголовки, текст, **сохранять иерархию разделов** (критично для глубокого ресёрча — нужна вложенность).
* Парсер доков платформы (формат уточнить по факту).
* Чанкинг с метаданными: путь раздела, заголовок, родитель, ссылки на соседние/дочерние. **Не рвать чанк посреди смысловой единицы.**
* OCR картинок: русский текст со скриншотов (формы, поля, ошибки), рядом с чанком. **Сохранять путь к файлу картинки** — чтобы агент мог показать форму пользователю.
* Эмбеддинг чанков через **FlagEmbedding** (BGE-M3, dense+sparse), запись в векторное хранилище с метаданными. **Хранить оба представления** (для гибрида и shadow-эксперимента).
* **Инкрементальность через Document Registry (задача 1a)** — НЕ изобретать собственный механизм хэширования. Каждый файл регистрируется в общем реестре; решение «индексировать / пропустить / переиндексировать» принимает реестр (`should_reindex()` по SHA-256 и статусу). При сбое чтения файла — `mark_error()`, при успехе — `mark_ok(chunks_count)`. Удалённые файлы и их чанки чистятся через механизм реестра.

## Definition of done

* Индексация офлайн-ИТС проходит, чанки с сохранённой иерархией разделов.
* OCR извлекает русский текст с тестового скриншота; путь к картинке в метаданных.
* Повторный запуск переиндексирует только изменённые файлы — через `DocumentRegistry.should_reindex()`, не через локальный самопис.
* Файлы со статусом `error` в реестре переиндексируются при следующем прогоне.
* Гибридный поиск по проиндексированной доке даёт осмысленные результаты на тестовом запросе.
