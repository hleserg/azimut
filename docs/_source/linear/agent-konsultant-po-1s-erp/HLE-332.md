Единая SQLite-база, в которой каждый парсер регистрирует файлы при индексации. Реестр — основа инкрементальной индексации, обнаружения сбоев и автоочистки удалённых файлов.

## Схема таблицы `document_registry`

```sql
CREATE TABLE document_registry (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    source_type     TEXT NOT NULL,        -- 'config_xml' | 'its_docs' | 'user_docs'
    file_path       TEXT NOT NULL UNIQUE,
    file_hash       TEXT,                 -- SHA-256 содержимого
    file_size       INTEGER,
    last_indexed_at TEXT,                 -- ISO-8601
    last_success    INTEGER NOT NULL DEFAULT 0,  -- 0/1
    error_message   TEXT,
    deleted         INTEGER NOT NULL DEFAULT 0,  -- 0/1, файл удалён с диска
    format_advisory INTEGER NOT NULL DEFAULT 0   -- 0/1, формат требует конвертации (xlsx и др.)
);
```

## Что нужно реализовать

### 1\. `RegistryManager` — Python-класс

* `upsert(file_path, source_type, hash, size)` — создать или обновить запись до индексации
* `mark_success(file_path)` — пометить успешную индексацию
* `mark_failure(file_path, error)` — пометить сбой с сообщением об ошибке
* `mark_advisory(file_path, format)` — файл обнаружен, но парсинг не поддерживается (xlsx и др.)
* `mark_deleted(file_path)` — файл удалён с диска
* `get_failed_files(source_type?)` — список файлов с last_success=0 и error_message
* `get_advisory_files(source_type?)` — список файлов с format_advisory=1
* `needs_reindex(file_path, current_hash)` → bool — True если хеш изменился или last_success=False
* `cleanup_deleted()` — удалить из реестра записи помеченные deleted=1 (вызывать после Qdrant cleanup)

### 2\. Логика инкрементальной индексации

При каждом запуске парсера:

1. Вычислить SHA-256 файла
2. Вызвать `needs_reindex(path, hash)`:
   * Если `last_success=True` И хеш совпадает → **пропустить**
   * Если хеш изменился → **переиндексировать**
   * Если `last_success=False` (был сбой) → **переиндексировать всегда**, даже если хеш тот же
3. Зафиксировать результат через `mark_success` или `mark_failure`

### 3\. Обнаружение удалённых файлов

При healthcheck и перед каждой индексацией:

* Пройти по всем записям реестра с `deleted=0`
* Если файл не существует на диске → вызвать `mark_deleted(path)`
* Передать список удалённых в Qdrant-слой для удаления чанков по `source_path` payload
* После подтверждения удаления из Qdrant → `cleanup_deleted()`

### 4\. MCP-инструмент `get_registry_status`

Возвращает сводку для агента:

```json
{
  "total": 1247,
  "indexed_ok": 1231,
  "failed": 8,
  "advisory": 12,
  "deleted_pending": 3,
  "failed_files": [
    {"path": "docs/Регламент.pdf", "error": "OCR timeout", "source": "user_docs"},
    ...
  ],
  "advisory_files": [
    {"path": "docs/Бюджет.xlsx", "format": "xlsx", "source": "user_docs"},
    ...
  ]
}
```

Агент вызывает этот инструмент после каждой индексации и при healthcheck, чтобы сообщить пользователю о проблемах.
