## Что сделать (по образцу atman)

* Скопировать подход atman: `pyproject.toml` (+ `uv`/`uv.lock`), `Makefile`, `.gitignore`, `.gitattributes`, `.markdownlint.json`, `.pre-commit-config.yaml`, `README.md`/`README-ru.md`, `CONTRIBUTING.md`, `SECURITY.md`, `CODE_OF_CONDUCT.md`.
* Структура каталогов: `src/`, `tests/`, `e2e/`, `fixtures/`, `docs/` (с `docs/development/`, `docs/adr/`, `docs/features/`).

### Правила для агентов

* `AGENTS.md` и `docs/development/DEVELOPMENT_STANDARD.md` — по образцу atman, с разделами: структура проекта, стандарт кода, тесты, документация, термины (глоссарий 1С), git-workflow (см. ниже), правила инструментирования (Sentry — позже, задача 0c).

### Тулинг качества (заложить в pre-commit + CI)

* **ruff** (линт+формат), **mypy** (типы), **pytest** + **pytest-cov** (порог покрытия) + **pytest-asyncio**, **pydantic** (модели/конфиг), **markdownlint**.
* Безопасность: **pip-audit**/safety (дыры в зависимостях), **detect-secrets**/gitleaks (не закоммитить ключи Cohere/OpenAI — критично!).
* **conventional commits** проверка.
* Шаблоны PR/issue, dependabot/renovate.

### CI (GitHub Actions, по образцу .github/ из atman)

* Workflow на pull_request: ruff, mypy, pytest+cov, markdownlint, pip-audit, detect-secrets.
* Релизы/прочее — как в atman.

### Интеграции (ручное, UI)

* **Devin как ревьювер** — подключить к репо (как в atman; посмотреть там настройку — метки Devin Playbooks !triage/!plan/!implement/!review уже есть в workspace). Devin должен авто-ревьювить PR.
* **Sentry** — НЕ подключать здесь. Сначала задача 0b (согласование подписки), потом 0c (инструментация). Тут только заложить место под DSN в конфиге, выключенным по умолчанию.

### Git-workflow для агентов (зафиксировать в AGENTS.md как ЖЁСТКОЕ правило)

1. Если на `main` — сделать ветку ПЕРЕД работой (checkout). Никогда не коммитить в main напрямую.
2. В одной ветке может быть несколько задач — это допустимо.
3. Закончил работу → PR (можно draft).
4. После пуша — снять draft, запустить все проверки.
5. **Дождаться ОКОНЧАНИЯ всех проверок и работы ВСЕХ ревьюверов** (включая Devin и любые «жёлтые»).
6. Исправить: ВСЕ комментарии ревью по этому пушу, ВСЕ ошибки CI, ВСЕ конфликты мерджа. Закрыть (resolve) починенные замечания ревьюверов.
7. Запушить исправления → **повторять цикл 5-6**, пока: все проверки пройдены и завершены, ноль незакрытых замечаний ревью по пушу, ноль ошибок CI, ноль конфликтов.
8. ТОЛЬКО при полном зелёном — мерджить.
9. Если добиться зелёного не выходит — НЕ мерджить, написать в чат «не могу» с причиной и остановиться.

* Механика ожидания: исполнение через **GitHub Action** (агент просыпается по событиям PR/вебхукам), не поллинг в терминале.
* **branch protection**: настроить так, чтобы merge был невозможен без зелёных required-проверок (это страхует правило технически, а не только текстом).
