Реализовать адаптивный выбор количества чанков, передаваемых агенту, вместо фиксированного `top_k`. Цель: сохранить качество ответов при минимально достаточном контексте.

## Проблема

Исследования показывают U-образную зависимость: с ростом числа переданных чанков качество сначала растёт, потом падает — модель «отвлекается» на нерелевантный контекст. Зафиксированный «контекстный обрыв» начинается примерно при 2500 токенах. Хардкодить конкретное top_k нельзя — оптимум зависит от вопроса.

## Алгоритм: адаптивный cut-off по скорам реранкера

После того как реранкер проставил скоры (Cohere Rerank v4 или fallback BGE-reranker):

```python
def adaptive_cutoff(ranked_chunks: list[ScoredChunk], 
                    min_k: int = 2, 
                    max_k: int = 10,
                    gap_threshold: float = 0.15) -> list[ScoredChunk]:
    """
    Находит точку, где скор резко падает — переход от
    'тесной группы релевантных' к 'менее релевантным'.
    """
    if len(ranked_chunks) <= min_k:
        return ranked_chunks
  
    selected = [ranked_chunks[0]]
    for i in range(1, min(len(ranked_chunks), max_k)):
        gap = ranked_chunks[i-1].score - ranked_chunks[i].score
        if gap > gap_threshold:
            break  # нашли обрыв — стоп
        selected.append(ranked_chunks[i])
  
    return selected
```

Параметры `gap_threshold`, `min_k`, `max_k` — в конфиге, не хардкод.
