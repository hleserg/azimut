Исследование и реализация адаптивного порога отсечения RAG-выдачи вместо фиксированного `top_k`.

## Контекст и мотивация

Текущая практика — фиксированный `top_k` (например, 5 чанков). Проблема: простые вопросы получают лишний контекст (деньги + шум), сложные — могут недополучить. Исследования (Coinbase CAR, 2024) показывают, что адаптивный порог даёт -60% токенов при сохранении качества.

**Принцип:** реранкер возвращает скоры. Между «кластером» релевантных чанков и нерелевантными есть gap в скорах — вот там и отрезаем.

## Что нужно сделать

### 1\. Анализ данных из задачи 11

Используем накопленную телеметрию: для каждого запроса смотрим распределение скоров реранкера по топ-20 чанкам. Ищем паттерн gap.

### 2\. Реализация CAR-подобного алгоритма

```python
def adaptive_cutoff(scored_chunks: List[Tuple[float, Chunk]], 
                    min_k: int = 1, 
                    max_k: int = 10) -> List[Chunk]:
    """
    Находит точку где скоры резко падают (gap) и отрезает там.
    Гарантирует min_k <= результат <= max_k.
    """
    scores = [s for s, _ in scored_chunks]
    # Находим наибольший gap между соседними скорами
    gaps = [(scores[i] - scores[i+1], i+1) for i in range(len(scores)-1)]
    if not gaps:
        return [c for _, c in scored_chunks[:min_k]]
  
    cutoff_idx = max(gaps, key=lambda x: x[0])[1]
    cutoff_idx = max(min_k, min(max_k, cutoff_idx))
    return [c for _, c in scored_chunks[:cutoff_idx]]
```

### 3\. Параметры через конфиг

```yaml
retrieval:
  adaptive_cutoff: true       # включить/выключить
  adaptive_min_k: 1           # минимум чанков
  adaptive_max_k: 10          # максимум чанков
  adaptive_gap_threshold: 0.1 # минимальный gap для отсечения
  fallback_k: 5               # если адаптивный отключён
```

### 4\. Телеметрия самого алгоритма

Логируем для каждого запроса: `chosen_k`, `gap_size`, `top_score`, `bottom_score`. Это данные для задачи 13 (финальный выбор стратегии).

### 5\. Контекстный обрыв \~2500 токенов

Исследования выявили деградацию качества после \~2500 токенов контекста. Добавляем hard cap: суммарный размер чанков в контексте ≤ 2500 токенов (настраивается). Если hit — логируем как `context_cap_hit`.
