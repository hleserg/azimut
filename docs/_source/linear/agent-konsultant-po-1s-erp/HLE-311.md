Механизм «живых процедур»: у агента есть записанные пошаговые инструкции; если реальность разошлась (1С перепрятали ссылку, изменили поля, UI поменялся) — агент с пользователем находит новый путь, переписывает процедуру и шлёт админу (Сергею) обновление с телеметрией.

## Зачем

Захардкоженные инструкции по получению доки ИТС / выгрузки конфы протухают при изменениях на стороне 1С. Процедура должна самоисцеляться руками пользователя, а админ — получать готовую заплатку.

## Что сделать

* Процедуры — версионируемые md-файлы в репо: `procedures/get_its_docs.md`, `procedures/dump_config.md`, `procedures/update_docs.md` (и расширяемо).
* Инструмент `get_procedure(name)` — агент читает процедуру и ведёт пользователя по шагам.
* Ветка «чиним вместе»: если шаг не сошёлся — агент с пользователем нащупывают новый рабочий путь.
* Инструмент `report_procedure_update(name, new_version_text, what_broke)` — записывает обновлённую процедуру + телеметрию (что сломалось, на каком шаге) в лог-папку для админа (вне гита; push невозможен, Сергей смотрит сам).
* Стартовое наполнение процедур — лучшее текущее знание (получение офлайн-ИТС: ссылки [its.1c.ru/itsoffline](<http://its.1c.ru/itsoffline>), активация по договору; выгрузка конфигуратором в XML). Часть может быть «предположением» — это нормально, первый реальный прогон уточнит.

## Definition of done

* Процедуры лежат в репо, версионируются.
* `get_procedure` отдаёт процедуру, агент ведёт по шагам.
* При сбое шага агент переходит в режим совместного поиска решения.
* `report_procedure_update` сохраняет новую версию + телеметрию в доступное админу место.
* Документировано, как админ ревьюит и вкатывает обновление в эталон.
