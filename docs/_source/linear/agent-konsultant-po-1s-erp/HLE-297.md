Не реверсить XML-формат 1С с нуля. Использовать как референс спецификации из **cc-1c-skills** ([github.com/Nikolay-Shirokov/cc-1c-skills](<http://github.com/Nikolay-Shirokov/cc-1c-skills>), MIT), завендоренные в `reference/1c-specs/` задачей 0d: `1c-config-objects-spec.md` (23 типа объектов), `1c-configuration-spec.md` (Configuration.xml, ConfigDumpInfo.xml, 44 типа ChildObjects), `1c-help-spec.md` (встроенная справка Help.xml), `1c-form-spec.md`, `1c-dcs-spec.md` (СКД), `1c-extension-spec.md` (расширения CFE). Также конспект `docs/development/1C_FORMATS.md` (из 0d). Это reference-материал, НЕ рантайм-зависимость — берём знание формата, не их код.

## Что сделать

* Рекурсивный обход дерева XML-выгрузки. Типы объектов: Справочники, Документы, Регистры (накопления/сведений/бухгалтерии/расчёта), Перечисления, Планы видов характеристик, Обработки, Отчёты, Общие модули и т.д.
* Для каждого объекта: имя (синоним + имя), тип, реквизиты, табличные части, предопределённые элементы, **встроенная справка** (по 1c-help-spec).
* Связи: какие регистры затрагивает документ (движения), типы реквизитов-ссылок, подчинённость справочников.
* Тексты модулей (для будущего поиска по коду 1С).
* **Мультиконфигурационность:** `config_name` + пространство имён, чтобы несколько конфигураций не путались.
* **Расширения (CFE):** парсить выгрузки расширений (по 1c-extension-spec), помечать заимствованные/добавленные объекты, связывать с базовой конфой.

## Definition of done

* На реальной выгрузке ERP строится структурный индекс: объект → реквизиты, связи, справка, модули.
* «Какие регистры затрагивает документ X» → корректный список из движений.
* Несколько конфигураций и расширения не конфликтуют.
* Юнит-тесты на синтетической мини-выгрузке в репо.
