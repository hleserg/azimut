## 1. Карта репо

### Структура верхнего уровня

```
.claude/skills/         # 68 скиллов для Claude Code (основная ветка = PowerShell)
docs/                   # спецификации XML + гайды (~35 файлов)
scripts/switch.py       # переключение платформ (13 платформ: PS, Python, bash и др.)
.claude-plugin/         # плагин-манифест (plugin.json)
.agents/plugins/        # marketplace.json
tests/                  # тесты (не изучались)
LICENSE                 # MIT
README.md               # обзор проекта
```

**Ключевая идея (из README.md):**
> «Навыки дают модели готовые абстракции над XML-форматами и CLI конфигуратора — чтобы работать с сутью задачи, а не с деталями реализации.»

### Структура одного скилла

Каждый скилл = папка `.claude/skills/<name>/` с файлами:
- `SKILL.md` — основной файл: frontmatter (name, description, argument-hint, allowed-tools) + инструкция для модели
- `scripts/<name>.ps1` (или `.py` / `.sh`) — исполняемый скрипт
- `reference/` (в части скиллов) — child-operations.md, properties-reference.md, json-dsl.md, types-*.md

**Frontmatter SKILL.md (образец meta-info):**
```yaml
---
name: meta-info
description: Анализ структуры объекта метаданных 1С из XML-выгрузки — реквизиты, 
             табличные части, формы, движения, типы. Используй для изучения структуры 
             объектов (вместо чтения XML-файлов напрямую) и как подготовительный шаг 
             при написании запросов и кода, работающего с объектами
argument-hint: <ObjectPath> [-Mode overview|brief|full] [-Name <элемент>]
allowed-tools:
  - Bash
  - Read
  - Glob
---
```

## 2. Промтовые приёмы

### 2.1. Анти-выдумка: запрет обращения к реальным примерам

**Источник:** `.claude/skills/meta-compile/SKILL.md`

> «Эта инструкция и reference-файлы — **полная документация для генерации. Не ищи примеры XML в выгрузках конфигураций.**»

**Применение у нас (Азимут):** Прямой аналог нашего «отвечай только из источника». В системный промт: запрет выдумывать имена атрибутов, форматы полей и структуру объектов «из головы» — только то, что задокументировано в системе или получено через инструменты. Формулировка может быть адаптирована дословно.

---

### 2.2. Абстракция через «вместо»: предлагать альтернативу, а не запрет

**Источник:** `description` в SKILL.md скиллов form-info, meta-info

- `form-info`: «**Заменяет чтение тысяч строк XML**»
- `meta-info`: «Используй **вместо чтения XML-файлов напрямую**»

**Применение у нас:** В описаниях инструментов MCP явно указывать, что именно этот инструмент заменяет. Пример: «Используй вместо прямого разбора конфигурации», «Заменяет ручной поиск по XML-выгрузке». Это направляет модель к правильному инструменту без лишних запретов.

---

### 2.3. Указание контекста использования в description

**Источник:** `description` в SKILL.md нескольких скиллов

- `form-info`: «Используй для понимания формы — **при написании модуля формы, анализе обработчиков и элементов**»
- `meta-info`: «Используй **как подготовительный шаг при написании запросов и кода**»
- `form-patterns`: «Вызывай **перед** проектированием формы через `/form-compile`, когда требования пользователя не детализируют расположение элементов»
- `cf-info`: «Используй для обзора конфигурации — **какие объекты есть, сколько их, какие настройки**»

**Применение у нас:** В `description` MCP-инструментов добавлять: *когда* использовать, *что предшествует* и *что следует за* этим вызовом. Модель выбирает инструменты по description — без контекста выбор будет случайным.

---

### 2.4. Явный рабочий цикл (порядок использования инструментов)

**Источник:** `docs/meta-guide.md`

> «Описание объекта (текст) → JSON DSL → /meta-compile → XML-исходники → /meta-validate»

Не просто перечень инструментов, а **их порядок**. Модели показывается, какой шаг идёт после какого.

**Применение у нас:** В системный промт добавить типовые воркфлоу для основных сценариев запросов (например: «сначала `search_metadata`, потом `get_object`, потом отвечай»).

---

### 2.5. Управление детализацией (режимы + drill-down + пагинация)

**Источник:** `.claude/skills/meta-info/SKILL.md`

Три уровня detail:
- `overview` (default) — краткая сводка
- `brief` — основные поля
- `full` — всё

Плюс: `-Name <элемент>` для drill-down в конкретный элемент, `-Limit 150 -Offset N` для пагинации.

**Применение у нас:** Проектировать MCP-инструменты с явными уровнями детализации и drill-down. Не отдавать всё сразу — только по запросу. Это управление токенами и фокусом модели.

---

### 2.6. Паттерны вместо правил (готовые архетипы)

**Источник:** `.claude/skills/form-patterns/SKILL.md`

Вместо абстрактных правил — именованные архетипы с готовой структурой:
```
Форма документа:
Шапка (horizontal, 2 колонки)
├─ Левая: НомерДата (H: Номер + Дата "от"), Контрагент, Договор
├─ Правая: Организация, Подразделение, ЦеныИВалюта
```

Плюс конвенции именования:
| Назначение | Имя | Тип |
|---|---|---|
| Шапка | `ГруппаШапка` | horizontal |
| Левая колонка | `ГруппаШапкаЛевая` | vertical |
| Обработчик OnChange | `<Поле>ПриИзменении` | — |

**Применение у нас:** Для области консультирования по ERP — аналогичный набор архетипов (типовые структуры регистров, типовые документы, типовые отчёты). Модель работает с «именем архетипа», а не с перечислением полей каждый раз.

---

### 2.7. Shorthand-синтаксис вместо полного JSON

**Источник:** `.claude/skills/meta-compile/SKILL.md`

```
"ИмяРеквизита: Тип | req, index"
"Товары: Ном: CatalogRef.Ном, Кол: Число(15,3)"
```

Сначала shorthand — для простых случаев, полный JSON — только для сложных.

**Применение у нас:** При проектировании запросов к нашей системе — предлагать краткие формы для типовых запросов. Снижает нагрузку на модель и вероятность ошибок форматирования.

---

### 2.8. Batch через разделитель

**Источник:** `.claude/skills/meta-compile/SKILL.md`

```
-Value "Комментарий: Строка(200) ;; Сумма: Число(15,2)"
```

Несколько операций в одном вызове через `;;` — сокращает число шагов агента.

**Применение у нас:** Если у нас будут инструменты с повторяющимися операциями — batch-режим через разделитель уменьшает round-trips и latency.

---

### 2.9. Явное ограничение инструментов на скилл

**Источник:** frontmatter SKILL.md (поле `allowed-tools`)

- `meta-info`: только `Bash`, `Read`, `Glob`
- `web-test`: все инструменты

**Применение у нас:** В системном промте или конфигурации MCP — явно ограничивать набор инструментов для конкретного режима работы агента. Консультант не должен писать файлы; разработчик — может.

---

### 2.10. Автоактивация по description (без знания имени скилла)

**Архитектурная идея:** Модель выбирает скилл по смысловому совпадению задачи с `description`, без явного указания имени. Это означает, что `description` должен точно описывать задачу (не функцию инструмента), включая *когда* и *вместо чего* его использовать.

**Применение у нас:** Описания MCP-инструментов — это не документация для разработчика, а инструкции для модели. Писать их с позиции «когда агент должен это вызвать».

### 3.2. Структура каталога объекта

**Источник:** `docs/meta-guide.md`

```
<MetaType>/<ObjectName>/
├── <ObjectName>.xml         # основное определение объекта
├── Ext/
│   ├── ObjectModule.bsl     # модуль объекта (не у всех типов)
│   ├── ManagerModule.bsl    # модуль менеджера (не у всех типов)
│   ├── Predefined.xml       # предопределённые (у справочников)
│   └── Flowchart.xml        # только бизнес-процессы
├── Forms/<ИмяФормы>/Ext/Form.xml
├── Templates/<ИмяМакета>/Ext/Template.xml
└── Commands/
```

**Матрица модулей по типам** (из `docs/1c-config-objects-spec.md`):

| Тип | ObjectModule | ManagerModule | RecordSetModule |
|---|---|---|---|
| Справочник | + | + | − |
| Документ | + | + | − |
| Регистры | − | + | + |
| Перечисление | − | + | − |
| Отчёт, Обработка | + | + | − |

**Переиспользуемость:** Готовая структура для объяснения агенту, где искать код. Не пересекается с HLE-317 (там XML-форматы, не файловая структура выгрузки). Брать напрямую.

### 3.3. Классификация 23 типов (уровень meta-* скиллов)

**Источник:** `.claude/skills/meta-info/SKILL.md` (русская классификация)

> **Ссылочные:** Справочник, Документ, Перечисление, Бизнес-процесс, Задача, План обмена, План счетов, ПВХ (ChartOfCharacteristicTypes), ПВР (ChartOfCalculationTypes)

### 3.4. DSL-абстракции: слой между моделью и XML

**Источник:** `docs/` + `.claude/skills/*/reference/`

Пять DSL-систем (JSON → XML):

| DSL | Файл | Назначение |
|---|---|---|
| Meta DSL | `docs/meta-dsl-spec.md` | 23 типа объектов метаданных; DSL↔XML маппинг; автогенерация синонимов из CamelCase |
| Form DSL | `docs/form-dsl-spec.md` | Form.xml; архетипы форм; конвенции |
| SKD DSL | `docs/skd-dsl-spec.md` | DataCompositionSchema.xml; 930 схем проанализировано |
| MXL DSL | `docs/mxl-dsl-spec.md` | SpreadsheetDocument.xml |
| Role DSL | `docs/role-dsl-spec.md` | Rights.xml |

Дополнительно: русские синонимы ключей JSON (`реквизиты`, `тч`, `измерения`) — снижает путаницу при работе с кириллическими именами.

**Переиспользуемость для Азимута:** DSL как таковые нам не нужны (мы не генерируем XML). Но **логика абстракций** — как структурировать знание о типах метаданных для модели — прямо применима. В частности: таблицы маппинга типов, автоматические синонимы, иерархические справочники свойств.

**Связь с HLE-317:** Сами DSL-спецификации (форматы XML) могут пересекаться с тем, что вендорит HLE-317. Согласовать: если HLE-317 берёт `docs/form-dsl-spec.md` или `docs/meta-dsl-spec.md` как источник XML-спеков — убедиться, что берём одну и ту же версию.

## 4. Что НЕ брать

| Элемент | Причина |
|---|---|
| `scripts/*.ps1`, `scripts/*.py` | Заточены под разработку: работают с файловой системой, запускают 1С-конфигуратор, управляют базами. Не применимо к MCP-консультанту. |
| Скиллы `db-*` | Управление базами данных 1С (создание, загрузка, обновление). Инфраструктурные, для CI/CD. |
| Скиллы `web-*` | Публикация web-расширений 1С, тестирование через Playwright. Не наша область. |
| Компиляторы XML (`meta-compile`, `form-compile`, `mxl-compile`, `skd-compile`, `role-compile`) | Генерируют XML-файлы конфигурации. Нам генерировать XML не нужно. |
| `scripts/switch.py` | Инструмент сборки для смены платформы исполнения (PS/Python/bash). Не промт-инжиниринг. |
| Формат `.claude/skills/` (скилл-архитектура) | Подключается как плагин в Claude Code / Cursor / Codex. У нас другая архитектура — MCP-сервер. Структура нерелевантна, содержимое SKILL.md — полезно. |
| `web-test` | Браузерное тестирование публикаций через Playwright. |
