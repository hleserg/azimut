# Агент-консультант по 1С ERP — полное описание проекта

> **Одной фразой:** не «улучшенный поиск», а **компетентный специалист** — собирает контекст проблемы, сам копает в реальных источниках по 1С, отвечает по делу и не выдумывает.

**Версия документа:** 1.0 · май 2026 · **Автор:** Сергей Хлебников
**Репозиторий:** [github.com/hleserg/1c-consultant](http://github.com/hleserg/1c-consultant) · **Лицензия:** MIT · **Язык:** Python ≥ 3.11

Это главный документ проекта — на него опираемся весь оставшийся путь. Первые разделы написаны простым языком (можно читать без технического бэкграунда), ближе к концу идёт техническая глубина.

---

## 1\. В двух словах

**Что это.** Личный ИИ-консультант по 1С ERP. Программа, которая подключается к Claude (умный чат-помощник) и даёт ему «глаза» на конкретную конфигурацию 1С: её устройство и всю документацию. После этого с Claude можно разговаривать про работу в 1С как с живым грамотным специалистом — спрашивать своими словами, кидать скриншоты, описывать проблему, — а он находит ответ в настоящей документации и объясняет, что делать.

**Для кого.** В первую очередь для нас двоих — Сергея и мамы — для ежедневной работы с 1С. Никаких чужих серверов: всё крутится на своём компьютере, данные никуда не утекают.

**Почему не просто «спросить у чата».** Обычный ИИ про 1С часто фантазирует: уверенно выдумывает кнопки и пути, которых нет. Официальная поддержка отвечает медленно и формально. Наш помощник устроен так, чтобы отвечать **только из настоящей документации**, а если чего-то не нашёл — честно идти искать дальше, а не сочинять.

---

## 2\. Проблема, которую решаем

Работа в 1С ERP постоянно упирается в вопросы «как тут сделать вот это», «почему документ не проводится», «куда нажать, чтобы получился нужный отчёт». Есть три привычных пути найти ответ — и у каждого свой изъян:

| Куда обращаются сейчас | В чём проблема |
| -- | -- |
| Линия консультаций ИТС (живые люди) | Медленно и часто формально — типовые отписки, не вникая в конкретную ситуацию. |
| Поиск по базе знаний ИТС | Работает коряво: выдаёт не то, тонет в нерелевантном, не понимает вопрос «своими словами». |
| Обычный ИИ-чат | Фантазирует: уверенно описывает несуществующие кнопки и пути. Не знает вашу конкретную конфигурацию и её доработки. |

**Суть проблемы:** знание уже существует — в документации ИТС, в техдокументации платформы и в самой конфигурации (её структуру можно выгрузить). Проблема не в отсутствии знаний, а в том, что до них тяжело и долго добираться, а имеющиеся инструменты либо ленивы, либо врут. Нужен посредник, который доберётся до правды быстро, поймёт вопрос по-человечески и ответит честно: из источников, а не из головы.

## 3\. Как мы это решаем

Идея: дать Claude **инструменты-«руки»**, которыми он сам лезет в реальные источники по 1С, и **строгие правила поведения**, которые заставляют его этими руками пользоваться и не выдумывать.

Технически это **MCP-сервер** (Model Context Protocol) — локальная программа, которая подключается к настольному приложению Claude и предоставляет ему набор инструментов: поиск по документации, просмотр структуры конфигурации, показ картинок, онбординг и другие.

Пользовательский путь предельно простой:

1. **Один раз настроить** — указать помощнику, где лежат данные (выгрузка конфигурации, документация); он сам подскажет, как их получить.
2. **Подождать индексацию** — помощник «прочитывает» всю документацию и устройство конфигурации, чтобы потом быстро искать.
3. **Пользоваться** — открываешь Claude и просто спрашиваешь про 1С. Он сам ищет, читает и отвечает.

> **Ключевая мысль архитектуры:** вся «дотошность» вынесена в **сервер**, а не в модель. Сервер сам отдаёт полные разделы, разворачивает вложенность и сообщает, сколько ещё материала осталось непросмотренным — чтобы модель не останавливалась на первом, прочитанном наполовину фрагменте.

## 4\. Откуда помощник берёт знания

Четыре источника. Последний (личные документы) — по желанию. Никакого скрейпинга закрытого ИТС: только то, к чему у пользователя есть законный доступ.

| Источник | Что даёт | Как получаем |
| -- | -- | -- |
| **Выгрузка конфигурации в XML** | Полное устройство: объекты, реквизиты, связи, движения регистров, встроенная справка | Конфигуратор → «Выгрузить конфигурацию в файлы» |
| **Офлайн-версия 1С:ИТС** | Прикладная документация по конфигурации (как делать в ERP то-то и то-то) | [its.1c.ru/itsoffline](http://its.1c.ru/itsoffline), по действующей подписке |
| **Портал техдокументации платформы** | Документация по самой платформе 1С:Предприятие | Бесплатный официальный портал |
| **Личные документы (по желанию)** | Свои регламенты, описания доработок, инструкции — помощник ищет и по ним | Папка на компьютере, указывается при настройке |

**Подспорье для разработки.** Чтобы не разбирать XML-формат 1С с нуля, опираемся на готовые спецификации форматов из открытого проекта **cc-1c-skills** ([github.com/Nikolay-Shirokov/cc-1c-skills](http://github.com/Nikolay-Shirokov/cc-1c-skills), MIT). Это справочный материал для нашего парсера, а не зависимость в работе готового продукта — спеки завендорены в `reference/1c-specs/`.

> **Про состав источников — открытый вопрос под эксперимент.** Подключать ли в поиск всю доку или только часть — пока не решено. Есть наблюдение из практики: лишняя документация может ронять качество ответов. Поэтому «какие источники держать в поиске» определим **по данным обкатки**, а не на старте (см. раздел 8 и задачу 13).

## 5\. Архитектура

Система — локальный MCP-сервер и набор слоёв внутри него. Сервер общается с настольным Claude, а Claude — с пользователем.

### 5.1. Общая схема

```
            ПОЛЬЗОВАТЕЛЬ
   вопрос своими словами + скриншоты
                  │
                  ▼
        Claude (настольное приложение)
   разговор, рассуждение, зрение для скриншотов
                  │
                  ▼
   MCP-СЕРВЕР · инструменты + поведенческий контракт
   search_documentation, get_object, show_image, healthcheck …
                  │
      ┌───────────┼───────────────┬──────────────────┐
      ▼           ▼               ▼                  ▼
 Поисковый    Менеджер        Реестр            Живые
   слой       модели        документов        процедуры
      │
      ▼
   ┌──────────────────┬──────────────────┬─────────────────┐
   ▼                  ▼                  ▼                 ▼
 Парсер           Индексатор         Парсер            PII-фильтр
 конфигурации     документации       личных            (маскировка
 (XML)            + OCR              документов        перед индексом)
   │                  │                  │
   ▼                  ▼                  ▼
 Выгрузка 1С      Офлайн-ИТС +       Папка личных
 (XML)            доки платформы     документов
                       │
                       ▼
                 ВЕКТОРНЫЙ ИНДЕКС (Qdrant, встроенный режим)
```

### 5.2. Слои и их роли

| Слой | Зачем |
| -- | -- |
| **Парсер конфигурации** | Читает XML-выгрузку: объекты, реквизиты, связи, движения регистров, встроенную справку. Источник истины о структуре. |
| **Индексатор документации** | Разбирает офлайн-ИТС и доку платформы, сохраняет иерархию разделов, делает OCR картинок (скриншоты форм), готовит к поиску. |
| **Парсер личных документов** | Читает свою папку: txt, md, pdf (текст и скан), docx, doc/odt, pptx, csv, xml, json, html, eml. Рекурсивно, с OCR. |
| **Реестр документов** | Помнит, какие файлы и когда обработаны и с каким результатом. Индексирует только изменившееся, ловит сбои, убирает из поиска удалённые файлы. |
| **PII-фильтр** | Маскирует персональные данные (ФИО, телефоны, ИНН, реквизиты счетов) в личных документах и нетиповом коде до того, как они попадут в поиск и в облако. Локально, на CPU. |
| **Менеджер модели** | Управляет локальной моделью эмбеддингов: грузит при необходимости, греет заранее, выгружает при простое. |
| **Поисковый слой** | Гибридный поиск + переранжирование, возврат полных разделов и «сигнала полноты». Сердце качества ответов. |
| **MCP-сервер** | Набор инструментов с обязывающими описаниями + онбординг. То, что видит Claude. |
| **Живые процедуры** | Пошаговые инструкции (получить доку, сделать выгрузку), которые самоисцеляются, если 1С поменяла интерфейс. |
| **Поведенческий контракт** | Системный промпт и правила, превращающие Claude в дотошного спеца. |

## 6\. Как это работает внутри

### 6.1. Подготовка данных (индексация)

Разовый процесс (и потом по обновлениям): из сырых источников строится поисковый индекс.

```
Сырые источники → Парсинг + иерархия → PII-фильтр (маскировка) →
→ Чанкинг + OCR → Эмбеддинги BGE-M3 (dense+sparse) → Индекс (Qdrant)
                                    │
                          Реестр документов фиксирует:
                       что обработано, статус, хеш, число чанков
```

### 6.2. Обработка вопроса пользователя

```
Вопрос пользователя
(при неясности — агент уточняет: версия, ошибка, скриншот)
        │
        ▼
Поиск: гибрид (плотный + лексический), BGE-M3 → топ кандидатов
        │
        ▼
Переранжирование: Cohere Rerank v4 (или локальный фолбэк)
        │
        ▼
Адаптивное отсечение по «разрыву» в скорах (не фиксированный top_k)
        │
        ▼
Сборка полных разделов + сигнал полноты («есть ещё N разделов»)
        │
        ▼
Claude дочитывает и отвечает из источников
(без фантазий; показывает скриншоты форм)
```

### 6.3. Два «гена» поведения, ради которых всё строилось

* **Проблема 1:** ИИ делает вид, что инструментов нет, пока его прямо не пнёшь.
  **Решение:** агрессивные обязывающие описания инструментов — «отвечать про 1С из памяти ЗАПРЕЩЕНО, только из результатов поиска».
* **Проблема 2:** ИИ читает по диагонали — первый документ наполовину, у вложенных пробегает по заголовкам.
  **Решение:** дотошность обеспечивает сервер — отдаёт полные разделы, разворачивает вложенность и явно сигналит «есть ещё N разделов».

## 7\. Технические решения и почему так

Часть решений зафиксирована сразу, часть осознанно отложена «до данных» — их примем по результатам реальной обкатки, а не угадывая на старте.

### 7.1. Зафиксированный стек

| Компонент | Выбор | Почему |
| -- | -- | -- |
| Язык / окружение | **Python ≥ 3.11, uv** | Богатая ML-экосистема, быстрый менеджер зависимостей uv (как в проекте atman). |
| MCP-фреймворк | **FastMCP** | Стандартный способ отдать инструменты настольному Claude. |
| Разговорная модель | **Облачный Claude** | Сильное рассуждение; нативно видит скриншоты; служит и судьёй качества. |
| Эмбеддинги | **BGE-M3 (FlagEmbedding)** | Локально, бесплатно, мультиязычно. Гибрид dense+sparse одним вызовом — точный поиск по терминам 1С. |
| Переранжирование | **Cohere Rerank v4** | Грант Cohere $1000 покрывает с запасом (\~$2/1000 поисков). Фолбэк — локальный BGE-reranker, если ключа нет. |
| Векторное хранилище | **Qdrant (встроенный режим)** | Хранит плотные и лексические векторы в одной коллекции, несколько представлений на чанк (под эксперимент), запускается прямо внутри программы — без отдельного сервера. Маме не ставить лишнюю сущность. |
| Картинки в доке | **OCR + текст** | Скриншоты форм через распознавание (pytesseract + easyocr для кириллицы). Мультимодал Cohere Embed 4 — кандидат на обкатку. |
| Защита данных | **Presidio + ru_core_news_sm** | Локальное распознавание персональных данных под русский язык и российские реквизиты. На CPU, до отправки в облако. |
| Учёт файлов | **Реестр на SQLite** | Инкрементальная индексация, детект сбоев и удалённых файлов — единый механизм для всех источников. |
| Оценка качества | **LLM-судья = Claude** | Вне поискового конвейера → независимый арбитр. Cohere Command — опционально второй голос. |
| Наблюдаемость | **Sentry** | Логи, трейсинг, метрики, AI-мониторинг. Телеметрия пользователя — строго по согласию (opt-in). |

**Почему Qdrant, а не Milvus / pgvector.** Milvus Lite капризен на Windows (а у пользователей Windows) и избыточен для однопользовательского сценария. pgvector требует поднимать и держать PostgreSQL — лишняя сущность для нетехнического пользователя. Qdrant embedded даёт нативный гибрид и ноль внешних сервисов.

### 7.2. Отложенные решения (примем по данным обкатки)

* **Финальная стратегия поиска** — dense / гибрид / гибрид+переранжирование: что победит по качеству и цене.
* **Картинки: OCR против мультимодального эмбеддинга** — что точнее на реальных скриншотах 1С.
* **Размерность и квантизация** эмбеддингов — по размеру индекса и качеству.
* **Состав источников под поиск** — вся ли дока подключается или часть (лишняя дока может ронять эффективность).

> **Принцип принятия решений:** где можно — решаем данными, а не интуицией. Поэтому в системе есть режим эксперимента: несколько стратегий работают параллельно, копится телеметрия, финальный выбор делается по фактам (качество против цены против скорости).

### 7.3. Ограничения, принятые как данность

| Ограничение | Как живём с ним |
| -- | -- |
| MCP-сервер не может сам написать пользователю | Нет push-уведомлений. Прогресс индексации и сигнал «готово» агент узнаёт, опрашивая сервер при каждом обращении (healthcheck / get_progress). |
| Грант Sentry привязан к организации | Использование под новый проект согласовано с поддержкой Sentry — можно. Блокер де-факто снят. |
| «Канарейка» о деградации | Система сама сигналит Сергею (админу) в Sentry до того, как пользователь начнёт раздражаться: латентность, здоровье индекса, RAM, рост фолбэков. Метрики собираются только в потоке работы (на запросах и healthcheck), фонового демона нет. |

## 8\. Конфиденциальность и безопасность данных

Важный для нас блок — особенно потому, что в личных документах бывает всё.

* **Личные документы — необязательный источник.** Если их не подключать, все механизмы вокруг них молчат.
* **Маскировка персональных данных.** Перед попаданием в поиск из личных документов и нетипового кода автоматически вычищаются ФИО, телефоны, ИНН, СНИЛС, паспорта, ОГРН/КПП, расчётные счета, БИК, email, IP, токены и пароли, номера карт. Делает это локальный фильтр (Presidio + русская модель). Исходные файлы на диске **не меняются** — маскируется только то, что уходит в индекс и потом в облако.
* **Предупреждение о коммерческой тайне.** При первом подключении личных документов с облачной моделью помощник один раз предупреждает: если в документах есть коммерческая тайна — стоит уточнить у руководства, можно ли отправлять фрагменты в облако.
* **Телеметрия — только по согласию (opt-in).** По умолчанию **выключено**. При онбординге помощник один раз мягко спрашивает разрешение отправлять разработчику технические отчёты об ошибках (без документов и личных данных). Отключается в любой момент командой.

## 9\. Возможности и функции

### 9.1. Что умеет помощник (для пользователя)

* **Отвечает из реальной документации**, а не из головы; при нехватке данных честно идёт искать дальше.
* **Понимает вопрос своими словами** и видит скриншоты.
* **Копает вглубь** — дочитывает вложенные разделы, не останавливается на первом фрагменте.
* **Помогает с настройкой по шагам** — как скачать документацию, как сделать выгрузку, ведёт за руку.
* **Ищет и по вашим документам** — если подключить папку со своими регламентами. Можно добавить/убрать в любой момент, просто попросив.
* **Бережёт данные и спрашивает разрешения** — маскирует персональные данные до отправки в облако, отчёты об ошибках шлёт только с согласия.
* **Ненавязчиво предлагает обновление** документации раз в месяц и согласует переиндексацию заранее.

### 9.2. Инструменты MCP-сервера (для разработки)

| Инструмент | Назначение |
| -- | -- |
| `search_documentation` | Гибридный поиск по документации. Обязателен к вызову перед любым ответом про 1С. |
| `get_section` | Раздел документации целиком по идентификатору (с дочерними). |
| `search_metadata` | Поиск по объектам конфигурации. |
| `get_object` | Полная карточка объекта: реквизиты, связи, движения, справка. |
| `list_related` | Связанные разделы и объекты (разворачивание вложенности). |
| `show_image` | Показ картинки/скриншота из документации. |
| `healthcheck` | Проверка готовности; заодно греет модель в фоне. |
| `validate_path` | Проверка указанного пути к данным. |
| `save_config` | Сохранение путей и запуск автоиндексации. |
| `get_progress` | Статус индексации: этап, процент, оценка времени. |

**Условный показ Cohere.** Инструменты и стратегии на базе Cohere (переранжирование, мультимодал, второй голос судьи) предлагаются только при наличии ключа. Нет ключа — опции просто не видны, всё работает на локальном стеке без падений.

## 11\. Как работают агенты-кодеры

Проект строят ИИ-агенты (Claude Code) с автоматическим ревью (Devin) и проверками. Чтобы они работали аккуратно, в каждую задачу и в описание проекта зашит жёсткий протокол.

### 11.1. Статусы и сдача задачи

* **Взял задачу** → перевёл в In Progress.
* **Не трогаю** задачу, чей блокер ещё не в Done.
* **Перед сдачей** — перечитать задание и Definition of Done, сверить каждый пункт.
* **Всё на 100% и уверен** → отчёт-комментарий + In Review (финальный Done ставит Сергей).
* **Частично/не уверен** → In Review + в комментарии что не сделано и почему. Не выдавать частичное за полное.

### 11.2. Git-workflow (жёсткое правило)

Цикл «до полного зелёного»:

1. На main — создать ветку перед работой; в ветке можно несколько задач.
2. PR (можно draft) → снять draft → дождаться окончания всех проверок и всех ревьюверов (Devin, Sentry-ревью, любые жёлтые).
3. Исправить все комментарии ревью, все ошибки CI, все конфликты; закрыть починенные замечания.
4. Повторять, пока не станет: все проверки пройдены, ноль замечаний, ноль ошибок, ноль конфликтов.
5. Только тогда — мерджить. Не выходит — написать «не могу» с причиной и остановиться.

> **Страховка:** branch protection в GitHub настроен так, что merge технически невозможен без зелёных обязательных проверок — правило держится не только на тексте.

## 12\. Реалистичная планка

«Полностью без фантазий» — это **направление, а не выключатель**. Стопроцентной гарантии не бывает ни у одной ИИ-системы. Наша цель — устойчивое поведение: **«почти всегда сам идёт в источники и отвечает по делу»**.
