## Три источника знаний агента

Агент работает с тремя независимыми папками данных — каждая индексируется отдельно:

| № | Источник | Что там | Кто настраивает |
| -- | -- | -- | -- |
| 1 | **Выгрузка конфигурации** | XML-выгрузка из Конфигуратора — структура объектов, реквизиты, связи, встроенная справка | Сергей при подготовке данных |
| 2 | **Офлайн-ИТС + платформа** | Техническая документация 1С:ИТС (по подписке) и портал документации платформы | Сергей при подготовке данных |
| 3 | **Личные документы** | Регламенты, описания доработок, инструкции, любая рабочая документация пользователя | Пользователь настраивает сам через агента |

Папка с личными документами — **опциональная**. При первой настройке агент спросит о ней. Если пользователь откажется — механика отключается и не беспокоит до тех пор, пока пользователь сам не попросит её добавить.

## Поддерживаемые форматы личных документов

**.txt, .md** — обычный текст и Markdown
**.pdf** — текстовые PDF и сканы (OCR)
**.docx** — Word-документы, включая встроенные картинки
**.doc, .odt** — старый Word и LibreOffice (конвертация через LibreOffice headless)
**.pptx** — презентации (текст слайдов и заметки)
**.csv** — таблицы как текст
**.xml, .json** — структурированные данные
**.html, .htm** — веб-страницы
**.eml** — письма

**Excel (.xlsx / .xls)** — не парсим содержимое. Если в файле описание процесса или регламент — лучше сохранить его как `.docx`. Если там расчёты — это за пределами специализации консультанта по 1С, и это нормально.

Файлы других форматов будут обнаружены, но не прочитаны — агент сообщит об этом и подскажет, что можно сделать.

**Вложенность папок** не ограничена — агент рекурсивно проходит по всем подпапкам любой глубины.

## Реестр документов (Document Registry)

Все три источника используют единый реестр (SQLite). Для каждого файла записывается:

* SHA-256 хеш содержимого (основа инкрементальной индексации)
* Дата и результат последней индексации
* Статус: успешно / ошибка / advisory (нужна конвертация) / удалён

Это даёт три практических преимущества:

**Инкрементальность** — повторная индексация неизменных файлов пропускается автоматически. Переиндексируется только то, что изменилось.

**Прозрачность** — агент знает, какие файлы не удалось прочитать, и сообщает об этом. Пользователь не теряет файлы «в тишине».

**Автоочистка** — удалённые файлы автоматически убираются из поиска. Программа не падает, выдача остаётся чистой.

## Мониторинг изменений в папке личных документов

При каждом возвращении пользователя агент проверяет: появились ли новые файлы, изменились ли существующие. Если да — предлагает переиндексацию ненавязчиво, раз за сессию:

> «Кстати, в папке с вашими документами что-то изменилось с последнего раза. Переиндексировать, чтобы обновления участвовали в поиске?»

---

## Умный контекст: адаптивный cut-off

Вместо фиксированного числа чанков в контексте агент использует **адаптивный алгоритм**: анализирует распределение скоров реранкера и останавливается там, где начинается резкий «обрыв» релевантности. Это защищает от засорения контекста менее релевантным материалом.

Параметры настраиваемые. Данные для тонкой настройки собираются в фазе 4 (задача 21).

## Кеш запросов (опционально)

Включается в конфиге:

* **Exact-match** — одинаковый вопрос → мгновенный ответ из кеша (TTL: 5 минут)
* **Semantic** — похожий вопрос (схожесть ≥ 95%) → результаты поиска из кеша (TTL: 1 час)

При включённом режиме эксперимента кеш автоматически отключается — нужны реальные прогоны.

## Телеметрия и конфиденциальность

Агент может отправлять анонимные технические отчёты об ошибках разработчику (через Sentry). Это помогает быстрее находить и устранять баги.

**Правила:**

* Спрашивается **один раз** при первом запуске, вежливо:
  *«Если что-то пойдёт не так, могу автоматически отправить отчёт об ошибке разработчику — без ваших документов и личных данных, только технические детали. Это помогает быстрее находить и чинить баги. Разрешить?»*
* По умолчанию — **выключено** до явного согласия (GDPR)
* Пользователь может отключить в любой момент: «отключи телеметрию»
* Содержимое документов и личные данные не отправляются никогда

Если используется облачная языковая модель (Claude, GPT и др.) — агент один раз предупредит, что содержимое личных документов будет отправляться в облако при ответах на вопросы. Стоит уточнить в компании, нет ли ограничений по коммерческой тайне.

## Протокол агентов-кодеров

*(Полная версия — в AGENTS.md репозитория)*

1. Взял задачу → **In Progress**
2. Не трогать задачу, если блокер не в **Done**
3. Перед сдачей: перечитать задание + DoD, сверить каждый пункт
4. Всё сделано и уверен → отчёт-комментарий + **In Review** (Done ставит Сергей)
5. Частично/не уверен → **In Review** + в комментарии что не сделано и почему
6. Метка **\[ТЫ\]** = ручная задача Сергея, агент не берёт

**Git:** main → ветка → PR (draft) → снять draft → дождаться ВСЕХ проверок (CI + Devin + Sentry) → исправить всё → мердж только при полном зелёном.
