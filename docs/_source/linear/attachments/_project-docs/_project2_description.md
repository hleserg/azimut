# Агент-консультант по 1С ERP

Локальный MCP-агент, который помогает работать с типовой конфигурацией **1С:ERP**. Пользователи — Сергей и его мама: мама задаёт вопросы голосом, агент находит ответы в трёх источниках данных и отвечает по делу.

**Репозиторий:** [github.com/hleserg/1c-consultant](https://github.com/hleserg/1c-consultant) · MIT · Python ≥ 3.11

---

## Чем агент отличается от «улучшенного поиска»

Агент — это **специалист**, а не колл-центр. Он:

* сам собирает контекст проблемы
* идёт в источники и ищет там — не фантазирует из памяти
* отвечает только на основании найденного
* сам определяет, когда данных не хватает, и ищет ещё

Реалистичная планка: «почти всегда сам в источники, по делу». 100% гарантии нет.

## Три источника данных

| Источник | Что это | Кто настраивает |
| -- | -- | -- |
| XML-выгрузка конфигурации | Структура + связи + встроенная справка | Сергей (путь в конфиге) |
| Офлайн-ИТС | Документация по конфигурации и платформе | Сергей (путь в конфиге) |
| Личные документы | Свои регламенты, описания доработок, инструкции | Опционально, через агента |

Личные документы — **опциональный источник**. Если пользователь не хочет их подключать — агент не настаивает и не запускает никаких механизмов. Подключить можно в любой момент, сказав агенту.

## Стек

* **Python ≥ 3.11**, uv, FastMCP
* **Разговорная модель:** облачный Claude (зрение для скриншотов + LLM-судья)
* **Эмбеддинги:** BGE-M3 через FlagEmbedding (гибрид dense+sparse), локально
* **Реранкинг:** Cohere Rerank v4 (грант $1000), фолбэк — локальный BGE-reranker-v2-m3
* **Векторное хранилище:** Qdrant в embedded-режиме (в процессе Python, без отдельного сервера)
* **Реестр документов:** SQLite (хеши, статусы, история индексации для всех трёх источников)
* **Мониторинг:** Sentry (условно, только с явного согласия пользователя)

## Поддерживаемые форматы личных документов

Полностью читаем: `.txt`, `.md`, `.pdf` (текст и сканы через OCR), `.docx`, `.doc` (LibreOffice headless), `.odt` (LibreOffice headless), `.pptx`, `.csv`, `.xml`, `.json`, `.html`/`.htm`, `.eml`

Advisory — советуем конвертировать в .docx: `.xlsx`, `.xls`

Всё остальное — сообщаем что не поддерживается, список поддерживаемых прилагается.

## Ключевые архитектурные решения

**Почему агент сам идёт в инструменты** — в описаниях MCP-инструментов прямо написано «отвечать из памяти про 1С ЗАПРЕЩЕНО». Без этого агент склонен сначала ответить из памяти, а потом смотреть.

**Почему дотошность на стороне сервера** — сервер возвращает полные разделы, разворачивает вложенность, отдаёт сигнал «есть ещё N разделов». Агент физически не может «прочитать по диагонали».

**Document Registry** — SQLite-реестр с хешами, статусами (ok/error/unsupported/advisory/deleted) и историей индексации для всех трёх источников. Основа инкрементальной индексации и отчётов об ошибках.

**Адаптивный контекст** — cut-off по gap в скорах реранкера вместо фиксированного top_k. Исследования показали -60% токенов при сохранении качества. Hard cap ~2500 токенов.

**Семантический кеш** — два уровня: exact-match и semantic similarity (порог 0.95). Опционально, включается в конфиге.

**Sentry — с согласия, по умолчанию выключена** — при первом запуске агент спрашивает человеческим языком, дефолт — выключено. DSN хранится вне репозитория, доставляется через скрипт установки. Из Sentry намеренно исключены: содержимое документов, пути к файлам пользователя, тексты запросов.

## Рабочий протокол для агентов-кодеров

```
Взял задачу → In Progress
Блокер не в Done → не трогать задачу
Перед сдачей → перечитать задание + DoD, сверить каждый пункт
Всё 100% → отчёт-комментарий → In Review
Частично/не уверен → In Review + что не сделано и почему
Done ставит только Сергей
```

Задачи с меткой **\[ТЫ\]** в заголовке — ручные задачи Сергея, агенты их не берут.

## Git-workflow (жёсткое правило, в AGENTS.md)

```
main → ветка перед работой
PR (draft) → снять draft → ждать ВСЕХ проверок (CI + Devin + Sentry)
Есть комментарии/ошибки → исправить → resolve → повторить
Всё зелёное → мержить
Не получилось → «не могу» + стоп
```
