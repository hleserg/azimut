# LLM-судья со спан-привязкой (Claude как арбитр)

## Context and Problem Statement

Чанк есть, но модель «дорисовала» — это самый коварный класс галлюцинаций: ответ выглядит правдоподобно, формально опирается на найденные данные, а на деле часть утверждений нечем подкрепить. Нужен механизм, который проверяет ответ ПО СПАНАМ: каждое утверждение должно отображаться в конкретный фрагмент retrieved-контекста, иначе оно флажится как неподкреплённое.

## Decision Drivers

* Точность детекции «дорисовок» (см. фон в `decisions--*.md`: «соблазн полноты — чанк отвечает на 80%, модель достраивает 20%»).
* Возможность работать с облачным Claude как с чёрным ящиком (без доступа к logits/attention) — судья работает с текстом ответа и текстом чанков.
* Совместимость с Р2 (faithfulness как отдельная метрика, ADR 0002) — судья даёт сигнал именно про faithfulness.
* Возможность встроить судью в Sentry-пайплайн (доля флагов = метрика качества генерации).

## Considered Options

* (A) LLM-судья со спан-привязкой: судья сравнивает спаны ответа со спанами retrieved-контекста и помечает неподкреплённые куски.
* (B) Эмбеддинг-сходство «ответ ↔ контекст» (косинус) как прокси faithfulness.
* (C) Полагаться на встроенные механики Claude (system-prompt «отвечай строго по контексту») без внешнего судьи.

## Decision Outcome

Chosen option: «(A) LLM-судья со спан-привязкой», because только она даёт локализованный сигнал «вот это конкретное предложение не подтверждено» — это нужно и для блокировки ответа (П1 / ADR 0008 уровень 1), и для плашки «частично из общих знаний» (П1 уровень 2), и для канарейки в Sentry (П1 уровень 3). Эмбеддинг-сходство (B) даёт скаляр без локализации; system-prompt (C) — известный паттерн, который всё равно пропускает галлюцинации (та самая причина, по которой нужен внешний контроль).

### Consequences

* Good, because faithfulness становится измеримой и локализуемой — можно показать пользователю «вот это утверждение в источниках не найдено».
* Good, because судья работает поверх любой модели-генератора (Claude текст → Claude судья), не требует доступа в потроха.
* Bad, because дополнительный LLM-вызов на каждый ответ — латентность и стоимость.
* Bad, because судья сам — LLM, у него есть свой шум; нужна калибровка порога флага.

### Confirmation

* В `05-building-block-view.md` LLM-судья отрисован как отдельный контейнер.
* В `06-runtime-view.md` сценарий «запрос → ретривинг → ответ → судья» зафиксирован Mermaid-диаграммой.
* В eval-харнессе judge-флаги коррелируют с экспертной разметкой (≥ 0.7 каппа на пилотном датасете — целевая планка, фиксируется в §10).

## Pros and Cons of the Options

### (A) LLM-судья со спан-привязкой

* Good, because локализует неподкреплённое утверждение.
* Good, because даёт вход для трёх уровней действий П1.
* Bad, because дополнительный LLM-вызов (стоимость, латентность).

### (B) Эмбеддинг-сходство

* Good, because дёшево и быстро.
* Bad, because даёт только скаляр; не отвечает «какое именно утверждение не подтверждено».
* Bad, because слабо коррелирует с экспертной разметкой на длинных ответах.

### (C) Только system-prompt

* Good, because нулевая инфраструктура.
* Bad, because известный паттерн — пропускает «дорисовки полноты»; именно поэтому Р3 и появился.
