# Второй проход ретривера при неуверенности (открытый триггер)

## Context and Problem Statement

После первого прохода ретривера агент-генератор может оказаться «неуверен»: ответ строится с натяжкой, или агент явно сигнализирует, что данных не хватает. Вместо того чтобы сдаваться в фолбэк-режим (Р7 / ADR 0007), можно сначала попробовать второй проход ретривера с переформулированным запросом. Открытый вопрос: что считать триггером и кто инициирует.

## Decision Drivers

* Сохранение пользы первого прохода: ре-retrieval не должен срабатывать «всегда» — это сделает каждый ответ в N раз дороже.
* Совместимость с Р5 (контроль ретривинга на сервере, ADR 0005): даже если триггер задаёт агент, исполнение и гейты — на сервере.
* Защита от lost-in-the-middle: добор второго прохода не должен бесконтрольно расширять окно.
* Соответствие Adaptive-RAG / Self-RAG 2026 (повторный ретривинг по семантически уточнённому запросу).

## Considered Options

* (A) Второй проход ретривера по переформулированному агентом запросу, инициатор — агент, исполнитель — сервер; гейт триггера — открытый вопрос.
* (B) Один проход всегда, без повторов (упрощение).
* (C) Автоматический ре-retrieval сервером при любом результате ниже порога релевантности (без участия агента).

## Decision Outcome

Chosen option: «(A) агент-инициатор, сервер-исполнитель, триггер открыт», because:

* (B) теряет ценные ответы в случаях, когда переформулировка помогла бы.
* (C) перегружает запросы — не всякая «низкая релевантность» означает, что переформулировка спасёт; иногда вопрос просто непокрыт индексом, и нужен Р7-фолбэк, а не повторный ретривинг.
* (A) использует сильную сторону агента (семантическая переформулировка) и сильную сторону сервера (измеримые гейты).

Статус оставлен `proposed`: критерий «инициатор — агент, по какому именно сигналу» не зафиксирован. До закрытия вопроса повторный проход вызывается явным флагом из агента; гейт сервера ограничивает не больше N повторов и не выходит за бюджет окна.

### Consequences

* Good, because использует семантические способности агента (переформулировка) без передачи ему управления добором (Р5 сохраняется).
* Good, because бюджет окна и потолок повторов — на сервере, риск lost-in-the-middle ограничен.
* Bad, because триггер открыт — могут быть либо ложные срабатывания (избыточные повторы), либо пропуски (агент не догадается переформулировать). Требует калибровки.
* Bad, because суммарная латентность ответа в случае повтора растёт минимум вдвое.

### Confirmation

* В `06-runtime-view.md` зафиксирован путь «агент → флаг переформулировки → повторный ретривинг → новая выборка».
* На сервере есть лимит «не более N повторов на запрос» и метрика `re_retrieval_triggered`.
* При закрытии вопроса о триггере ADR переводится в `accepted`.

## Pros and Cons of the Options

### (A) Инициатор-агент, исполнитель-сервер

* Good, because разделение сильных сторон.
* Good, because бюджет на сервере.
* Bad, because триггер открыт, нужна калибровка.

### (B) Один проход

* Good, because просто и предсказуемо по стоимости.
* Bad, because ценные случаи (переформулировка спасает) теряются.

### (C) Авто-повтор сервером

* Good, because нет зависимости от агента.
* Bad, because низкая релевантность ≠ всегда «помогла бы переформулировка».
* Bad, because размывает границу с Р7 (фолбэк).
