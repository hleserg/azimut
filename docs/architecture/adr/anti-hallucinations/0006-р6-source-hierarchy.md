# Иерархия источников при конфликте: код → справка → ИТС

## Context and Problem Statement

Ретривер вытаскивает чанки из нескольких источников: код из конкретной базы 1С, встроенная справка из конфигурации, официальные доки с ИТС (its.1c.ru). Эти источники могут противоречить друг другу. Раннее (Р1, ADR 0001) предлагалось «убрать оба», но это лишает пользователя пользы. Нужен явный приоритет, по которому ответ объясняет варианты.

## Decision Drivers

* Польза для пользователя: ответ должен предложить наиболее вероятный вариант, а не отказ от выдачи.
* Семантика приоритетов: код базы — это «как сделано здесь и сейчас», доки конфига — «как задумано», ИТС — «как в типовой». Это естественный порядок «от факта к рекомендации».
* Совместимость с Р1 (ADR 0001): противоречие всё равно инкрементит метрику в Sentry — наблюдаемость не теряется.

## Considered Options

* (A) Иерархия источников: код → встроенная справка → ИТС. При конфликте объясняем все варианты по порядку, первый помечаем как наиболее вероятный.
* (B) Выбрасывать оба конфликтующих чанка (исходная формулировка Р1).
* (C) Отдавать все источники без приоритета и оставлять выбор пользователю.

## Decision Outcome

Chosen option: «(A) явная иерархия с объяснением вариантов», because:

* (B) убивает пользу: пользователь не узнаёт ни одного из вариантов.
* (C) перекладывает экспертизу на пользователя — он не обязан знать, какой источник перевешивает в его контексте.
* (A) даёт ответ с правильной семантикой («скорее всего так, потому что в коде; в типовой — иначе»).

### Consequences

* Good, because пользователь всегда получает ответ, причём с правильным приоритетом.
* Good, because противоречие не маскируется: оно ВИДНО в ответе («в коде так, в ИТС — иначе»).
* Good, because противоречие всё равно учитывается метрикой Р1 — наблюдаемость сохранена.
* Bad, because требуется аккуратная классификация источника каждого чанка (откуда он пришёл) — это техническое требование к индексу.

### Confirmation

* Ретривер маркирует каждый чанк типом источника (`code` / `config_help` / `its`).
* В системном промпте генератора есть правило приоритета.
* В §8 (cross-cutting) приоритет источников описан как сквозной принцип.
* На пилотном датасете противоречивых пар: ответ всегда начинается с варианта из источника верхнего приоритета.

## Pros and Cons of the Options

### (A) Явная иерархия

* Good, because пользователь получает ответ + информацию о расхождении.
* Good, because противоречие наблюдаемо (Р1).
* Bad, because требует маркировки источника в индексе.

### (B) Выбрасывать оба

* Good, because безопаснее «по умолчанию».
* Bad, because пользователь остаётся без ответа в самой ценной ситуации (есть нюанс).

### (C) Без приоритета

* Good, because честно показывает все варианты.
* Bad, because перекладывает экспертизу на пользователя; ответ выглядит как «вот вам три варианта, разбирайся».
