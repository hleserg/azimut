# Контроль ретривинга — на сервере (планка релевантности, триггер добора, потолок окна)

## Context and Problem Statement

Управление добором данных и решение «хватит ли контекста для ответа» должно где-то жить. Два варианта: на стороне агента-генератора (Claude) или на стороне MCP-сервера. У агента нет измеримых критериев «надо ещё / хватит» и нет управляемости — он может проигнорить даже хорошие данные или, наоборот, доехать на «общих знаниях». На сервере же всё измеримо: релевантность, бюджет токенов, наличие триггера добора, лог запросов.

## Decision Drivers

* Измеримость и калибруемость: пороги должны быть видимыми и настраиваемыми.
* Воспроизводимость: один и тот же запрос при одних и тех же данных должен давать один и тот же набор гейтов.
* Соответствие Self-RAG 2026 (сервер держит измеримые гейты, агент помогает по смыслу).
* Защита от lost-in-the-middle: добор не должен расширять окно бесконтрольно.

## Considered Options

* (A) Контроль ретривинга на MCP-сервере (планка релевантности, триггер добора, потолок окна, лог в Sentry). Агент — только семантическая переформулировка для повторного прохода.
* (B) Контроль на агенте (Claude сам решает «надо ещё / хватит» через function-calling).
* (C) Гибрид: агент решает, сервер — только исполнитель.

## Decision Outcome

Chosen option: «(A) контроль на сервере», because у сервера измеримые критерии (релевантность, бюджет, тренд в Sentry), у агента — нет. Гибрид (C) фактически сваливается в (B), потому что управляющие решения уходят туда, где принимаются. Агент остаётся полезен в семантической переформулировке запроса (это его сильная сторона) — но не в роли диспетчера ретривинга.

### Consequences

* Good, because гейты ретривинга калибруются объективно (по метрикам), а не «по ощущениям модели».
* Good, because добор имеет потолок (бюджет токенов) — снижается риск lost-in-the-middle.
* Good, because лог каждого решения «добирать / не добирать» доступен в Sentry — есть, что калибровать.
* Bad, because при плохих порогах сервер может «зажать» полезные данные (нужен тюнинг).
* Bad, because добор ИНОГДА ВРЕДЕН (вопрос уже закрыт; добор тащит шум) — гейт триггера должен это учитывать.

### Confirmation

* В `05-building-block-view.md` контейнер «MCP-сервер» имеет компоненты «Гейт релевантности», «Триггер добора», «Бюджет окна».
* В Sentry есть метрики `retrieval_below_threshold`, `re_retrieval_triggered`, `context_budget_hit`.
* В runbook описано, как двигать пороги и какой эффект ожидать.

## Pros and Cons of the Options

### (A) На сервере

* Good, because измеримость, калибруемость, наблюдаемость.
* Good, because Self-RAG-совместимо.
* Bad, because агент теряет «волю» решать про добор (но он и не должен).

### (B) На агенте

* Good, because гибко и быстро итерируется через prompt-engineering.
* Bad, because нечего калибровать — у агента нет измеримых критериев.
* Bad, because агент может проигнорить хорошие данные («оно у меня и так есть»).

### (C) Гибрид

* Good, because внешне выглядит «лучшее из двух миров».
* Bad, because управляющие решения всё равно стекают в (B), где их принимают; сервер становится «glorified кеш».
