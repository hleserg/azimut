# Роутинг поиска по коду — fallback-цепочка graph → metadata → grep

## Context and Problem Statement

Поиск по BSL-коду — не единственный инструмент Азимута. Агент может дёрнуть граф вызовов, семантический индекс, или файловый grep. Без явного порядка агент соскакивает с мощных инструментов после первой неудачной формулировки запроса и уходит в grep как в «универсальный ответ», минуя граф. Это дорого в токенах и теряет структурированные связи.

Research HLE-461 разобрал репо `comol/ai_rules_1c` (~227★) — набор правил поведения ИИ-агента-разработчика 1С, обкатанный на боевых задачах. Правила дают готовый ответ на открытый вопрос: в каком порядке дёргать инструменты поиска и когда переходить вниз по каскаду. Синтез HLE-461 = решение; утверждено Сергеем 2026-05-27.

## Decision Drivers

* Граф вызовов — самый информативный источник, но работает только если граф построен; без него нужен graceful fallback.
* Переход вниз по каскаду не должен срабатывать с первой неудачи: агент мог ошибиться в формулировке, а не исчерпать инструмент.
* Файловый grep — последний резерв, не первый; он медленный, шумный, не понимает семантику.
* «Внешние знания» (платформа 1С, БСП, живая ИБ) — другой вектор, не смешивать с поиском по проекту.
* Токены дороги: ответ нужно отдавать на нужном уровне детализации, без избыточного кода.

## Considered Options

* **Каскад graph → metadata → grep** по образцу `comol/ai_rules_1c` с явным триггером перехода.
* **Простой линейный порядок: сначала семантический индекс, потом граф, потом grep** — без чёткого триггера.
* **Граф как единственный инструмент** — отклонено (граф реализован частично, ADR 0025 proposed).

## Decision Outcome

Chosen option: **«Каскад graph → metadata → grep с двойным триггером»**, because это обкатанный внешний ответ (comol/ai\_rules\_1c, боевые задачи 1С) на нашу открытую задачу, совпадающий с нашими ограничениями (граф — самый мощный, grep — самый дорогой).

**Fallback-цепочка (применяется только к поиску по исходникам проекта):**

1. **Граф** (Neo4j / SQLite adjacency) — высший приоритет. Структурированные вызовы, подписки на события, квалифицированный резолв процедур.
2. **Семантический индекс** (`code-metadata`, FTS + dense + sparse, BGE-M3) — когда граф недоступен или вернул пусто/нерелевантно.
3. **Семантический индекс с `grep=true`** — substring-retry **внутри индекса** (не файловый grep). Для точных идентификаторов, фрагментов запроса, путей метаданных, имён обработчиков, текста ошибок.
4. **Файловый grep** (`rg`) — только когда всё MCP исчерпано, с обязательной пометкой «что пробовал и почему не помогло».

**Ключевой триггер перехода вниз:** пусто/non-actionable **дважды на существенно разных запросах** по одной цели. Один провал = переформулировать, а не переходить вниз. Защита от преждевременного отказа от мощного инструмента.

**Принцип остановки:** стоп, как только собранных данных достаточно. Каждый вызов должен закрывать конкретный пробел и не дублировать предыдущий. No-change repeats запрещены.

**Два независимых вектора диспетчера:**

| Вектор | Цепочка | Комментарий |
|---|---|---|
| Поиск по проекту | graph → index → index+grep → file grep | Каскад выше, grep в хвосте |
| Внешние знания | платформа 1С / БСП / живая ИБ | По нужде, без grep-эквивалента |

Сервер считается доступным только если его инструменты **реально экспонированы** в сессии (наличие в конфиге ≠ доступность).

**Уровни детализации ответа (`detail_level` L0–L3):**

| Уровень | Содержимое | Когда |
|---|---|---|
| L0 | Полный код процедуры без усечения | Нужен полный код конкретной процедуры |
| L1 | Сигнатура + описание + вызываемые (default) | Стандартный поиск |
| L2 | Карточка (имя, владелец, модуль, export) | Обзор списка |
| L3 | Только имя и score | Широкий обзор с большим top\_k |

### Consequences

* Good, because порядок инструментов зафиксирован — агент не соскальзывает к grep после первой неудачи.
* Good, because двойной триггер защищает от преждевременного отказа от мощных инструментов.
* Good, because разделение векторов (проект vs внешние знания) исключает смешение источников.
* Good, because L0–L3 управляют токенами: агент берёт ровно столько данных, сколько нужно.
* Bad, because graph-ветвь каскада полноценно работает только когда ADR 0025 (резолв одноимённых) реализован; до тех пор фактически `index → grep`, граф-уровень — задел.
* Bad, because `grep=true` — substring внутри индекса, не в файловой системе; это нужно явно отличать от файлового grep в реализации.

### Confirmation

В `docs/architecture/05-building-block-view.md` §«Диспетчер MCP» описан каскад инструментов с указанием триггера перехода. Eval (RAGAS) показывает Context Precision ≥ 0.80 при стандартных запросах — агент использует граф/индекс, а не только grep.

## Pros and Cons of the Options

### Каскад graph → metadata → grep (выбрано)

* Good, because обкатан на боевых задачах 1С (comol/ai\_rules\_1c).
* Good, because явный триггер предотвращает преждевременный переход к grep.
* Bad, because требует реализации ADR 0025 (граф) для полного потенциала.

### Линейный порядок без чёткого триггера

* Good, because проще в первой реализации.
* Bad, because агент соскальзывает к grep после первой неудачи — теряем граф и индекс.
