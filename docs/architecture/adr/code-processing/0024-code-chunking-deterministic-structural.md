# Детерминированная структурная резка кода поверх Азимута

## Context and Problem Statement

Азимут (форк `bsl-atlas`) реализует базовый структурный чанкинг: одна процедура/функция = один чанк, если тело ≤ 2000 символов (`vector_indexer.py:499–503`); функции < 5 строк пропускаются. Для тел > 2000 символов применяется `RecursiveCharacterTextSplitter` (chunk\_size=1000, overlap=100) — то есть **символьная резка без учёта границ блоков**. Реальная `ОбработкаПроведения` в типовой ERP — сотни строк кода, почти всегда превышает порог → fallback срабатывает почти всегда → чанк может разорваться посреди `Если`, потерять заголовок процедуры. Агент видит осколок без контекста — прямой источник галлюцинаций по Приоритету №1.

Нужно решить: оставить символьный fallback из форка или заменить его на детерминированную структурную резку?

## Decision Drivers

* Приоритет №1 — «почему документ сменил счёт» — требует понимания полных цепочек проведения; обрыв посреди `Если`/`Цикл` делает чанк непригодным.
* LLM для резки недопустима: недетерминированность = источник галлюцинаций (LLM остаётся кандидатом на **обогащение**, не резку).
* Порог 2000 символов ≈ 25–30 строк кода 1С; реальная процедура проведения — 200–500 строк → fallback срабатывает почти на всех значимых процедурах.
* Эмбеддер BGE-M3 считает окно в **токенах**, русские идентификаторы дают иное соотношение символ/токен, чем ASCII → порог должен быть в токенах, не символах.
* Парсер BSL в форке двойной: tree-sitter (грамматика `alkoleft/tree-sitter-bsl`) — первичный, regex-fallback — если `.so` не скомпилирован. В Docker `.so` компилируется → tree-sitter активен по умолчанию.

## Considered Options

* **Оставить символьный fallback** из форка (`RecursiveCharacterTextSplitter`).
* **Детерминированная структурная резка по top-level блокам** (`Если`/`Цикл`/`Попытка`/`Область`), граница определяется tree-sitter AST.
* **LLM для резки** — недетерминированно, отклонено немедленно.

## Decision Outcome

Chosen option: **«Детерминированная структурная резка по top-level блокам»**, because только она сохраняет семантические границы при любом объёме процедуры и остаётся детерминированной (граница = AST-узел, не произвольная позиция в строке).

**Алгоритм резки (наш код поверх форка):**

1. **Функция ≤ порога (~1000–1500 токенов) → один чанк** — поведение форка сохраняется.
2. **Функция > порога → режем по top-level блокам** (`Если`/`Цикл`/`Попытка`/`Область`). Блок никогда не разрывается посередине.
3. **Сам блок > порога → рекурсивно заходим внутрь** по его вложенным блокам. Метки иерархические: блок `4`, внутри — `4.1`, `4.2`. Стоп на глубине 1–2 уровня (глубже — бессмысленные микрочанки).
4. **«Шапка контекста» в каждом подчанке**: `«ОбработкаПроведения, блок 4.2: внутри Если ВалютаУчёта <> ВалютаДокумента»`. Без шапки подчанк — сирота, агент не понимает куда он попал.
5. **Метка подчанка обязательна**: «блок n из N, процедура X, модуль Y».

**Запросы в строках (`Запрос.Текст = "..."`) — отдельный тип контента:**
- tree-sitter видит строку как атом, содержимое языка запросов 1С не разбирает.
- Для длинных запросов (бывают по 1000+ строк, пакеты с временными таблицами) — режем по `|;` (официальный разделитель пакетов). Каждый пакет (`ВЫБРАТЬ`/`ПОМЕСТИТЬ`) = логическая граница.
- Лёгкие метки временных таблиц одной регуляркой (не полным парсером запросов): что пакет создаёт (`ПОМЕСТИТЬ X`) и что читает (имена из `ИЗ`/соединений). Если читаемой таблицы нет ни в одном `ПОМЕСТИТЬ` — флаг `external_temp_table`.

**`МенеджерВременныхТаблиц` — честная слепая зона (runtime, не статика):**
Менеджер течёт через процедуры как параметр/свойство структуры; где он наполнен — определяется трассой выполнения, статически не доказуемо. Берём дёшевое детерминированное покрытие:
- **Уровень 0** — флаг `external_temp_table` на чанке запроса (выше).
- **Уровень 1** — пометить процедуры, которые создают `Новый МенеджерВременныхТаблиц` или принимают его параметром (пул кандидатов «где искать наполнение»).

### Consequences

* Good, because цепочки проведения никогда не рвутся посереди блока → агент получает семантически целые куски.
* Good, because детерминированность: одинаковый код → одинаковые чанки → воспроизводимый eval (RAGAS).
* Good, because `|;`-резка запросов + флаг `external_temp_table` закрывают наиболее болезненный частный случай длинных запросов с временными таблицами.
* Bad, because структурный сплиттер + резка запросов — **наш код** поверх форка (в форке этого нет); нужно реализовать.
* Bad, because порог в токенах требует токенизатора на этапе индексации (один вызов tokenizer.encode на чанк — допустимо, но лишняя зависимость).
* Bad, because `МенеджерВременныхТаблиц` остаётся честной слепой зоной: статический анализ не может восстановить трассу выполнения.

### Confirmation

Чанкер проходит eval (RAGAS Context Recall ≥ 0.75) на корпусе с длинными процедурами проведения. Ни один тестовый чанк не содержит осколка `Если`/`Цикл` без парного закрытия. Флаг `external_temp_table` проставлен на всех запросах, читающих таблицы без `ПОМЕСТИТЬ` в том же пакете.

## Pros and Cons of the Options

### Детерминированная структурная резка (выбрано)

* Good, because граница = AST-узел, воспроизводимо и понятно при отладке.
* Good, because шапка контекста делает каждый подчанк самодостаточным.
* Bad, because требует реализации нашего кода поверх форка.

### Символьный fallback (отклонено)

* Good, because уже работает в форке, ноль новой реализации.
* Bad, because разрывает процедуры посреди `Если` → агент видит осколок → галлюцинации именно на Приоритете №1.
* Bad, because порог 2000 символов ≠ токены BGE-M3 → реальное окно непредсказуемо.
