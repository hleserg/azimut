# Портировать техники `mcp-1c` (feenlace) в Python — не переписывать на Go

## Context and Problem Statement

Research HLE-456 выявил `mcp-1c` (feenlace, Go, MIT) как исключительно быстрый индексатор: ~7 секунд на 13 000+ модулей против ~30 минут у `1c-mcp-metacode`. Достигнуто за счёт конкретных инженерных техник (GC-off, FNV-шардирование, дисковый кеш, манифест-diff). Исходная формулировка реш. 2.4 предлагала «переписать логику bsl-atlas на Go» и называла `mcp-1c` «основным донором» — фактически вводя второй язык в рантайм. Это противоречит фундаменту Python+FastMCP (реш. 1.1, 1.2) и ограничению «мама не ставит лишнее» (`_resolutions.md` #2).

Нужно решить: брать техники feenlace и язык Go вместе, или отделить идеи от реализации?

## Decision Drivers

* Фундамент Python+FastMCP (реш. 1.1, 1.2) — второй язык в рантайме противоречит принципу «маме не ставить лишнее».
* Техники скорости из feenlace реальны и ценны (×25–250 к metacode) — терять их нельзя.
* Лицензия feenlace — MIT (Copyright (c) 2025); код можно читать и использовать как образец.
* `mcp-1c` не индексирует тексты BSL-кода — чанкинг на уровне модуля (грубо), графа нет. Код не переносится, только идеи.
* BSL-синонимный анализатор из feenlace (рус↔англ) — критичен для поиска: разработчики мешают термины.

## Considered Options

* **Портировать техники в Python** — брать идеи из Go-реализации, реализовывать на нашем Python-стеке.
* **Форкнуть `mcp-1c` и дописать BSL-индексацию на Go** — два языка в рантайме.
* **Игнорировать feenlace** — не использовать его техники вообще.

## Decision Outcome

Chosen option: **«Портировать техники в Python»**, because это отделяет ценные инженерные идеи от языка реализации. Python-аналоги всех ключевых техник существуют и достигают сопоставимого ускорения. Второй язык в рантайме — не вводим.

**Список техник для портирования (из `_resolutions.md` #2 и HLE-456):**

| Техника feenlace (Go) | Python-аналог |
|---|---|
| GC-off (`debug.SetGCPercent(-1)`) на этап массовой индексации | `gc.disable()` / `gc.set_threshold(0, 0, 0)` на этапе batch-индексации, `gc.enable()` после |
| FNV-шардирование по CPU (`numCPU` воркеров, каждый свой шард) | `multiprocessing.Pool` / `concurrent.futures.ProcessPoolExecutor`, шардирование по `cpu_count()` |
| `batch` + `unsafe_batch` + неблокирующий старт | `asyncio` / `concurrent.futures`, batch-submit без ожидания каждого результата |
| Дисковый кеш по SHA-256 пути (`~/.cache/feenlace/...`) | `hashlib.sha256(path)`, кеш в `~/.cache/azimut/` или рядом с индексом |
| Манифест-diff (сравниваем манифест текущих файлов с предыдущим) | `{path: mtime+size}` JSON-манифест; при переиндексации обрабатываем только изменённые файлы |
| BSL-синонимный анализатор (рус↔англ, ~40 ключевых слов + ~180 функций) | Python-модуль с таблицей `СтрНайти ↔ StrFind`, `Если ↔ If`, …; расширяемый словарь |

**Инкрементальность:** старт на file-level манифесте (проще, из feenlace) → function-hash (bsl-atlas) как апгрейд позже.

**Что НЕ берём:**
- Код на Go — несовместимый стек, второй язык в рантайме.
- Чанкинг feenlace (уровень модуля, без AST) — у нас структурная резка ADR 0024.
- Bleve (Go full-text search) — у нас Qdrant + BGE-M3.

### Consequences

* Good, because техники скорости сохраняются без введения Go в рантайм.
* Good, because BSL-синонимный анализатор (рус↔англ) критичен для точного поиска — разработчики мешают термины.
* Good, because манифест-diff делает переиндексацию инкрементальной с самого старта.
* Good, because `_resolutions.md` #2 зафиксировал правку: противоречие «Go vs Python» устранено.
* Bad, because портирование требует самостоятельной реализации каждой техники — нельзя просто скопировать Go-код.
* Bad, because GC-off-аналог в Python (`gc.disable()`) даёт меньший эффект, чем в Go (Python GIL, другая природа сборщика) — оценить выигрыш только на бенчмарке.
* Bad, because BSL-синонимный словарь (~40 ключевых слов + ~180 функций) нужно портировать вручную из Go-источника.

### Confirmation

Индексатор на дымовом прогоне (реальная ERP) укладывается в целевой порог производительности (< N сек на 13 000 модулей — конкретный порог калибруется на прогоне). BSL-синонимный анализатор покрывает базовые ключевые слова (проверяется юнит-тестами с русскоязычными и английскими запросами).

## Pros and Cons of the Options

### Портировать техники в Python (выбрано)

* Good, because один язык в рантайме — проще деплой, проще поддержка.
* Good, because Python-аналоги всех техник реализуемы.
* Bad, because нельзя просто скопировать код — нужна реализация каждой техники.

### Форкнуть `mcp-1c` на Go

* Good, because техники работают из коробки.
* Bad, because два языка в рантайме: Go + Python → «мама не ставит лишнее» нарушено.
* Bad, because feenlace не индексирует тексты BSL — код пришлось бы полностью переписывать.

### Игнорировать feenlace

* Good, because нет дополнительной работы.
* Bad, because теряем ×25–250 к скорости индексации и BSL-синонимный анализатор.
