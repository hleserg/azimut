# XML-выгрузка конфигурации как второй источник данных

> **Статус: proposed.** Решение не принято — нужен прототип и оценка трудоёмкости парсера XML. Рассматривается при работе над темой 2 (HLE-414) и темой 4 (HLE-416).

## Context and Problem Statement

ADR 0011 и ADR 0024 опираются на `DumpConfigToFiles` — текстовую выгрузку конфигурации 1С: модули BSL (`*.bsl`) + plain-text метаданные. Это единственный входной формат, который понимает `bsl-atlas`. Однако система 1С:ERP имеет второй формат выгрузки — **XML-выгрузка из Конфигуратора** (файлы `.xml`): структура объектов конфигурации (формы, регистры, реквизиты, роли, метаданные в полном объёме). Эти два формата **не пересекаются**: текстовая выгрузка даёт код и вызовы, XML-выгрузка даёт объектную модель без кода. Нужно решить: является ли XML-выгрузка необходимым вторым источником, и если да — как её парсить и индексировать?

## Decision Drivers

* Для полного понимания структуры ERP (ответ на вопросы о реквизитах, формах, регистрах) — BSL-модулей недостаточно; нужна объектная модель конфигурации.
* v1.x-требование HLE-297 («парсер XML-выгрузки конфигурации: структура + связи + встроенная справка») явно называет этот источник.
* `bsl-atlas` (ADR 0011) работает только с текстовым форматом — XML-парсер придётся писать самостоятельно или адаптировать существующие решения (⚠️ предположение).
* Трудоёмкость XML-парсера неизвестна до прототипа; риск недооценки высок (XML-схема 1С:ERP большая и плохо задокументирована).

## Considered Options

* **Вариант A — реализовать XML-парсер в v1**: написать парсер структуры конфигурации из XML-выгрузки, индексировать объектную модель в Qdrant рядом с BSL-чанками.
* **Вариант B — отложить до v2 / темы 4**: зафиксировать как known gap, в v1 работать только с DumpConfigToFiles; XML-выгрузку открыть при работе над темой 4 (HLE-416).
* **Вариант C — адаптировать `FSerg/mcp-1c-v1`**: этот MCP-сервер (TypeScript, MIT) индексирует метаданные конфигурации через собственный парсер — возможно, покрывает часть потребности без написания парсера с нуля (⚠️ предположение: требует проверки, что именно он парсит).

## Decision Outcome

**Решение не принято.** Требуется:

1. Исследовать, какую часть XML-схемы конфигурации покрывает `FSerg/mcp-1c-v1` (вариант C как быстрый путь).
2. Прototипировать парсер XML на одном реальном объекте (например, документ «РеализацияТоваровУслуг»): оценить трудоёмкость и полноту.
3. Принять решение A или B при открытии темы 2 (HLE-414) или темы 4 (HLE-416).

### Confirmation

Решение считается принятым, когда выбран вариант (A, B или C) и зафиксирован в этом ADR со статусом `accepted`. Прототип XML-парсера (если вариант A/C) проходит на реальной выгрузке ERP: парсит ≥ 3 типа объектов (документ, регистр, справочник), данные попадают в Qdrant в отдельную коллекцию или отдельный payload-тип.

## Pros and Cons of the Options

### Вариант A — XML-парсер в v1

* Good, because полная объектная модель ERP доступна агенту → лучшие ответы о структуре конфигурации.
* Bad, because XML-схема 1С:ERP большая и слабо задокументирована — высокий риск недооценки трудоёмкости.
* Bad, because два источника (BSL + XML) усложняют схему Qdrant и логику сборки контекста.

### Вариант B — отложить до v2 / темы 4

* Good, because фокус v1 сохраняется на коде BSL и Приоритете №1 — не распыляемся.
* Good, because можно валидировать ценность v1 без XML, прежде чем инвестировать в парсер.
* Bad, because вопросы о реквизитах, формах, регистрах остаются без ответа в v1.

### Вариант C — адаптировать FSerg/mcp-1c-v1

* Good, because готовый парсер метаданных (MIT, не нужно писать с нуля).
* Bad, because TypeScript-стек, в Python-проект не переносится код напрямую; нужно либо порт, либо sidecar.
* Bad, because неизвестно, что именно парсит — требует аудита (⚠️ предположение).
