# Push к пользователю через веб-морду — замена отсутствующего push в MCP

## Context and Problem Statement

MCP-протокол не поддерживает server-initiated push: сервер отвечает на запросы клиента, но не может сам инициировать сообщение. На локальном деплое это нормально — Сергей и мама видят результат в момент запроса. Но в VDS-сценарии с мульти-арендой возникает задача уведомления: контора загрузила новый дамп конфигурации → переиндексация заняла несколько минут → нужно дать сигнал, что данные обновились и ИИ-ассистент снова актуален. Через MCP это сделать нельзя. Веб-морда (если она будет в теме 7) может стать push-каналом: уведомление в браузере или email о завершении переиндексации.

## Decision Drivers

* MCP фундаментально request-response; push в его нынешней спецификации отсутствует.
* Контора, загрузившая новые данные, хочет знать когда они готовы — без этого сервис «непрозрачен».
* Веб-морда (тема 7) планируется как управляющий UI: загрузка данных, настройки тенанта, мониторинг.
* Реализация push через SSE/WebSocket/polling — стандартная для веб-фронтенда.
* Локальный деплой (Сергей/мама): push-уведомление не нужно — переиндексация локальная, видно сразу.

## Considered Options

* **Push через веб-морду (SSE или polling)** — статус переиндексации транслируется в браузер через Server-Sent Events или polling.
* **Email-уведомление** — письмо после завершения фоновой задачи.
* **Webhook на URL конторы** — активный push на эндпоинт тенанта.
* **Не уведомлять — клиент сам перепрашивает** — тенант повторяет запрос и видит актуальный результат.

## Decision Outcome

Chosen option: **«Push через веб-морду (SSE)»**, because веб-морда в теме 7 всё равно нужна как управляющий UI, и добавление SSE-стриминга статуса задачи — минимальное расширение. Email и webhook — полезные дополнения, но требуют внешней инфраструктуры (SMTP, зарегистрированных URL). «Не уведомлять» нарушает принцип прозрачности.

> **Статус ADR: proposed.** Конкретная технология (SSE vs WebSocket vs polling) и дизайн веб-морды определяются в теме 7 (HLE-419). Этот ADR фиксирует принцип: push-канал = веб-морда.

### Consequences

* Good, because тенант видит статус переиндексации в реальном времени в браузере.
* Good, because SSE — simple и stateless со стороны сервера; не требует WebSocket-сессии.
* Good, because веб-морда как push-канал — единая точка взаимодействия, не отдельный нотификационный сервис.
* Bad, because требует веб-морды — без неё push-уведомления недоступны (для MVP темы 7 это зависимость).
* Bad, because на локальном деплое (без веб-морды) этот механизм не работает — там он и не нужен.
* Bad, because admin-видимость «всех тенантов через Sentry» ≠ уведомление самой конторы; два разных канала.

### Confirmation

ADR переходит в `accepted` когда в теме 7 (HLE-419): веб-морда реализована, SSE (или polling) транслирует статус задачи переиндексации с `tenant_id`, интеграционный тест подтверждает что тенант видит статус «готово» в браузере.

## Pros and Cons of the Options

### Push через веб-морду (SSE)

* Good, because инфраструктура уже нужна для темы 7; нет отдельного нотификационного сервиса.
* Good, because SSE прост в реализации на FastAPI (`StreamingResponse` или `sse-starlette`).
* Bad, because без открытой вкладки браузера уведомление теряется (SSE — pull со стороны клиента).

### Email-уведомление

* Good, because доставляет уведомление даже без открытого браузера.
* Bad, because требует SMTP-инфраструктуры + настройки адресов на тенанта.
* Bad, because latency email-доставки может быть неудовлетворительной.

### Webhook на URL конторы

* Good, because активный push, не зависит от открытого UI.
* Bad, because требует регистрации URL на стороне конторы; дополнительный attack surface.

### Не уведомлять (повторный запрос)

* Good, because нулевая инфраструктура.
* Bad, because тенант не знает когда данные готовы; нарушает UX и принцип прозрачности.
