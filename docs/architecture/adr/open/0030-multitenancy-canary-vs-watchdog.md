# Канарейка-в-потоке vs фоновый сторож для VDS — как разбудить «протухшее в покое»

## Context and Problem Statement

Для локального деплоя канарейка работает «в потоке»: при каждом запросе пользователя проверяется актуальность индекса, состояние ключей, здоровье Qdrant — всё в контексте живого запроса. Это достаточно, потому что пользователь (Сергей или мама) регулярно обращается к системе. На VDS с мульти-арендой ситуация другая: одна из контор может «протухнуть в покое» — её индекс устарел, ключ истёк, но пользователь этой конторы давно не заходил. Когда он наконец зайдёт — получит деградированный сервис без предупреждения. Вторая контора при этом работает нормально. Нужно решить: достаточно ли канарейки-в-потоке для VDS или нужен фоновый сторож?

## Decision Drivers

* «Канарейка сигналит до раздражения пользователя» — принцип из §8 crosscutting; на VDS это означает сигнализировать до следующего входа незаходящей конторы.
* Мониторинг должен различать тенантов: деградация одной конторы не должна тонуть в усреднённых метриках по всем.
* Фоновый демон добавляет инфраструктурную сложность (cron/celery/asyncio task); для локального деплоя избыточен.
* Ресурсы VDS ограничены: фоновый сторож должен быть lightweight — не пересчитывать весь индекс, только проверять health.

## Considered Options

* **Канарейка только в потоке** — проверка актуальности при каждом запросе, никакого фона.
* **Фоновый сторож (watchdog)** — периодическая задача (async task / celery beat) проверяет health по всем тенантам.
* **Гибрид: в потоке + фоновый по расписанию** — основная проверка в потоке, лёгкий health-пинг раз в N минут фоном.
* **Push-уведомление от конторы** — тенант сам триггерит переиндексацию (исключает passive stale).

## Decision Outcome

Chosen option: **«Гибрид: в потоке + лёгкий фоновый сторож»** — проверка в потоке остаётся (отвечает за latency и качество в реальном времени), фоновый сторож раз в N минут делает только health-пинг (живой ли индекс, истёк ли ключ, основные метрики) по каждому тенанту без пересчёта. При обнаружении проблемы — алерт в систему мониторинга с `tenant_id`.

> **Статус ADR: proposed.** Конкретный интервал пинга и механизм (asyncio task vs celery beat) — открытый вопрос темы 7.

### Consequences

* Good, because «протухшее в покое» обнаруживается до следующего входа пользователя.
* Good, because алерты размечаются по `tenant_id` — видно, у какой конторы проблема.
* Good, because health-пинг lightweight — не пересчитывает индекс, только проверяет состояние.
* Bad, because добавляет фоновую задачу к инфраструктуре VDS.
* Bad, because выбор механизма (asyncio vs celery vs apscheduler) — дополнительный architectural drift.
* Bad, because для локального деплоя (Сергей/мама) фоновый сторож избыточен — нужен конфигурационный выключатель.

### Confirmation

ADR переходит в `accepted` когда в теме 7 (HLE-419): выбран механизм фоновой задачи, реализован health-пинг с `tenant_id`, интеграционный тест подтверждает детекцию «протухшего тенанта» до следующего входа, локальный деплой работает без фонового сторожа.

## Pros and Cons of the Options

### Канарейка только в потоке

* Good, because нет лишних сервисов, простая модель.
* Bad, because «протухшее в покое» не обнаруживается пока тенант не зайдёт — нарушает принцип «до раздражения».
* Bad, because на VDS с несколькими конторами одна деградировавшая тонет в общих метриках.

### Фоновый сторож (watchdog) — полный

* Good, because активный мониторинг всех тенантов в любой момент.
* Bad, because сложнее в реализации; риск resource contention если проверяет индексы тяжело.
* Bad, because избыточен для локального деплоя.

### Гибрид: в потоке + лёгкий фоновый пинг

* Good, because лёгкий пинг не нагружает систему; основная логика остаётся in-request.
* Good, because выключается конфигом для локального деплоя.
* Bad, because интервал пинга нужно калибровать под ожидаемую частоту входов тенантов.

### Push-уведомление от конторы

* Good, because нет фоновых задач на сервере.
* Bad, because требует интеграции с веб-мордой или нотификационным каналом; passive stale всё равно остаётся риском.
