# Механика детектирования противоречивости — как технически детектировать, порог, поведение при множестве конфликтов

## Context and Problem Statement

ADR 0001 (Р1) зафиксировал принцип: перед выдачей чанков агенту-генератору проверяем их на взаимную противоречивость; при обнаружении — инкрементируем метрику. Принцип принят, но открытыми остались три технических вопроса из `decisions.md` Р1: (1) **как именно детектировать противоречие** — правила по метаданным, косинусное расстояние эмбеддингов, или LLM-судья? (2) **какой порог** считать «конфликтом», а не просто «дополняют друг друга»? (3) **что делать при множестве конфликтов** — блокировать весь ответ, агрегировать, показывать все варианты? Этот ADR документирует открытые механики и кандидатов; финальный выбор — в теме 5 (HLE-417).

## Decision Drivers

* Детектирование должно работать на уровне ретривера (сервер), не агента-генератора (ADR 0005, Р5).
* Ложные срабатывания хуже пропущенных: лучше пропустить слабое противоречие, чем заблокировать релевантный ответ.
* Производительность: детектирование in-request (каждый запрос), не в фоне — latency критична.
* «Дополняют друг друга» ≠ «противоречат» — чанки из разных версий 1С могут говорить разное не конфликтуя.
* Связь с LLM-судьёй (ADR 0003, Р3): возможна синергия — судья уже смотрит на groundedness; contradiction-детектор либо отдельный, либо специализированная часть судьи.

## Considered Options

### По методу детектирования

* **Правила по метаданным** — конфликт если два чанка из одного модуля/функции с разным содержимым (hash mismatch).
* **Косинусное расстояние** — чанки с высокой семантической близостью, но низким overlap — потенциальный конфликт.
* **LLM-судья** — передаём пару чанков судье (ADR 0003) с запросом «противоречат ли?».
* **Гибрид: правила-first, LLM-fallback** — быстрая проверка по метаданным; при неопределённости — LLM.

### По порогу

* **Жёсткий порог** — фиксированное значение метрики (например, если ≥ 2 чанков конфликтуют → флаг).
* **Адаптивный порог** — порог от контекста: для кода строже, для справки мягче.
* **Без порога, только флаг** — фиксировать факт конфликта, решение об ответе — на агенте.

### По поведению при множестве конфликтов

* **Блокировка ответа** — при ≥ N конфликтов ответ не выдаётся, агенту возврат ошибки.
* **Выдать с предупреждением** — все чанки в ответе, но с аннотацией `contradiction_detected: true`.
* **Иерархия источников (ADR 0006, Р6)** — применить иерархию и выдать победителя + объяснение.
* **Агрегация** — показать все конфликтующие варианты, упорядоченные по иерархии источников.

## Decision Outcome

**Решение не принято; ADR фиксирует открытый вопрос для темы 5 (HLE-417).**

Наиболее вероятный кандидат по итогам анализа: **«Гибрид: правила-first + LLM-fallback» с адаптивным порогом + поведение через иерархию источников**:
- Первый проход: правила по метаданным (модуль + версия + hash) — быстро, zero-latency.
- При неопределённости: LLM-судья (уже в стеке по ADR 0003) с prompt «эти два чанка противоречат или дополняют?».
- Порог: адаптивный — для кода (одна функция, две версии) строже, для справочного текста мягче.
- При конфликте: применить иерархию Р6, выдать победителя с аннотацией; при неразрешимом конфликте — агрегация.

> **Статус ADR: proposed.** Финальный выбор требует эксперимента на реальном корпусе; уточняется в теме 5.

### Consequences

* Good, because правила-first дают нулевую latency в 80%+ случаев.
* Good, because LLM-fallback через уже имеющегося судью (ADR 0003) — нет новых зависимостей.
* Bad, because адаптивный порог нужно откалибровать на золотом наборе (зависимость от темы 6).
* Bad, because LLM-вызов в fallback-ветке добавляет latency — нужен timeout/fallback «не блокировать».
* Bad, because интеграция с иерархией Р6 требует явного mapping «тип чанка → уровень иерархии».

### Confirmation

ADR переходит в `accepted` когда в теме 5 (HLE-417): выбран финальный метод детектирования, откалиброван порог на eval-наборе, определено поведение при N конфликтах, реализован и протестирован в интеграционном тесте.

## Pros and Cons of the Options

### Правила по метаданным

* Good, because детерминированны, быстры, отлаживаемы.
* Bad, because покрывают только структурные конфликты (одна функция, разный hash); семантические конфликты пропускают.

### Косинусное расстояние

* Good, because ловит семантически близкие чанки с расхождением.
* Bad, because высокое косинусное расстояние ≠ противоречие (чанки могут просто быть о разном).
* Bad, because создаёт много ложных срабатываний — нужен тщательный порог.

### LLM-судья (только)

* Good, because высокая точность при правильном prompt.
* Bad, because latency на каждый запрос; N чанков = N*(N-1)/2 пар для попарной проверки.

### Гибрид: правила-first, LLM-fallback

* Good, because правила закрывают 80%+ случаев мгновенно; LLM только для неоднозначных.
* Good, because переиспользует уже имеющегося судью (ADR 0003).
* Bad, because логика ветвления сложнее; тест-покрытие обоих путей обязательно.
