# Мульти-аренда: Qdrant embedded vs server — развилка по режиму деплоя

## Context and Problem Statement

ADR 0015 зафиксировал: на локальном деплое (Сергей, мама) используем Qdrant embedded — чтобы не поднимать вторую сущность. Это решение принималось для однопользовательского сценария. При переходе к VDS с несколькими конторами (мульти-аренда) это ограничение отпадает: там скорее нужен серверный Qdrant с разделением индексов по тенантам (через коллекции или tenant-фильтр `group_id`). Нужно решить: embedded остаётся только для локального режима, а для онлайн/VDS всегда сервер? Или допустимы конфигурации «сервер локально»?

## Decision Drivers

* Локальный деплой: минимальные зависимости — embedded Qdrant не требует отдельного Docker-контейнера.
* VDS/мульти-аренда: несколько тенантов = несколько коллекций или один индекс с tenant-фильтром; embedded для этого не оптимален.
* Corpus > ~500 k чанков → embedded упирается в RAM; квантизация + серверный режим дают преимущество.
* Qdrant 1.16+ Tiered Multitenancy — выделенный шард для крупных тенантов; работает только в серверном режиме.
* Конфигурационный переключатель лучше хардкода — одна кодовая база, два режима.

## Considered Options

* **Embedded только для локального, server для VDS** — режим выбирается через конфиг (`QDRANT_MODE=embedded|server`).
* **Всегда embedded** — единый режим, упрощённый деплой; теряем серверные фичи мульти-аренды.
* **Всегда server** — единый режим, нужен Docker для мамы тоже.
* **Embedded с эмуляцией multi-tenant через namespace-метки** — пограничный вариант.

## Decision Outcome

Chosen option: **«Embedded только для локального, server для VDS» (через конфиг)**, because это минимально инвазивное изменение ADR 0015: принцип «маме не ставить вторую сущность» сохраняется, VDS-режим разблокируется через переключатель. Финальная реализация и выбор механизма изоляции (отдельные коллекции vs единый индекс с `group_id`) решается в теме 7 (HLE-419).

> **Статус ADR: proposed.** Решение зафиксировано как принцип; конкретный механизм multi-tenant в Qdrant (коллекции vs `group_id` с ACORN-фильтрацией) — открытый подвопрос темы 7.

### Consequences

* Good, because локальный сценарий не усложняется — embedded остаётся по умолчанию.
* Good, because VDS-режим разблокирован без переписывания кодовой базы.
* Good, because Qdrant 1.16 Tiered Multitenancy доступен только в серверном режиме — тенанты с большим корпусом получают выделенный шард.
* Bad, because конфигурационный переключатель нужно сопровождать и тестировать в обоих режимах.
* Bad, because выбор между «отдельная коллекция на тенанта» и «единый индекс с `group_id`» остаётся открытым (лимит ~1000 коллекций в Qdrant — аргумент против отдельных коллекций).

### Confirmation

ADR переходит в `accepted` когда в теме 7 (HLE-419): выбран механизм изоляции тенантов в Qdrant, реализован конфигурационный переключатель, локальный интеграционный тест и VDS-интеграционный тест зелёные.

## Pros and Cons of the Options

### Embedded локально / server для VDS (конфиг)

* Good, because минимально нарушает текущий ADR 0015 и локальный сценарий.
* Good, because серверный режим позволяет Tiered Multitenancy и ACORN-фильтрацию.
* Bad, because два пути деплоя = два набора интеграционных тестов.

### Всегда embedded

* Good, because единый деплой, нет Docker-зависимости на любой машине.
* Bad, because embedded не поддерживает Tiered Multitenancy; при ~500 k+ чанков — RAM-ограничение.
* Bad, because требует non-trivial изоляции (namespace-метки не дают гарантии).

### Всегда server

* Good, because единый режим, полные серверные возможности.
* Bad, because для мамы — дополнительный Docker-контейнер; нарушает принцип минимального деплоя.

### Embedded с namespace-метками

* Good, because нет отдельного сервиса.
* Bad, because нет настоящей изоляции; один тенант может «увидеть» чанки другого при ошибке фильтра.
* Bad, because не поддерживается Qdrant официально как multi-tenant паттерн.
