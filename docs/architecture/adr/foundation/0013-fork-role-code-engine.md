# Роль `bsl-atlas`: только «движок понимания кода» (берём парсер BSL + граф вызовов + каркас MCP + docker; меняем хранилище/эмбеддер/реранк; дописываем поведенческий контракт и оркестрацию)

## Context and Problem Statement

Решение форкнуть `bsl-atlas` (ADR 0011) само по себе не отвечает на вопрос: что именно берём, что меняем, что пишем с нуля? Без явной границы есть риск либо переоценить форк («он делает почти всё»), либо недооценить («почти всё перепишем»). Нужно зафиксировать роль форка и границу «готовое vs наш код», чтобы не дублировать работу и не пропустить конкурентное ядро.

## Decision Drivers

* Форк ценен ровно одним: парсер BSL + детерминированный граф вызовов — это не хочется писать с нуля.
* Хранилище, эмбеддер и реранкер форка не подходят для нашего целевого стека — их нужно менять.
* Конкурентное ядро (поведение агента, гарантии ответов, оркестрация) — нет в опенсорсе, пишем сами.
* Граница должна быть явной, чтобы команда понимала где «трогаем форк» и где «наш новый код».

## Considered Options

* **Минимальный форк** — берём только BSL-парсер, всё остальное пишем своё.
* **Форк как «движок»** — берём парсер + граф + каркас MCP + docker; меняем хранилище/эмбеддер; дописываем оркестрацию.
* **Форк как «база»** — берём по максимуму, включая ChromaDB и текущий эмбеддер, минимизируем изменения.

## Decision Outcome

Chosen option: **«Форк как движок»**, because ценность форка — ровно один дорогой и рискованный кусок (парсер BSL + граф), а не весь стек. Остальное либо не дотягивает до нашего целевого уровня (хранилище, эмбеддер), либо отсутствует (реранк, анти-галлюцинационная логика). Относимся к форку как к двигателю, на который ставим свой кузов.

**Три категории компонентов:**

**Оставляем (ради этого и форк):**
- Парсер текста BSL: **tree-sitter** (`alkoleft/tree-sitter-bsl`, MIT, компилируется в `.so` при Docker-сборке) — первичный; regex-fallback — если `.so` отсутствует. BSL Language Server (Java, BSL LS) рассматривался, **отложен до v2**: сложность Java-интеграции и отсутствие Python-биндингов.
- Детерминированный граф вызовов (SQLite: таблица `calls(caller_id, callee_name)`).
- Каркас MCP-сервера (точка подключения к Claude / Cherry Studio).
- Docker-упаковку.
- Поддержку русских идентификаторов и локального инференса через Ollama.

**Меняем/выкидываем:**
- Хранилище ChromaDB → Qdrant (ADR 0015).
- Эмбеддер → BGE-M3.
- Чанкер: структурный fallback через `RecursiveCharacterTextSplitter` заменяем на детерминированную резку по блокам tree-sitter (ADR 0024).
- Добавляем реранкер (`rerankers`/BGE/Cohere) — в форке его нет вообще.

**Дописываем своим кодом (конкурентное ядро):**
- Поведенческий контракт агента (портативный системный промпт, ADR 0009).
- Серверный контроль добора (Р5): планка релевантности, триггер добора, потолок окна.
- Сборка полных разделов + «сигнал полноты».
- Иерархия источников при конфликте (Р6) + метрика противоречивости (Р1).
- Адаптер к разговорной модели (ADR 0020).
- LLM-судья / eval-харнесс (ADR 0003).
- Оркестрация: MCP-сервер связывает граф + поиск + реранк + контракт + судью.

### Consequences

* Good, because объём «нашего» кода смещается с парсинга (берём готовое) на оркестрацию и поведение — именно туда, где конкурентное ядро.
* Good, because чёткая граница предотвращает случайное «переизобретение» парсера или случайное «наследование» неподходящего хранилища.
* Bad, because изменения хранилища/эмбеддера означают, что форк нельзя использовать «из коробки» — нужна фаза миграции (ADR 0015).
* Bad, because граф вызовов в форке — только 1 уровень (прямые вызовы), рекурсивный обход достраиваем сами.
* Neutral, because роль форка ограничена **статическим анализом кода**: bsl-atlas читает выгрузку `DumpConfigToFiles`, не имеет доступа к живой 1С. **Runtime-слой** (1c_mcp-style: живые запросы к базе, процедуры в реальном времени, gitsync) — вне роли форка; это отдельный архитектурный слой (зона 2/3 по ADR 0022, исследование HLE-464).

## Pros and Cons of the Options

### Минимальный форк (только парсер)

* Good, because минимальные зависимости от форка, максимальная свобода.
* Bad, because выбрасываем граф вызовов и каркас MCP, которые уже работают — лишняя работа.

### Форк как движок (выбрано)

* Good, because берём максимум ценного из форка, не берём то, что не подходит.
* Good, because граница явная — команде понятно что форк, а что наш код.
* Bad, because небольшая сложность поддержки форка при апстрим-обновлениях bsl-atlas.

### Форк как база

* Good, because минимальные изменения на старте.
* Bad, because ChromaDB не держит hybrid dense+sparse — упрёмся в потолок сразу при теме 3.
* Bad, because «временное» хранилище присосётся, а переделывать потом дороже.
