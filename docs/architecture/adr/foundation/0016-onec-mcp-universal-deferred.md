# MCP-шлюз `onec-mcp-universal` — отложен до темы 7 (на локальном сценарии не нужен; Claude Desktop тянет несколько MCP-серверов напрямую)

## Context and Problem Statement

`AlekseiSeleznev/onec-mcp-universal` — MCP-гейтвей-агрегатор (~400 строк Python/Starlette): один MCP-вход маршрутизирует запросы на несколько backend-1С-MCP-серверов. Обсуждался как шлюз перед Азимутом — «сократить наш MCP-сервер». Нужно решить: включать в фундамент или нет?

## Decision Drivers

* На локальном сценарии Claude Desktop сам поддерживает несколько MCP-серверов через конфиг — отдельный агрегатор не нужен.
* Конкурентное ядро Азимута — оркестрация поиска и поведение агента (Р5, Р6, сигнал полноты). Шлюз эту работу не делает.
* Преждевременная зависимость усложняет фундамент без реальной пользы на текущем сценарии.
* Шлюз осмыслен при сетевом деплое с авторизацией по тенанту — это тема 7.

## Considered Options

* **Отложить до темы 7 (HLE-419)** — в фундамент не берём, вернёмся при мульти-аренде.
* **Включить в фундамент как шлюз** — Азимут + справочник платформы подключаем через `onec-mcp-universal`.
* **Написать свой минималистичный шлюз** — контроль над маршрутизацией.

## Decision Outcome

Chosen option: **«Отложить до темы 7»**, because на локальном сценарии шлюз не нужен: Claude Desktop поддерживает несколько MCP-серверов нативно (перечисление в конфиге). Экономия мала: сам гейтвей крошечный, а «свой MCP-сервер» дорог не транспортом, а оркестрацией (Р5, Р6) — её шлюз не делает. Включение сейчас — лишняя зависимость без профита.

**Когда вернёмся (тема 7):**
- Backend-серверов реально много И/ИЛИ на VDS нужен единый вход с авторизацией по конторе (`tenant_id` в JWT).
- Тогда решим: взять `onec-mcp-universal` за основу или написать тонкий свой.
- Стыкуется с ADR 0029 (Qdrant embedded vs server) и ADR 0032 (изоляция по тенантам).

### Consequences

* Good, because фундамент проще — одним компонентом меньше.
* Good, because решение не теряется: явно зафиксировано как кандидат темы 7.
* Bad, because при расширении до VDS/мульти-аренды потребуется дополнительное архитектурное решение — но его контекст будет богаче именно в теме 7.

### Confirmation

В `workspace.dsl` нет элемента `onec-mcp-universal` — он не в архитектурной модели до темы 7. При открытии HLE-419: ADR 0016 пересматривается или superseded.

## Pros and Cons of the Options

### Отложить до темы 7 (выбрано)

* Good, because не усложняем фундамент преждевременно.
* Good, because Claude Desktop тянет несколько MCP-серверов нативно.
* Bad, because если потребуется раньше темы 7 — придётся возвращаться.

### Включить в фундамент

* Good, because единая точка подключения клиента.
* Bad, because шлюз не решает задачу оркестрации — только транспорт.
* Bad, because дополнительная точка отказа без реальной пользы локально.

### Написать свой минималистичный шлюз

* Good, because полный контроль над маршрутизацией.
* Bad, because любой шлюз — это задача по теме 7, не по фундаменту.
* Bad, because дублируем уже существующий `onec-mcp-universal`.
