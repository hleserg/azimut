# Граница «форк/готовые библиотеки vs наш код» — форк даёт понимание кода, библиотеки дают механику RAG, наш код — поведение, гарантии, оркестрацию

## Context and Problem Statement

Азимут строится поверх форка bsl-atlas и набора готовых библиотек. Без явной границы
каждое новое решение («это мы пишем или берём готовое?») становится спором.
Нужна чёткая карта: что входит в форк, что — в готовые библиотеки, что — в наш код.
Граница также определяет, где живут поведенческие гарантии проекта (Р5, Р6) и кто
отвечает за их корректность при обновлении зависимостей.

## Decision Drivers

* Форк bsl-atlas — основной движок понимания кода (ADR [0011](0011-fork-bsl-atlas-as-core.md), [0013](0013-fork-role-code-engine.md)); нельзя класть в него поведение Азимута
* Готовые библиотеки (BGE-M3, Qdrant, реранкеры) — проверенные компоненты RAG; не переписываем
* Поведенческий контракт (Р5 — контроль поиска, Р6 — иерархия источников) должен жить в нашем коде — только так его можно тестировать и гарантировать
* Адаптер к разговорной LLM (ADR [0020](0020-cloud-llm-via-adapter.md)) — наш код, не библиотека
* При обновлении форка или библиотеки наш код не должен ломаться

## Considered Options

* Всё в форке — форк расширяется под задачи Азимута
* Тонкая обёртка поверх готовых компонентов без явной границы
* Явная трёхзонная граница: форк / библиотеки / наш код

## Decision Outcome

Chosen option: "Явная трёхзонная граница", because это единственный вариант, при котором поведенческие гарантии поддаются изолированному тестированию, а обновление форка или библиотеки не затрагивает бизнес-логику Азимута.

**Три зоны (реш. 1.9):**

### Зона 1 — Форк (bsl-atlas, upstream: AGPL-3.0)

Что входит: BSL-парсер, построитель графа вызовов, MCP-скелет (transport/protocol), Docker-образ базы.

Правило: в форке живёт только то, что относится к пониманию BSL-кода как такового. Поведение Азимута (как отвечать, что проверять, какой источник приоритетнее) — не в форке.

### Зона 2 — Готовые библиотеки (зависимости, не форки)

Что входит: BGE-M3 (эмбеддинги), Qdrant (векторное хранилище), реранкеры (cross-encoder/Cohere), [alkoleft/mcp-bsl-platform-context](../foundation/0017-mcp-bsl-platform-context-included.md) (справочник платформы), Docling / аналог (парсинг документов).

Правило: подключаем как pip/docker-зависимость. Не форкаем без причины. При критическом баге — патч поверх, не форк.

### Зона 3 — Наш код (поведение, гарантии, оркестрация)

Что входит:
- Поведенческий контракт (правила ответа, тональность, ограничения)
- Р5 — контроль поиска (server-controlled retrieval: сервер решает, что искать)
- Р6 — иерархия источников (код > документация > LLM-знания)
- LLM-адаптер (интерфейс к разговорной модели, ADR [0020](0020-cloud-llm-via-adapter.md))
- Оркестрация MCP-инструментов
- Eval-харнесс (тема 6)

Правило: всё, что является обещанием Азимута пользователю — живёт здесь. Тестируется изолированно.

### Consequences

* Good, because форк обновляется независимо — наш код не затрагивается (если форк соблюдает контракт)
* Good, because Р5/Р6 тестируются unit-тестами без запуска форка или Qdrant
* Good, because новый разработчик сразу видит, куда класть новую фичу
* Bad, because граница требует дисциплины — легко соблазниться «быстро добавить в форк»; митигируется code review
* Bad, because при крупном рефакторинге форка может потребоваться ревизия адаптерного слоя

### Confirmation

`05-building-block-view.md` содержит три явных раздела: «Форк (bsl-atlas)», «Библиотеки», «Азимут-ядро (наш код)». Unit-тесты Р5/Р6 проходят без Docker. CI-линт зелёный.

## Pros and Cons of the Options

### Явная трёхзонная граница (выбрано)

* Good, because поведенческие гарантии изолированы и тестируемы
* Good, because обновление любой зоны не рушит другие (при соблюдении контракта)
* Bad, because требует явного решения при каждой новой фиче: «в какую зону?»

### Всё в форке

* Good, because один репозиторий, простая структура
* Bad, because форк теряет связь с upstream bsl-atlas; нельзя принять upstream-патч без конфликта
* Bad, because поведенческие гарантии Азимута смешаны с BSL-парсером

### Тонкая обёртка без границы

* Good, because быстрый старт
* Bad, because при росте кодовой базы граница размывается, рефакторинг дорожает
