# Миграция стека: гибрид по времени — один дымовой прогон `bsl-atlas` на ChromaDB, затем сразу Qdrant+BGE-M3 (ни строчки нового кода под Chroma)

## Context and Problem Statement

Форк `bsl-atlas` ездит на ChromaDB (векторное хранилище) + SQLite (метаданные). Наш целевой стек — Qdrant (embedded локально) + BGE-M3 (эмбеддер) + реранкер. Нужно ответить: когда и как мигрировать? Мигрировать сразу на старте — или сначала запустить форк as-is, а потом менять?

## Decision Drivers

* ChromaDB не поддерживает нативно hybrid dense+sparse (нужен для BGE-M3 в одной коллекции) и multi-vector — упрёмся в потолок сразу при теме 3.
* Решение 1.1 стоит на допущении «bsl-atlas реально парсит нашу ERP на нашем объёме» — это допущение нужно проверить до вложений в миграцию.
* «Временное» хранилище под ChromaDB — двойная работа: сначала напишешь код под Chroma, потом всё перепишешь под Qdrant.
* Первый дымовой прогон форка as-is — самый дешёвый способ проверить парсер на реальном объёме.

## Considered Options

* **Сразу Qdrant+BGE-M3** — мигрируем до первого прогона.
* **Гибрид по времени** — один дымовой прогон на ChromaDB (as-is), затем сразу Qdrant+BGE-M3.
* **ChromaDB как постоянный стек** — оставить и не менять.

## Decision Outcome

Chosen option: **«Гибрид по времени»**, because он сочетает проверку допущения (дымовой прогон as-is) и неизбежную миграцию (ChromaDB не держит наш целевой поиск). Правило: **ни строчки нового кода под Chroma** — форк запускается как есть, только ради проверки парсера, не ради кода под ChromaDB.

**Последовательность:**
1. **Дымовой прогон** — запустить `bsl-atlas` as-is на реальной ERP. Цель: проверить парсит ли BSL-модули, строит ли граф, не давится ли на объёме (десятки тысяч модулей). Зафиксировать как «Открытый риск» в `roadmap.md`.
2. **Оценить coupling** — насколько жёстко `bsl-atlas` сцеплен со своим хранилищем: если слой storage размазан по коду, миграция дороже.
3. **Миграция** — первая задача фазы после прогона: Qdrant+BGE-M3 становится основным хранилищем. Весь новый код — только под Qdrant.

### Consequences

* Good, because проверяем ключевое допущение (парсер работает на нашем объёме) до вложений в миграцию.
* Good, because не пишем двойной код: нет «временного» слоя под ChromaDB.
* Good, because после миграции весь код — под целевым стеком; нет технического долга «ChromaDB-слой».
* Bad, because дымовой прогон — это риск: если парсер не работает на нашем объёме, решение 1.1 под вопросом. Риск зафиксирован явно.
* Bad, because нет параллельного сравнения ChromaDB vs Qdrant на одном корпусе — метрики качества поиска появятся только после миграции.

### Confirmation

Дымовой прогон зафиксирован как пункт в `roadmap.md` §«Открытые риски». Миграция завершена, когда: `bsl-atlas` использует Qdrant embedded + BGE-M3; ChromaDB и SQLite из форка не используются в production-пути; в кодовой базе нет ни одного импорта `chromadb` в новом коде (только унаследованный форк до миграции).

## Pros and Cons of the Options

### Гибрид по времени (выбрано)

* Good, because проверяем парсер «бесплатно» — форк запускается as-is.
* Good, because нет двойного кода и технического долга под Chroma.
* Bad, because нужен явный контроль «ни строчки нового кода под Chroma» — иначе «временное» присосётся.

### Сразу Qdrant+BGE-M3

* Good, because сразу на целевом стеке — нет «дымового» этапа.
* Bad, because миграция до проверки парсера — вкладываем в Qdrant-слой, не зная работает ли движок вообще.

### ChromaDB как постоянный стек

* Good, because нулевые изменения хранилища на старте.
* Bad, because ChromaDB не держит hybrid dense+sparse и multi-vector — упрёмся в потолок при теме 3.
* Bad, because придётся мигрировать позже, но под давлением уже накопившегося кода.
