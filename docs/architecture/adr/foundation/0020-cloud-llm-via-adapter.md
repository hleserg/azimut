# Разговорная модель — облачная и подключаемая через адаптер; внутри MCP-сервера нет разговорной LLM; лёгкие модели (BGE/реранкер/судья) — локально по умолчанию, грант Cohere — опциональный апгрейд

## Context and Problem Statement

MCP-сервер Азимута выполняет поиск, чанкинг, граф вызовов и контроль галлюцинаций.
Разговорный интерфейс («ответ на вопрос консультанта») требует LLM. Вопрос: встраиваем
ли разговорную LLM внутрь MCP-сервера или оставляем её снаружи, на стороне клиента?
Смежный вопрос: локальные vs облачные модели для лёгких задач (реранкинг, судья).

## Decision Drivers

* Железо мамы и Сергея не тянет локальную разговорную LLM (32B+ параметров)
* Нужна возможность переключить модель (DeepSeek → Claude → Qwen) без переписывания сервера
* Защита от галлюцинаций (Р5/Р6) — на стороне сервера, не модели; должна работать с любой LLM
* Бюджет: грант Cohere есть, но тратить его как дефолт нельзя до оценки cost-per-query
* MCP-протокол не предполагает разговорной LLM внутри сервера — она на стороне клиента

## Considered Options

* Разговорная LLM встроена в MCP-сервер (сервер сам вызывает облачную API)
* Разговорная LLM снаружи — клиент (Cherry Studio / Claude Desktop) управляет, MCP предоставляет инструменты
* Локальная разговорная LLM (Ollama / llama.cpp) — исключено из-за требований к железу

## Decision Outcome

Chosen option: "Разговорная LLM снаружи, адаптер к ней с первого дня", because MCP-протокол именно для этого: сервер предоставляет инструменты, клиент оркеструет разговор с LLM. Встроенная LLM в сервере создаёт двойную точку конфигурации (и у клиента, и у сервера) без архитектурного выигрыша.

**Четыре принципа реш. 1.8:**

1. **Разговорная LLM не встроена в MCP-сервер.** Сервер предоставляет инструменты; LLM вызывает их через клиент.
2. **Адаптер с первого дня.** Интерфейс `LLMAdapter` изолирует остальной код сервера от конкретной модели. Переключение DeepSeek → Claude → Qwen — замена конфига, не рефакторинг.
3. **Антигаллюцинации — агностика модели.** Серверные ворота Р5 (контроль поиска) и Р6 (иерархия источников) работают независимо от того, кто спрашивает — DeepSeek или Claude.
4. **Лёгкие модели (BGE-M3, реранкер, судья) — локально по умолчанию.** Cohere Rerank — опциональный апгрейд (грант), не дефолт.

### Consequences

* Good, because MCP-сервер не зависит от конкретного провайдера LLM
* Good, because переключение модели — замена env/конфига без изменения кода сервера
* Good, because антигаллюцинационные ворота Р5/Р6 защищают от всех моделей одинаково
* Good, because лёгкие модели локально — нулевые расходы на реранкинг и eval в разработке
* Bad, because адаптерный слой — дополнительная абстракция; нужно поддерживать интерфейс при обновлении API провайдеров
* Bad, because Cohere как опциональный апгрейд означает две конфигурации (базовая и с грантом) — документируется в конфиге

### Confirmation

`LLMAdapter` реализован в коде сервера; `config.yaml` содержит секцию `llm.provider`; переключение DeepSeek ↔ Claude меняет только конфиг. `05-building-block-view.md` §«Adapter-слой» описывает интерфейс и реализации.

## Pros and Cons of the Options

### Разговорная LLM снаружи (выбрано)

* Good, because следует MCP-протоколу: сервер = инструменты, клиент = оркестрация
* Good, because не нужна отдельная конфигурация API-ключей разговорной модели в сервере
* Bad, because качество ответа зависит от модели клиента — смягчается тем, что ворота Р5/Р6 на сервере

### Разговорная LLM встроена в сервер

* Good, because можно контролировать промпт от сервера, не от клиента
* Bad, because нарушает MCP-паттерн; дублирует конфигурацию
* Bad, because привязывает сервер к одному провайдеру или требует конфиг в двух местах

### Локальная разговорная LLM

* Bad, because MacBook Air M1 и рабочий ноутбук мамы не тянут 32B+ модели
* Bad, because нет смысла при наличии доступных облачных API
