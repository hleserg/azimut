> arc42 §8 — анти-галлюцинации, мониторинг, безопасность/приватность, лицензии, правило источников, процесс разработки.

## 8.1 Анти-галлюцинации: поведенческий контракт

Контракт: система не выдаёт факты о коде 1С без подкреплённого источника. Механизм — 7 решений (Р1–Р7) + 3 предложения (П1–П3).

### Принятые решения (Accepted)

| ID | Принцип | Суть | ADR |
|---|---|---|---|
| Р1 | Метрика противоречивости | Считать противоречия между чанками ДО генерации; не блокировать выдачу, но логировать в Sentry (`contradiction_count`) | [0001](adr/anti-hallucinations/0001-р1-metric-contradiction.md) |
| Р2 | Faithfulness vs relevance | Реранкер балансирует два сигнала: релевантность запроса и верность источнику | [0002](adr/anti-hallucinations/0002-р2-faithfulness-vs-relevance.md) |
| Р3 | LLM-судья со спан-привязкой | После генерации — Claude API арбитрирует faithfulness + groundedness; при недостаточном score — сигнал на добор | [0003](adr/anti-hallucinations/0003-р3-llm-judge-spans.md) |
| Р4 | ~~Честный тупик~~ | **Снят** — заменён на Р7. «Ответить не знаю» без альтернативы = бесполезно | [0004](adr/anti-hallucinations/0004-р4-honest-deadend-retired.md) |
| Р5 | Server-controlled retrieval | Планка релевантности, триггер добора, потолок окна контекста — на стороне сервера, не агента | [0005](adr/anti-hallucinations/0005-р5-server-controlled-retrieval.md) |
| Р6 | Иерархия источников | При конфликте: **код > справка 1С > ИТС**. Метрика противоречивости (Р1) применяется до этой иерархии | [0006](adr/anti-hallucinations/0006-р6-source-hierarchy.md) |
| Р7 | Фолбэк = смена режима | Если локальный индекс не набирает релевантности → дип-ресёрч в интернете с явной пометкой «не из локального индекса» | [0007](adr/anti-hallucinations/0007-р7-fallback-mode-switch.md) |

✅ проверено: ADR 0001–0007 (статусы и тексты)

### Предложения под калибровку (Proposed)

| ID | Принцип | Суть | ADR | Блокер |
|---|---|---|---|---|
| П1 | Детектор groundedness | 3 уровня реакции на сигнал судьи: «стенка» (блок) / «плашка» / «канарейка» (лог) | [0008](adr/anti-hallucinations/0008-п1-groundedness-detector.md) | Пороги не откалиброваны |
| П2 | Повторный ретривинг | Второй проход ретривера с переформулированным запросом при недостаточном score | [0009](adr/anti-hallucinations/0009-п2-re-retrieval.md) | Механика открыта |
| П3 | Оценка достаточности запроса | Перед ретривингом — оценить, достаточно ли запрос специфичен | [0010](adr/anti-hallucinations/0010-п3-query-sufficiency.md) | Механика открыта |

### Открытый хвост Р1

Механика детектирования противоречий (как технически считать «противоречие», какой порог) не закрыта.
Выделена в отдельный открытый ADR 0033.
✅ проверено: ADR 0001 §Consequences; [ADR 0033](adr/open/0033-r1-contradiction-detection-mechanics.md) status=proposed

## 8.2 Мониторинг и канарейка

**Принцип:** «канарейка сигналит до раздражения пользователя» — каждое нарушение контракта видно в дашборде раньше, чем его почувствует Мама.

### Ключевые метрики Sentry

| Метрика | Что означает | Источник |
|---|---|---|
| `contradiction_count` | Противоречия между чанками перед генерацией | ADR 0001 (Р1) |
| `retrieval_below_threshold` | Ретривинг не набрал планки релевантности | ADR 0005 (Р5) |
| `re_retrieval_triggered` | Запущен повторный добор | ADR 0005 (Р5) |
| `context_budget_hit` | Достигнут потолок окна контекста | ADR 0005 (Р5) |
| `fallback_triggered` | Активирован дип-ресёрч (Р7) | ADR 0007 (Р7) |
| `groundedness_below_hard_threshold` | Блок выдачи — ответ ниже жёсткого порога | ADR 0008 (П1) |
| `groundedness_below_soft_threshold` | Ответ отдан с плашкой | ADR 0008 (П1) |

### Выбор инструмента (открытый вопрос)

Конфликт AGPL-3.0 × Sentry for Open Source: страница sentry.io/for/open-source требует «friendly license» (MIT/Apache), AGPL «friendly» не считается. Текущий статус:

- **План А** (текущий): запросить Sentry напрямую — грант одобряют люди, не автомат.
- **План Б1** (резервный): GlitchTip — Sentry-совместимый SDK, self-hosted, OSS. ⚠️ предположение: лицензия GlitchTip не проверена по файлу на 2026-05-28 — сверить перед выбором.
- **Исключён**: self-hosted Sentry (`getsentry/self-hosted`) — требует 16 GB RAM, несовместимо с локальным деплоем. ✅ проверено: ADR 0028 §«План Б2»

Задачи HLE-315/318/319 в Hold до закрытия ADR 0028.
✅ проверено: [ADR 0028](adr/open/0028-sentry-vs-agpl.md)

## 8.3 Безопасность и приватность

**Основной принцип:** BSL-код клиента — конфиденциальный актив. Уходит только в локальный Qdrant embedded, не в облако.

| Слой | Политика | Обоснование |
|---|---|---|
| Индексация | BSL-файлы → Qdrant embedded (локально). Ни raw-код, ни чанки не передаются в облачные API | On-prem first — ADR 0011 (форк bsl-atlas), ADR 0013 (роль: только движок) |
| Генерация | В LLM уходят вопрос + релевантные чанки; полный исходный код не передаётся | ⚠️ предположение: будет подтверждено при проектировании промпт-шаблона |
| LLM-судья | В Claude API уходит черновик ответа + спан; raw-чанки не передаются | ADR 0003 (Р3) |
| Мониторинг | В Sentry/GlitchTip уходят только метрики и стектрейсы без BSL-кода | ⚠️ предположение: конфигурация scrubbing уточняется при реализации |
| Сеть | Внешние API вызываются только по HTTPS; локальный Qdrant — HTTP без TLS (on-prem) | ✅ проверено: workspace.dsl (технология связей) |

**152-ФЗ:** если в BSL-коде клиента содержатся персональные данные (имена, телефоны в комментариях) — политика уточняется при расширении до SaaS. Для on-prem деплоя данные не покидают машину.

## 8.4 Лицензии и атрибуция

**Корень:** форк `bsl-atlas` (AGPL-3.0) → весь Азимут наследует AGPL-3.0. Перелицензировать невозможно без переписи ядра.

### Практические следствия AGPL-3.0 §13

- Любой пользователь SaaS-версии Азимута вправе запросить исходный код (сетевой пункт §13).
- GPL-совместимые зависимости (MIT, Apache 2.0, GPL) — допустимы. Проприетарные — нет.
- subprocess-граница: компоненты под GPL/LGPL могут вызываться через subprocess без линковки. ✅ проверено: ADR 0023

## 8.5 Правило источников

**Определено в ADR 0023.** Любое фактическое утверждение в документации (`_source/`, ADR, arc42-главах) несёт маркер:

- **`✅ проверено: <файл/url>`** — факт подтверждён в указанном первичном источнике
- **`⚠️ предположение`** — факт не проверен; требует верификации перед принятием решения

Утверждения без маркера считаются предположениями при ревью. Маркер относится только к проверяемым фактам: ADR-ссылки, лицензии, технические характеристики. Общеизвестные технологические факты (Python — интерпретируемый язык) маркировать не нужно.

✅ проверено: [ADR 0023](adr/foundation/0023-license-checklist-and-source-rule.md) §«Правило источников»

## 8.6 Процесс разработки и Definition of Done

### ADR-процесс

Любое архитектурное решение фиксируется в [MADR](adr/template.md)-формате:
1. Создать через `./scripts/new-adr.sh <подпапка> <kebab-title>`
2. Обновить `workspace.dsl` если изменилась архитектура (workspace.dsl — единственный источник истины для C4)
3. Обновить индекс: `./scripts/update-adr-index.sh`
4. При новом инфраструктурном элементе — обновить главу 13 (Lead Operating Manual)

Папки ADR по темам:

| Папка | Тематика |
|---|---|
| `adr/foundation/` | Форк, стек, клиенты, лицензии, граница кода |
| `adr/anti-hallucinations/` | Р1–Р7, П1–П3 |
| `adr/code-processing/` | Чанкинг, граф, роутинг, feenlace |
| `adr/open/` | Открытые вопросы: Sentry, мульти-аренда, Р1-механика |
| `adr/tooling/` | Structurizr DSL (ADR 0034) |

✅ проверено: AGENTS.md §«Definition of Done»; `workspace.dsl` (adr-link properties)

### Статус-протокол Linear

- `Done` ставит только Сергей

