# 12. Глоссарий (Glossary)

> arc42 §12 — термины 1С, проектные, технические; 25+ BSL entry-points.

---

## Термины 1С

| Термин | Определение |
|---|---|
| **BSL** | Built-in Script Language — встроенный язык программирования платформы 1С:Предприятие 8. Используется для написания бизнес-логики конфигураций. |
| **Конфигурация** | Прикладное решение на платформе 1С (например, 1С:ERP, 1С:Бухгалтерия). Содержит объекты метаданных, модули на BSL, формы. |
| **Расширение (.cfe)** | Механизм дополнения типовой конфигурации без изменения основного кода. Файл `.cfe` содержит перекрывающиеся/добавленные объекты и BSL-модули. |
| **Внешняя обработка (.epf)** | Отдельный бинарный файл с BSL-кодом и формой, не входящий в конфигурацию. Запускается интерактивно или программно. |
| **Внешний отчёт (.erf)** | Аналог `.epf` для отчётов — бинарный файл вне конфигурации. |
| **АГРЕГАТ** | Объект метаданных платформы 1С 8.2+: накопительные таблицы для ускорения запросов к регистрам накопления. Вычисляются и хранятся платформой автоматически. |
| **ПодпискаНаСобытие** | Объект метаданных: позволяет модулю-обработчику реагировать на событие другого объекта (ПриЗаписи, ПередЗаписью и т.д.) без прямого вызова. Механизм слабой связанности в 1С. |
| **МенеджерВременныхТаблиц** | Объект платформы для создания временных таблиц внутри транзакции запроса. Передаётся между процедурами как параметр; место наполнения определяется только runtime-трассой. |
| **ОбработкаПроведения** | Предопределённая процедура-обработчик документа: выполняется при проведении документа, формирует движения по регистрам. Главная точка входа бизнес-логики документов. |
| **НачатьВыполнение** | BSL-функция асинхронного вызова процедуры. Разрывает синхронную цепочку вызовов — результат возвращается через `ОписаниеОповещения`. |
| **ОписаниеОповещения** | Объект-колбэк: ссылка на процедуру + контекст, передаётся в асинхронные методы. Реализует pattern callback в 1С. |

## BSL Entry-Points (25+ точек входа)

Полный список точек входа из `call_graph.go` библиотеки `mcp-bsl-lsp-bridge` (источник: HLE-463). Это «откуда начинаются цепочки вызовов» — стартовые узлы для анализа графа и поиска неиспользуемого кода.

### Объект / Документ

| BSL | Описание |
|---|---|
| `ПриЗаписи` / `OnWrite` | Запись объекта (после выполнения) |
| `ПриПроведении` / `Posting` | Проведение документа |
| `ПриОтменеПроведения` / `UndoPosting` | Отмена проведения |
| `ПередЗаписью` / `BeforeWrite` | До записи объекта |
| `ПередУдалением` / `BeforeDelete` | До удаления объекта |
| `ПриКопировании` / `OnCopy` | При создании копии объекта |
| `ОбработкаЗаполнения` / `Filling` | Заполнение реквизитов по умолчанию |
| `ОбработкаПроверкиЗаполнения` / `FillCheckProcessing` | Проверка заполнения перед записью |

### Форма

| BSL | Описание |
|---|---|
| `ПриСозданииНаСервере` / `OnCreateAtServer` | Инициализация формы на сервере |
| `ПриОткрытии` / `OnOpen` | После открытия формы на клиенте |
| `ПриЗакрытии` / `OnClose` | Перед закрытием формы |
| `ПередЗаписьюНаСервере` / `BeforeWriteAtServer` | До записи объекта из формы |
| `ПриЗаписиНаСервере` / `OnWriteAtServer` | При записи объекта из формы |
| `ПослеЗаписиНаСервере` / `AfterWriteAtServer` | После записи объекта из формы |
| `ПриЧтенииНаСервере` / `OnReadAtServer` | После чтения объекта в форму |
| `ОбработкаОповещения` / `NotificationProcessing` | Обработка межформенных оповещений |
| `ОбработкаНавигационнойСсылки` / `URLProcessing` | Переход по навигационной ссылке |

### Команды

| BSL | Описание |
|---|---|
| `ОбработкаКоманды` / `CommandProcessing` | Выполнение команды объекта |
| `ПриВыполнении` / `OnRun` | Выполнение команды (альтернатива) |

### Сессия / Система

| BSL | Описание |
|---|---|
| `ПриНачалеРаботыСистемы` / `OnSystemStart` | Старт сеанса работы 1С |
| `ПриЗавершенииРаботыСистемы` / `OnSystemEnd` | Завершение сеанса |
| `ПередНачаломРаботыСистемы` / `BeforeSystemStart` | До старта системы |
| `ПередЗавершениемРаботыСистемы` / `BeforeSystemEnd` | До завершения системы |

### Регламентные задания

| BSL | Описание |
|---|---|
| `ОбработчикРегламентногоЗадания` / `ScheduledJobHandler` | Выполнение фонового/регламентного задания |

### HTTP / Web-сервисы

| BSL | Описание |
|---|---|
| `ОбработкаВызоваHTTPСервиса` / `HTTPServiceHandler` | Обработка входящего HTTP-запроса |
| `ОбработкаВызоваWebСервиса` / `WebServiceHandler` | Обработка входящего SOAP-запроса |

## Проектные термины

| Термин | Определение |
|---|---|
| **Азимут** (`azimuth`) | Имя форка `bsl-atlas` — MCP-сервер для понимания кода 1С. Навигационная метафора: даёт «направление поиска», не вердикт. |
| **Азимут-ядро** | Форк `bsl-atlas`: парсер BSL + SQLite-граф вызовов + каркас MCP + Docker. Отвечает за «понимание кода 1С». |
| **MCP-оркестратор** | Компонент Азимута: связывает граф + поиск + реранк + контракт + LLM-судья в один поведенческий контракт агента. |
| **Шапка контекста чанка** | Метаданные чанка кода: имя процедуры, путь к файлу, тип объекта 1С, признак расширения. Формирует «откуда это» для агента. |
| **Поведенческий контракт** | Системный промпт MCP-сервера: портативный (модель-независимый) набор правил поведения агента при ответах, источниках, фолбэках. |
| **Дымовой прогон** | Первый запуск `bsl-atlas` на реальной ERP-конфигурации: цель — верифицировать допущение «парсит наш объём, строит граф» (ADR 0015). |
| **Шапка источника** | Атрибуция ответа: «из кода конфигурации», «из справки 1С», «из ИТС», «из интернета» — каждый ответ помечен источником (ADR 0006). |

## Технические термины

| Термин | Определение |
|---|---|
| **ISCF** | 1C ISS Cabinet Format — проприетарный формат базы данных 1С:ИТС. Три файла: `Data.cab` (контент, магия `ISCF`), `Data.dir` (индекс, магия `CFHD`), `.dat/.dbs/.dbv` (B-tree таблицы). Требует собственного парсера для извлечения HTML-документации. |
| **MADR** | Markdown Architectural Decision Records — формат документирования архитектурных решений. Секции: Context → Decision Drivers → Considered Options → Decision Outcome → Consequences. |
| **RRF** | Reciprocal Rank Fusion — алгоритм слияния нескольких ранжированных списков поиска в один. Используется в гибридном поиске (vector + BM25 → RRF). |
| **Self-RAG** | Self-Reflective Retrieval-Augmented Generation — подход, при котором LLM сама оценивает нужность retrieval и quality ответа (groundedness, relevance). Влияет на дизайн контракта Азимута. |
| **Faithfulness** | Метрика RAGAS: доля утверждений в ответе, которые поддержаны извлечёнными чанками. Измеряет «не придумал ли генератор». Целевой порог: ≥ 0.80 (§10). |
| **Context Relevance** | Метрика RAGAS: насколько извлечённые чанки релевантны запросу. Измеряет «нашёл ли ретривер нужное». Считается независимо от faithfulness (ADR 0002). |
| **BGE-M3** | BAAI/bge-m3 — многоязычная embedding-модель (Chinese, English, Russian). Используется для векторизации чанков кода 1С и запросов в Qdrant. |
| **Cohere Rerank v4** | Cohere Rerank v4 — облачный нейросетевой реранкер. Используется как второй проход после vector search для повышения точности. Грант Cohere for AI — опциональный апгрейд (ADR 0020). |
| **AGPL §13** | Секция 13 лицензии GNU AGPL-3.0: требует предоставлять исходный код пользователям, взаимодействующим с программой по сети. Применяется к Азимуту при сетевом деплое (тема 7). |
| **FastMCP** | Python-фреймворк для создания MCP-серверов. Основа Python-стека Азимута. |
| **Qdrant** | Векторная база данных. Целевое хранилище Азимута (после дымового прогона на ChromaDB, ADR 0015). |
| **SQLite `calls`** | Таблица в `bsl-atlas`: `calls(caller_id, callee_name)` — рёбра графа вызовов BSL-кода. Основа для поиска «кто вызывает / кого вызывает». |
| **RAGAS** | Фреймворк для eval RAG-систем. Предоставляет метрики faithfulness, context relevance, answer relevance. Используется в eval-харнессе (тема 6, HLE-418). |
| **tree-sitter** | Инкрементальный парсер с поддержкой BSL. Используется в Азимуте для детерминированной структурной резки кода (ADR 0024). |
